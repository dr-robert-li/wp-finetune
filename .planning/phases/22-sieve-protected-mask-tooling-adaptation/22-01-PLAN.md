<objective>
Adapt the CPU/code half of the MoE-Sieve profiler + protected-mask + k-sweep stack from the old
Qwen3-30B-A3B assumption (top-8-of-128, 48 uniform-attention layers, no shared expert) to the v4 judge's
Qwen3.6-35B-A3B architecture (top-8-of-256 routed + 1 always-on shared expert, 40 layers in a
10×(3×DeltaNet-MoE + 1×Gated-Attention-MoE) pattern), plus the vLLM router-mask patch — all under
mock-based pytest with NO GPU. Every architecture fact becomes config/data-derived so v3 and v4 both work.

Purpose: GATE4-02 — the tooling must be audited and 256-expert/strata/shared-expert-correct BEFORE
Conditional Gate B (Phase 25) profiles the v4 judge. This plan does the deterministic code+unit-test work;
Plan 22-02 proves it empirically on the real model with one bounded GB10 load.

Output: scripts/sieve_arch.py (new arch helper), the six adapted Sieve/profiler scripts, the adapted vLLM
patch, and updated/extended unit tests — all green on CPU.
</objective>

    - arch_dims(config) returns (n_layers, n_experts): reads num_hidden_layers + num_experts from a HF config
      or its text_config; on the v4 base returns (40, 256); on a v3 Qwen3-MoE config returns (48, 128).
    - layer_strata(config) returns a list[str] of length n_layers: "attention" where layer_types[i] ==
      "full_attention", else "deltanet". On the v4 base this yields exactly 30 "deltanet" + 10 "attention"

      key exists (v3 uniform base) — never raises.
    - resolve_moe_layers(model) returns list of (layer_idx, mlp_module) for every decoder layer that has an
      mlp.gate, trying candidate roots in order (model.model.layers, then model.model.language_model.layers,

      RuntimeError naming all tried paths if none resolve — so a silent zero-hook run (the exact 20-04
      failure mode) is impossible.

      max(layer_idx)+1, n_experts = max(int expert-id across expert_counts_* dicts)+1. Used where no model
      object is loaded (mask extraction, cross-seed load).
    - resolve_task_token_ids(tokenizer, default_gen, default_judge) returns (gen_id, judge_id): looks up
      "<wp_gen>"/"<wp_judge>" in the tokenizer's vocab; returns the defaults only when both resolve, else
      (None, None) so callers degrade to total-only tagging on a base without the task-token extension
      (the v4 judge, vocab 248320, has none — see 20-04).
  </behavior>
