    **Step 0: Setup**

    - Clone wp-bench if not present: `git clone https://github.com/WordPress/wp-bench.git` into project root
    - Run wp-bench setup script to build Docker WordPress runtime image (per Pitfall 6)
    - Verify vLLM health endpoint before each eval run

    **Step 1: Base-Model Profiling (per D-01, D-15)**
    - `run_profiling()` function:
      - Load base model from `models/Qwen3-30B-A3B` with `torch_dtype=torch.bfloat16, device_map="auto"`
      - Load extended tokenizer from `adapters/tokenizer/` (NOT base tokenizer — anti-pattern from RESEARCH.md)
      - Call `profile_base_model()` from Plan 01's script with all 5 ratio data paths
      - Write JSONL to `output/profiling/base_model_eeff.jsonl`
      - Write summary to `output/profiling/base_model_eeff_summary.md`
      - Check `has_downward_eeff_trend()` — if True, print "E_eff trending down: 60/40 training warranted" (per D-05)
      - Unload model to free GPU memory before eval step

    **Step 2: Sequential Adapter Eval (per D-06, D-07)**
    - `run_eval_for_ratio(ratio)` function:
      - Start vLLM with LoRA adapter: use dgx_toolbox.run_service("vllm") with `--enable-lora --lora-modules qwen3-wp=/workspace/wp-finetune/adapters/qwen3-30b-wp-{ratio} --max-lora-rank 64 --max-model-len 4096 --gpu-memory-utilization 0.92`
      - Wait for vLLM health (poll `{vllm_endpoint}/health` up to 5 min, per RESEARCH.md)
      - **Verify model name** (Pitfall 3): GET `{vllm_endpoint}/v1/models`, assert "qwen3-wp" in available models. If model name differs, log warning and adapt.
      - Run `eval_gen.run_eval(output_path=f"output/eval_triage/ratio_{ratio}/eval_gen_results.json")`
      - Run `eval_judge.run_eval(output_path=f"output/eval_triage/ratio_{ratio}/eval_judge_results.json")`
      - Run `eval_gate.run_gate(results_dir=f"output/eval_triage/ratio_{ratio}")`
      - Stop vLLM before next ratio

    - `run_wpbench_for_ratio(ratio)` function:
      - Assumes vLLM is already running (called between eval steps or as part of run_eval_for_ratio)
      - Update wp-bench config output path to `output/eval_triage/ratio_{ratio}/wp_bench_results.json`
      - Run wp-bench against the live vLLM endpoint
      - Collect results

    - Loop over `["30_70", "40_60", "50_50"]` — serve each sequentially, run full eval + wp-bench per D-07

    **Fallback for vLLM LoRA (Open Question 1 from RESEARCH.md):**
    If vLLM fails to load adapter due to `modules_to_save` tensors, fall back to:
    1. Run `scripts/merge_adapter.py` to create merged checkpoint
    2. Serve merged model as full model (no `--lora-modules`)
    3. Continue eval as normal

    **Step 3: Triage Decision (per D-12, D-17)**
    - Call `triage_ratios.load_eval_results("output/eval_triage")`
    - Call `triage_ratios.triage_ratios(eval_results, profiling_summary)`
    - Call `triage_ratios.write_triage_decision()` to `output/triage_decision.md`
    - Print summary: survivors, eliminated, E_eff trend, 60/40 training recommendation

    **CLI:** argparse with --skip-profiling (to skip step 1 if already done), --skip-wpbench (if wp-bench not ready), --ratios (override which ratios to eval, default "30_70,40_60,50_50")
