### Section B — NEW concerns introduced by the bf16 pivot

The shift to bf16 introduces a different set of risks, primarily related to the tight memory margins on the GB10 host.

*   **1. vLLM→Unsloth handoff co-residence risk (63 GiB + 57 GiB ≈ 120 GiB): MEDIUM**

The plan relies on a **procedural "bf16 handoff precondition"**—an instruction to the executor to wait for free RAM to recover to ~110 GiB before launching Arm 2. While the plan includes a watchdog backstop (~18 GiB trip), it lacks a hard code-enforced "wait-loop" script. This is a "tight-clearance" maneuver where an operator's haste could still trigger a host crash.
*   **2. Sequential-load discipline (Arm 2 vs. Arm 3): LOW**
    *   **Justification**: The plan mandates that Arms 2 and 3 be run in separate process lifetimes ("SEQUENTIAL... separate ~57 GiB bf16 process"). Process termination is a reliable OS-level mechanism for memory release, making the risk of co-residence between these two arms negligible compared to the vLLM handoff.
*   **3. MoE-binding guard's revalidation on the bf16 path: LOW**
    *   **Justification**: The plan's "activation-delta" probe (Task 1) is a highly robust and scientifically sound verification of runtime binding. By requiring a nonzero output delta from the experts submodule when the adapter is toggled, it provides empirical proof that the LoRA is contributing to the model's computation. Testing the "tester" (Task 1e unit test) further mitigates the risk of false positives.
*   **4. n=120 sample size precision: LOW**
    *   **Justification**: While 120 samples is a relatively small window for a 10% target rate, the plan adopts a rigorous Wilson-CI discipline. The `ENGINE_CI_OVERLAP` cross-check is a particularly strong addition; by comparing two merged bf16 arms across different engines (vLLM vs. Unsloth), it isolates engine-specific artifacts from merge-math failures.

## Gemini Review

# Plan Review: 04.3-03 (Merge-vs-Training Discriminator)

## 1. Summary
The plan is meticulously designed with strong scientific discipline, utilizing pre-registered thresholds, confidence intervals, and strict runtime guards. However, it is fundamentally invalidated by hardware and architectural realities discovered during execution. The load-bearing decision to use 4-bit precision for the primary arms is unexecutable due to unified memory exhaustion (~108 GiB transient peak) and scientifically invalid because 4-bit quantization is known to break Qwen3-MoE routing. The plan's foundational premise—that 4-bit is memory-safe and bf16 is dangerous—is empirically backwards for this environment. The plan must be pivoted to a bf16-only approach.

## 2. Strengths
- **Scientific Rigor:** The use of pre-registered Wilson-CI thresholds and machine-checked logic prevents human bias and confirmation bias when interpreting results.
- **Runtime Binding Guard:** The non-negotiable requirement to verify MoE expert-LoRA binding by inspecting the loaded model object's `named_modules` (and a logits functional backstop) is excellent. It prevents the insidious risk of testing a silently un-bound base model.
- **Confound Isolation:** The three-arm design perfectly isolates the variables: Unsloth vs. vLLM (engine check) and merge-math vs. `load_adapter` (the core discriminator).
- **Strict Scope Fencing:** Explicitly forbidding the promotion of checkpoints or launching corrective retrains keeps the plan focused purely on measurement and diagnosis.

## 3. Concerns

- **[HIGH] Scientific Invalidity of 4-bit Measurements:** The prior phase (04.3-01) established that 4-bit quantization breaks Qwen3-MoE router weights, leading to degenerate output. Measuring the "terse-JSON rate" on a model that produces token loops or stalls will yield garbage data. A discriminator built on these measurements cannot be trusted.
- **[HIGH] Unexecutable Memory Profile (OOM Risk):** The plan mandates on-the-fly 4-bit loading for Arms 2 and 3. Execution state reveals this creates a massive transient memory peak (~108 GiB, holding bf16 shards + 4-bit copies) that exceeds the available pool on the live desktop, guaranteeing a watchdog trip or system crash.
- **[HIGH] Flawed Safety Premise:** The plan states "4-bit is the safe default, bf16 is dangerous." Empirical execution proves the opposite: a bf16 in-process load is a flat ~57 GiB (comfortably safe), while the 4-bit path triggers a lethal ~108 GiB memory spike.
- **[MEDIUM] Pre-quantization Workaround Failure:** Attempts to bypass the on-the-fly memory spike via pre-quantization resulted in an attention-only nf4 artifact. This artifact is incompatible with the Unsloth fused-MoE adapter binding required for Arm 3 (`load_adapter`).

## 4. Suggestions
- **Pivot to a bf16-Only Discriminator:** Change Arm 2 to `merged-72-Unsloth-bf16` and Arm 3 to `unmerged-72-Unsloth-bf16`. Since Arm 1 is already bf16, this aligns all three arms to the same precision, removing precision as a confounding variable entirely.
- **Leverage the Safe bf16 Load Path:** Execute the in-process loads using bf16, taking advantage of the empirically validated ~57 GiB peak memory profile which is safe for the live desktop environment.
- **Update the Binding Guard for bf16:** Ensure the `checkpoint_parse_check.py` runtime binding guard logic explicitly supports verifying the PEFT/Unsloth adapter binding against a bf16 base model, rather than relying on bnb-4bit specific data structures.
- **Scrap 4-bit Pre-quantization:** Abandon the pre-quantization script and the resulting incompatible nf4 artifacts, as they are no longer necessary for memory safety and are structurally incompatible with the test design.

## 5. Risk Assessment
**HIGH RISK.** If executed as written, the plan is destined to fail mechanically (due to the ~108 GiB unified memory exhaustion during 4-bit loading). Even if the memory constraint could be bypassed, the plan would fail logically because measuring format stability on a 4-bit MoE model with broken routing will produce a corrupted discriminator verdict. Pivoting to the bf16 load path resolves both the memory blocker and the architectural validity issue, which would lower the risk to LOW.
