### Auto-fixed Issues

**1. [Rule 1 - Bug] Preserved --exclude-lm-head merge_type on non-MoE-only path**
- **Found during:** Task 2
- **Issue:** PATTERNS.md lines 167-172 only set `merge_type` in the `is_moe_only` branch of the lm_head-excluded else block. The original code (line 384) also set `merge_type="tinker_per_expert_moe_plus_peft_attention_NO_lm_head"` for the `--exclude-lm-head` (non-MoE-only) case. Copying PATTERNS verbatim would silently leave that path with the wrong default merge_type.
- **Fix:** Added `else: report["merge_type"] = "tinker_per_expert_moe_plus_peft_attention_NO_lm_head"` to preserve the original --exclude-lm-head behavior.
- **Files modified:** scripts/merge_tinker_v3.py
- **Commit:** 223cecc

**2. [Rule 3 - Blocking] transformers _LazyModule bypasses standard mock patching**
- **Found during:** Task 3
- **Issue:** `patch('transformers.AutoModelForCausalLM')` and `patch.object(transformers, 'AutoModelForCausalLM')` both fail to intercept `from transformers import AutoModelForCausalLM` inside `_run_merge`. transformers uses `_LazyModule` whose `__getattr__` always re-derives the class from its internal mapping, bypassing `__dict__` writes made by `patch`.
- **Fix:** Force-resolve the class objects at test module import time (`_AMFCLM = transformers.AutoModelForCausalLM`), then patch `from_pretrained` directly on the resolved class objects via `patch.object(_AMFCLM, "from_pretrained", return_value=fake_model)`. The lazy module always returns the same class object, so patching that object's method is visible from any subsequent `from transformers import` call.
- **Files modified:** tests/phase4_4/test_tinker_merge_convention.py
- **Commit:** f181348
