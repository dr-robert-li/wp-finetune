## Caveats — RTRN-04 Post-hoc Gate Architecturally Invalid for Qwen3-MoE

**The parse-check ran clean (no OOM after 4-bit + max_memory mitigation) but reported 100% parse-fail rate (3/3 and 10/10 runs).** Diagnostic dump of the 3 failed samples on 2026-05-28T01:33Z showed degenerate generation, not parser strictness:

| Sample | Output | Pattern |
|---|---|---|
| 1 (idx=0) | `## Admin` repeated ~80× (3417 chars) | Token-loop mode collapse |
| 2 (idx=1) | `<think>\n` (8 chars) | Stalled at reasoning start |
| 3 (idx=2) | `@param int $time_limit / @return self` repeated ~80× (3012 chars) | Docblock loop |

**Root cause:** int4 quantization of the Qwen3-MoE router weights flips top-8 expert routing decisions → wrong-expert pathways → degenerate output. **This is a known MoE-quantization failure mode and is independent of the trained adapter.** A base Qwen3-30B-A3B model loaded at 4-bit on this hardware would produce the same pattern.

**Why 4-bit was used:** GB10 has 121 GiB unified memory shared between system RAM and GPU. Loading 30B bf16 weights (~60 GiB) via HF Transformers + bitsandbytes pre-allocates additional driver memory that exhausts the unified pool — 2026-05-28 10:13 NVRM `NV_ERR_NO_MEMORY` at 16 GiB available confirmed this. 4-bit was the only path that fit; it now turns out to also be the only path that breaks the model.

**Why vLLM bf16 wasn't used:** the adapter has `target_parameters: [mlp.experts.*]` — a parameter-level MoE-expert LoRA that vLLM does not support as a runtime adapter (per `scripts/serve_30_70_vllm.sh` precedent: vLLM only supports module-level LoRA). The merged-then-served pattern requires an Unsloth-aware merge first, which itself faces the same GB10 bf16 memory problem.

**What the loss curve tells us:** training succeeded. 30% loss reduction over 72 steps, grad norms stable at ~0.2, no NaN, no divergence. The clean curve is consistent with a healthy continued-FT run. **Treat RTRN-04 as PARTIAL — the abort hook is in place; the post-hoc verdict at 4-bit is invalid for this architecture.**

## Carried to Phase 4.4 — Eval-Architecture Blocker

Phase 4.4 (reasoning-eval-and-merge) must solve **eval at bf16 on GB10 for an MoE-expert LoRA**. Three plausible paths:

1. **Unsloth FastLanguageModel bf16 in-process eval with careful memory management** — load merged base + adapter in unsloth's optimized loader; cap max_memory; serve via direct generate(). Requires verifying bf16 fits with no other large processes on host (kill Chromium pre-flight; raise floor to ~70 GiB).
2. **Unsloth merge → vLLM serve** — `FastLanguageModel.save_pretrained_merged(save_method='merged_16bit')` to materialize a true bf16 merged model with the MoE-expert LoRA baked into expert weights, then serve via the existing `serve_30_70_vllm.sh` pattern (GPU_MEM_UTIL=0.55 worked previously). Verify Unsloth merge supports `target_parameters` — not battle-tested on Qwen3-MoE.
3. **GGUF Q8 conversion via llama.cpp** — Q8 preserves MoE routing far better than int4; persistent artifact for fast eval cycles. Cost: ~20 min one-time conversion; requires confirming llama.cpp Q8 support for target_parameters-merged Qwen3-MoE.

The decision must be made at Phase 4.4 planning time. Until then, **adapter ckpt-72 is the authoritative output of Phase 4.3** and is unblocked for downstream eval despite the post-hoc gate being inconclusive.
