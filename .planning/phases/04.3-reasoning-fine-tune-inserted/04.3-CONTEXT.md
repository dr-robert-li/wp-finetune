## Phase Boundary

**Corrective Tinker retrain** of the reasoning LoRA so its **MoE judge skill is less
codegen-destructive when merged into the base** — without losing judge skill — then re-merge
and re-gate REVL-04. This is a targeted re-open of an already-promoted model (`wp-reasoning-v3`),
NOT a from-scratch redo.

**Why re-opened (the live problem):** Phase 04.4's merge attribution (D-IT-02 / dit02) proved
merging the v3 LoRA into base damages codegen (wp-bench 0.4537 → 0.3716) and that this damage is
**entangled in the MoE component** — MoE deltas carry BOTH the judge skill AND the codegen
damage; attention deltas are **net-harmful** (add codegen damage, contribute no judge skill).
The MoE-only merge already beat v3_full on BOTH axes (judge Spearman 0.3124 vs 0.2446; codegen
0.4071 vs 0.3716) but still fails REVL-04 (< baseline 0.4537). HUMAN DECISION 2026-06-10:
**retrain** to find the rank/replay sweet spot that closes the codegen gap while keeping judge.

**In scope:** retrain the LoRA on Tinker with new targeting/rank/replay levers; per-candidate
cheap judge gates + merge + REVL-04 wp-bench; select a grid winner; hand a merge-ready adapter
to Phase 04.4 re-gate.

**Out of scope (do NOT reopen):** the REVL-05 judge-QUALITY fix (resolved in v3 — keep the
augmented-593 set + verdict policy + confusion gate); the local GB10/Unsloth load path
(obviated — Tinker only); CoT backfill / Phase 4.1-4.2 re-open (the v3 data set is the base).

<decisions>
## Implementation Decisions

### LoRA target + rank (the "smaller, MoE-focused perturbation" lever)
- **D-N1 — Drop the attention target → MoE-only LoRA.** Target the MoE expert/router-adjacent
  linears only; exclude attention q/k/v/o. dit02: attention deltas are net-harmful (codegen
  damage, zero judge contribution); MoE-only merge already beat v3_full on both judge and
  codegen. (v3 was "all-linear" — this is the change.)
- **D-N2 — Sweep MoE LoRA rank {8, 16, 32}.** Rank is the primary empirical codegen↔judge
  knob: lower rank = smaller MoE perturbation = less codegen damage, but judge lives in MoE so
  too-small risks losing it. v3 used rank 32.
- **D-N3 — Hold MoE LoRA LR at 4.99e-4 (v3 value); move rank only.** Isolate one perturbation
  variable for clean attribution. Revisit LR only if no rank closes the gap. (NOTE: this is the
  Tinker-regime LR, NOT the ROADMAP RTRN-01 "≤2e-5" figure — see ROADMAP-drift note below.)

### Codegen replay (protect base coding through the SFT)
- **D-N4 — Center wp_gen codegen replay at ~30%; sweep replay as an axis.** More wp_gen replay
  during SFT = the LoRA preserves codegen = less merge damage; complements the rank lever.
  Current was ~15% (old D-02). Replay axis = {15%, 30%, 50%} with 30% as the center/default.
  **RESOLVED 2026-06-11 (research blocker #2): the % is PURE wp_gen fraction** — each cell = that
  fraction of the training mix is wp_gen codegen replay, remainder = augmented-593 reasoning.
  Single clean axis (NOT the v1.2 mixed "30% judge + 20% wp_gen" recipe — that parenthetical was a
  magnitude comparison only, now retired). Tradeoff accepted: more replay dilutes reasoning-signal
  density on a small (~593) set, so the grid measures it rather than guessing.

### Search strategy (cost-no-object)
- **D-N5 — rank×replay grid + hardened eval.** Full grid = rank {8,16,32} × replay {15,30,50%}
  = **9 candidates**. Compute is not the constraint; the binding constraint is statistical
  validity (multiple-comparisons on a noisy wp-bench), so spend the freed budget on:
  - **Wider wp-bench n** per candidate → tighter Wilson CI (real separation, not noise).
  - **Pre-registered selection rule** decided before seeing scores (see D-N8).
  - **Confirmation re-run** of the grid winner before it is promoted.

### Judge-survival verification + acceptance bars
- **D-N6 — Gate judge-survival pre-merge (cheap filter) + confirm post-merge (winner).**
  Score each Tinker checkpoint's judge skill DIRECTLY on the Tinker sampler — offline, no merge
  (`eval/eval_judge.py --responses-jsonl`, P4/P5 path): REVL-01A Spearman + invalid-PHP
  sentinel + two-sided confusion. Only judge-passing candidates advance to the expensive
  merge + wp-bench. Re-confirm judge POST-merge on the grid winner (the merge is the thing that
  damages, so a pre-merge pass is necessary but not sufficient).
- **D-N7 — Judge accept bar = POINT Spearman ≥ v3's 0.263.** "Do no harm": never ship a judge
  worse than the promoted v3. **RESOLVED 2026-06-11 (research blocker #1):** the bar is the POINT
  estimate ≥ 0.263, NOT the CI-lower bound. Fisher-z proved "CI-lower ≥ 0.263" at n≈113 requires
  point ≥ 0.427 — *stricter* than the 0.3124 point bar this decision explicitly rejected, and v3
  itself (point 0.263) would fail it (CI-lower 0.082). The CI-lower is retained ONLY as a
  noise-guard: reject a candidate whose judge measurement has an anomalously wide CI (untrustworthy
  estimate), not as the accept threshold. This honors the original intent — lenient vs v3, does NOT
  fight the rank-reduction lever. Still rejected ≥0.3124 as a hard bar for the same lever reason.
  **Mandatory regardless of Spearman:** invalid-PHP sentinel 0 policy false-pass + confusion gate
  (REVL-05 — non-negotiable, do not regress).
- **D-N8 — REVL-04 wp-bench accept bar = strictly ≥ baseline 0.4537 (HARD gate).** The HUMAN
  DECISION's stated goal is to *close* the codegen gap. The interim MoE-only 0.4071 is NOT an
  acceptable ship bar (it still regresses vs baseline). If NO grid candidate reaches baseline →
  this is an **escalation decision point** (widen grid / human-signed best-available / rethink
  approach), NOT an auto-ship.
- **Pre-registered selection rule (RESOLVED 2026-06-11):** **max wp-bench codegen subject to judge
  POINT Spearman ≥ 0.263 AND REVL-05 gates pass; tie-break on higher judge. CI-lower used only to
  reject anomalously-wide-CI (untrustworthy) judge measurements, not as the accept threshold.** (So
  a candidate that also clears 0.3124 with strong codegen still wins — the bar rewards extra judge
  without requiring it.)

### Lineage — locked, still binding
- **D-04 (retained) — Base lineage = `qwen3-30b-wp-30_70` winner.** Tinker base =
  `Qwen/Qwen3-30B-A3B` (stock; the 30_70 lineage is the data/skill ancestry, not a literal
  checkpoint load — Tinker trains from the stock base). The dead `experiment_001_20260411`
  adapter is NOT a candidate.
- **Format stability (RTRN-05) still binds** — FS-terse on cot+ctf ≤ 10% (Wilson-upper ≤ 15%).
  v3 hit 0.0%/1.1%; the retrain must not regress it.

### Claude's Discretion (→ planner/researcher, not user decisions)
- **Tinker MoE-only targeting mechanics** — confirm the Tinker LoRA API supports targeting MoE
  expert linears only (exclude attention); resolve the exact module-name pattern for Qwen3-MoE

- Exact wp-bench n for the "wider n" hardening (CI-width target), grid execution order, epochs
  (v3 used 3), per-candidate checkpoint selection (v3 promoted ep3).
- Exact construction of the 3 replay-ratio training variants from the augmented-593 + wp_gen
  replay source (reuse `build_augmented_train.py` / `build_reasoning_negatives.py`).
- Whether to short-circuit the grid (early-stop a rank/replay region that clearly can't clear
  both bars) to save the cost the user de-prioritized — optional optimization.
