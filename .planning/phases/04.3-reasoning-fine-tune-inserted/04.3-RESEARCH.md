## D-01 RESOLUTION — Option B (LoRA on merged model) [PRIMARY DELIVERABLE]

**Decision: Option B. Train a FRESH LoRA on `models/qwen3-30b-wp-30_70-merged/`. Confidence: HIGH.**

### Why Option A is not viable (framework-level blocker)

The Phase 3 30_70 adapter applies LoRA to the MoE expert weights via PEFT's `target_parameters` API, because in current Transformers the Qwen3-MoE experts are fused into stacked `nn.Parameter` 3D tensors (`mlp.experts.gate_up_proj`, `mlp.experts.down_proj`) rather than `nn.Linear` modules:

```json
// adapters/qwen3-30b-wp-30_70/adapter_config.json  [VERIFIED]
"peft_version": "0.18.1",
"target_modules": ["gate_up_proj","k_proj","v_proj","down_proj","o_proj","q_proj"],
"target_parameters": ["mlp.experts.gate_up_proj", "mlp.experts.down_proj"]
```

The authoritative PEFT documentation states the blocking constraint verbatim:

> "It is currently not possible to add multiple LoRA adapters (via `model.add_adapter` or `model.load_adapter`) that use `target_parameters` at the same time."
> — [CITED: github.com/huggingface/peft `docs/source/developer_guides/lora.md`]

Option A as literally specified ("stack a NEW LoRA on `adapters/qwen3-30b-wp-30_70/`") requires exactly this prohibited operation: an existing `target_parameters` adapter + a second trainable `target_parameters` adapter on the same model. **Not supported in PEFT 0.18.1.**

A corroborating in-repo signal that the `target_parameters` expert-LoRA is fragile to *bind* (let alone stack): `scripts/eval_judge_unsloth.py` is a STUB whose docstring records that *"Unsloth's FastLanguageModel.from_pretrained() handles the MoE expert-LoRA binding that raw PEFT 0.18.1 cannot"* and that even vLLM serving of this adapter was an open Phase 0.3 risk [VERIFIED: `scripts/eval_judge_unsloth.py:1-23`].

### The only "continue the existing adapter" path Unsloth documents — and why it still isn't Option A

Unsloth DOES support reloading a saved LoRA adapter to continue training:

```python
# [CITED: unsloth.ai/docs/basics/continued-pretraining]
from unsloth import FastLanguageModel
model, tokenizer = FastLanguageModel.from_pretrained(
    model_name = "LORA_MODEL_NAME",   # path to a saved adapter dir
    max_seq_length = max_seq_length, dtype = dtype, load_in_4bit = load_in_4bit,
)
# "The optimizer state will be reset."
```

This continues training the **SAME** adapter's weights (not a new stacked adapter). It is a valid alternative, but it (a) mutates the proven Phase 3 30_70 adapter weights in-place rather than producing a clean, separately-named reasoning adapter, (b) inherits the `target_parameters` serving/eval fragility documented in the stub, and (c) does NOT match the ROADMAP's literal "new LoRA / `base_adapter:`" framing any better than B does. It offers no advantage over B and carries the MoE-LoRA-binding risk B avoids entirely.

### Why Option B is correct and de-risked

1. **Artifact exists & is canonical.** `models/qwen3-30b-wp-30_70-merged/` = 57GB, 13 safetensors shards [VERIFIED: `du -sh` + `ls`], referenced by the live serving script `scripts/serve_30_70_vllm.sh:39` and CONTEXT.md as the Option B base.
2. **Standard continued-FT recipe.** A merged model is a plain `Qwen3MoeForCausalLM`; a fresh single LoRA with `target_parameters` is the documented, supported configuration for Qwen3-MoE [CITED: PEFT lora.md — "Mixtral / Qwen3-MoE / GPT-OSS" with `mlp.experts.gate_up_proj` / `down_proj`].
3. **Minimal code delta.** Existing `train_model.py` already does `FastLanguageModel.from_pretrained(model_name=local_dir)` + `get_peft_model`. Option B = set `model.local_dir: models/qwen3-30b-wp-30_70-merged` in the new config. **Zero changes to the load/LoRA path.**
4. **No nested-LoRA uncertainty, no serving/eval binding risk** (Phase 4.4 wp-bench needs a merged checkpoint anyway).

### Pattern 1: Router-freeze verification in the dry-run path (RTRN-03) [PRIMARY]

The current `--dry-run` path prints a config summary but **NO trainable-parameter count** [VERIFIED: `train_model.py:401-427` `print_training_summary`, `:470-472` dry-run early-return]. Add a walker so the dry run produces the auditable evidence RTRN-03 requires:

```python
# NEW helper, called inside the dry-run branch (after apply_lora, before return)
def assert_router_frozen_and_report(model):
    trainable, total = 0, 0
    router_trainable = []
    import re
    for name, p in model.named_parameters():
        total += p.numel()
        if p.requires_grad:
            trainable += p.numel()
            if re.search(r"\bmlp\.gate\.", name):       # Qwen3MoeSparseMoeBlock gating linear
                router_trainable.append(name)
    print(f"  Trainable params: {trainable:,} / {total:,} ({100*trainable/total:.4f}%)")
    assert not router_trainable, f"ROUTER NOT FROZEN: {router_trainable[:5]}"
    print("  Router-freeze check PASSED — no mlp.gate.* parameter is trainable (RTRN-03)")
```

The planner writes a concrete verification check: dry-run output must contain "Router-freeze check PASSED" and a trainable% in the LoRA-expected range (single-digit %, NOT ~tens of % which would indicate embeddings/router leaked in).

### Pattern 2: Checkpoint parse-failure abort (RTRN-04) — Path C [RECOMMENDED]

`eval_judge.py` is a vLLM HTTP client and cannot evaluate a raw adapter checkpoint. RTRN-04 only needs a **parse-failure rate**, not Spearman. Recommended minimal new script (~80 lines):

```python
# scripts/checkpoint_parse_check.py  (run in unsloth_studio container, in-process)
# 1. FastLanguageModel.from_pretrained(merged_base) ; load_adapter(checkpoint_dir)  [Unsloth binds MoE expert-LoRA]
# 2. Sample 50 <wp_judge> prompts from data/reasoning_dataset/openai_val.jsonl
# 3. Generate; for each, call eval.eval_judge.parse_judge_response(text)  [REUSE existing parser]
# 4. parse_fail = count(parsed is None or "overall_score" not in parsed)
# 5. print rate; exit non-zero if parse_fail/50 > 0.05  -> run-training Step 7 aborts the run
```

Wire as a **post-checkpoint hook in run-training Step 7** (separate process — needs the full model), NOT as a `TrainerCallback`. This reuses `eval/eval_judge.py:parse_judge_response` so the parse logic stays identical to production eval.

### Pattern 3: LR / warmup for a ~700-example continued FT (RTRN-01)

- **LR:** Phase 3 = **2e-4** [VERIFIED: `config/train_config.yaml:50`]. 10x reduction → **2e-5** (within "≤2e-5"). Cosine scheduler (matches Phase 3, `lr_scheduler_type: cosine`).
- **Warmup:** total steps ≈ ceil(700 × epochs / eff_batch). At eff_batch 16, 2 epochs ≈ 88 steps. `warmup_ratio: 0.05` = ~4 steps — too few for stability. Use **absolute `warmup_steps: 20`** (~20-25% of total). Unsloth's "5-10% of total" rule degenerates at tiny step counts → prefer absolute steps. The grad-norm <3 target in the first 100 steps is directly served by a longer warmup at the low LR.
- **`SFTConfig` already supports `warmup_steps`** — but the current code only reads `warmup_ratio` [VERIFIED: `train_model.py:378`]. Planner: add a `warmup_steps` passthrough OR set `warmup_ratio: 0.20`.

### Anti-Patterns to Avoid

- **Applying PEFT's `effective_r = r // num_experts` rank rule.** With 128 experts that yields r=0. Phase 3 used flat **r=32**; copy it verbatim. The `rank_pattern` recommendation in PEFT docs is for fresh budget-matched setups, not for reproducing an existing adapter's architecture.
- **Re-saving `embed_tokens`/`lm_head` via `modules_to_save`.** The merged base already has Phase 3-tuned `<wp_gen>`/`<wp_judge>` embeddings baked in. Re-training them adds ~1GB trainable params and risks drifting the very tokens that already work. **Omit** (see Pitfall 4).
- **Confusing `output_router_logits` with router training.** `output_router_logits=True` only enables the load-balance aux-loss MONITOR. Router weights are frozen by being absent from `target_modules`/`target_parameters`. Keep the flag on (TRNG-04 continuity).
- **Reusing `config/train_config_30_70.yaml` as the template.** It is CONTAMINATED (see Pitfall 1).

## Common Pitfalls

### Pitfall 1: Templating from the contaminated `train_config_30_70.yaml`
**What goes wrong:** That file's `output_dir` = `qwen3-30b_experiment_001_20260411` (the DEAD experiment per D-04), and its hyperparameters do not represent the 30_70 winner.
**Why it happens:** Filename strongly implies it is the 30_70 config; it was overwritten/mislabeled.
**How to avoid:** Authoritative Phase 3 30_70 hyperparameters come from **`adapters/qwen3-30b-wp-30_70/adapter_config.json`** (r=32, alpha=64, dropout=0.0, target_modules + target_parameters) and **`config/train_config.yaml`** (LR 2e-4, cosine, warmup_ratio 0.05, batch 4, grad_accum 4, 2 epochs, max_seq 4096). Build `train_config_reasoning.yaml` from those, not from `train_config_30_70.yaml`.
**Warning signs:** Any 4.3 artifact referencing `experiment_001`.

### Pitfall 2: Router silently NOT frozen because dry-run never reports param counts
**What goes wrong:** RTRN-03 requires "router parameters excluded from the optimizer parameter count," but the current `--dry-run` prints no param counts at all — the freeze is invisible/unverifiable.
**Why it happens:** `print_training_summary` only echoes config; no `named_parameters()` walk.
**How to avoid:** Add Pattern 1's walker to the dry-run path; gate on "Router-freeze check PASSED".
**Warning signs:** Trainable% in tens-of-% (embeddings/router leaked) instead of single-digit %.

### Pitfall 3: Format collapse / generation regression on a tiny continued FT (MoE-specific)
**What goes wrong:** Low-data continued FT can overwrite the broad gen/judge distribution (catastrophic forgetting), collapsing output format or regressing PHPCS.
**Why it happens:** Small reasoning set + too-high LR + too-few warmup steps + overfitting; MoE routing already baked into merged weights can shift if effective LR is high.
**How to avoid (locked + recommended):** 15% replay (D-02) preserves base distribution; LR 2e-5 (10x down); `warmup_steps: 20`; 1-2 epochs (Unsloth: >3 epochs = diminishing returns + overfitting risk); watch train loss — **<0.2 indicates overfitting** [CITED: unsloth.ai LoRA hyperparameters guide]. Phase 4.4 REVL-02 (PHPCS within 2pp) and REVL-08 (length distribution) are the downstream catch-nets.
**Warning signs:** Train loss <0.2; val loss rising while train falls; checkpoint parse-failure climbing toward 5%.

### Pitfall 4: Re-tuning special-token embeddings via `modules_to_save`
**What goes wrong:** `<wp_gen>`/`<wp_judge>` embeddings (proven in Phase 3, baked into the merged model) get re-trained on a tiny reasoning set and drift.
**Why it happens:** Copy-pasting Phase 3's `modules_to_save=["embed_tokens","lm_head"]` (needed when the tokens were NEW) into a phase where they're already trained.
**How to avoid:** **OMIT `modules_to_save`** in Phase 4.3 — keep embeddings frozen at Phase 3 values. `[ASSUMED]` — surface in discuss-phase for confirmation.
**Warning signs:** +~1GB trainable params in dry-run; embedding-norm shift in telemetry.

### Pitfall 5: 8192 OOM / driver deadlock on DGX unified memory
**What goes wrong:** Doubling seq length (4096→8192) roughly doubles activation memory; a bad batch size can cause a driver-level deadlock (no clean CUDA OOM) on DGX Spark.
**Why it happens:** Unified memory pool shared by model/optimizer/dataloader/telemetry.
**How to avoid:** Start batch=1/grad_accum=16; Step-6a dry run + Step-6b warmup probe before committing; let `adaptive-planner` (Step 8.5) tune; gradient checkpointing already `"unsloth"`. Prefer lightweight monitor over 6-agent observe if headroom <15GB (observe adds ~3-4GB).
**Warning signs:** `mem_available_gb` < 2 in thermal JSONL; MemoryWatchdog graceful-exit triggers.

### Sampling Rate (Nyquist)
- **Gradient norm <3 (first 100 steps):** MLflow logs every step (`logging_steps` configurable, Phase 3 = 10). Per-step >> the 100-step window → over-sampled. To resolve the first-100-step requirement precisely, set `logging_steps: 5` for this run.
- **Loss divergence:** train loss per `logging_steps`; val loss per `eval_steps`. Both ≥ the cadence at which divergence becomes visible.
- **Parse-failure <5%:** at each (or every-other) `save_steps` checkpoint — the resolution at which the abort condition can act before wasting a full epoch.
- **Router freeze:** one-shot pre-train (dry run) — a static property, single sample sufficient.
- **OOM at 8192:** pre-train (dry-run + warmup probe) + per-step MemAvailable during training.
