## Summary

**GATING VERDICT (Item #1 — read first; everything else is conditional on this):**
**MoE-only targeting is SUPPORTED.** The Tinker SDK's `create_lora_training_client()` exposes
`train_attn: bool = True` and `train_mlp: bool = True` as first-class parameters
[VERIFIED: `.venv-tinker/.../tinker/types/lora_config.py` + `service_client.py`].
Setting `train_attn=False, train_mlp=True, train_unembed=False` produces a LoRA that targets
MoE expert linears (w1/w2/w3 = gate_proj/down_proj/up_proj across 128 experts × 48 layers)
while excluding attention q/k/v/o and the unembed_tokens projection. The existing SFT driver
(`tinker_reasoning_sft.py`) currently passes no targeting flags (v3 used all-default = all-linear
[VERIFIED: `tinker_reasoning_sft.py` line 190]). The code change to implement D-N1 is a
one-line addition to the `sc.create_lora_training_client(...)` call.

There is **no finer-grained targeting within MoE** (e.g. per-layer subset or router-only):
`train_mlp=True` enables all MoE expert projection matrices uniformly. The router/gate-score
linears are NOT in the Tinker LoRA surface — the v3 adapter contains no router tensors
[VERIFIED: `merge_tinker_v3.py` docstring + key patterns].

The corrective retrain is a rank×replay grid (9 candidates) run on Tinker using the existing
SFT driver with two new flags. Each candidate is evaluated for judge survival offline on the
Tinker sampler (cheap pre-merge filter), then merged via a **modified** `merge_tinker_v3.py`
(the unmodified script hard-fails on MoE-only input at three specific assert lines), then
evaluated against the REVL-04 wp-bench HARD gate. A confirmation re-run of the winner provides
the statistical hardening required by D-N5.

**BLOCKING OPEN QUESTION (must be resolved before the selection rule is encoded):** D-N7
specifies "CI-lower ≥ 0.263," but at the expected n_pairs ≈ 113, achieving CI-lower ≥ 0.263
requires a **point** Spearman ≥ 0.427 [VERIFIED: Fisher-z at n=113]. v3 itself has a point
estimate of 0.263, which yields CI-lower ≈ 0.082 — far below the bar. The MoE-only probe
(0.3124, explicitly rejected as "too harsh") gives CI-lower ≈ 0.136, still well below 0.263.
In effect, "CI-lower ≥ 0.263" at n=113 is *stricter* than the bar D-N7 rejected. Either (a)
D-N7 intends point-estimate ≥ 0.263 with CI-lower used only as a noise guard on wide-CI
measurements, or (b) the judge eval-n must be expanded (e.g. 3× val set → n≈340 → CI-lower
≥ 0.263 at point ≈ 0.34). The planner MUST surface this to the user before encoding the
selection rule — see Open Questions #1.

**Primary recommendation:** Implement D-N1 via `train_attn=False, train_unembed=False` in the
existing SFT driver; modify `merge_tinker_v3.py` to detect MoE-only adapters and skip the
attention+unembed stages before running any grid merge; resolve D-N7 bar semantics with the
user before scripting the selection rule; run the 9-candidate grid; apply the pre-merge judge
filter before incurring merge+wp-bench cost.

A MoE-only adapter (train_attn=False, train_unembed=False) will contain ONLY:

No `self_attn.q_proj/k_proj/v_proj/o_proj` tensors. No `unembed_tokens` tensors.
Router/gate-score linears are NOT targeted — the Tinker MoE surface is expert projections only.

## Item #2: wp-bench n for CI Hardening (D-N5)

### Full suite capacity

`config/wp-bench.yaml` has `limit: null` = 344 tests per run. [VERIFIED: codebase]
The baseline 0.4537 was measured in a single 344-test run (`output/eval_reasoning_v3/04.4_wp_bench_results.json`).

### Wilson CI analysis

```
Relevant gap to resolve: baseline 0.4537 vs interim MoE-only 0.4071 = 0.047

n=344, p≈0.45: Wilson CI half-width ≈ 0.052
```

The full-suite half-width (0.052) slightly exceeds the 0.047 gap — a single run cannot
guarantee non-overlapping CIs between a candidate at the baseline level and the interim level.
Within-grid candidates differing by ~0.02 are entirely unresolvable by a single run.

### Recommendation

**Run one full 344-test wp-bench pass per candidate** (matches baseline measurement
methodology — correct for the HARD gate point-estimate comparison).

**Run a confirmation re-run (second full 344-test pass) on the grid winner before promotion.**
Averaging two runs gives n_eff = 688, half-width ≈ 0.037 — the statistical hardening D-N5
requires. This is the correct "wider n" strategy: the suite is fixed at 344 tests; multiple
runs are the variance-reduction mechanism.

Cost: 9 initial runs (grid screening) + 1 confirmation run (winner) = 10 total wp-bench runs.
With `concurrency: 4` in the config, each run takes roughly the same wall-clock as a prior
04.4 run.

### Spearman CI — key finding that feeds the D-N7 blocking question

[VERIFIED: Fisher-z computation at n=113, python3 on live machine]

```
n_pairs ≈ 113 (val set wp_judge rows after parse filtering, from dit02 result)

At n=113:
  v3 point rho=0.263  → CI-lower = 0.082
  probe rho=0.3124    → CI-lower = 0.136
  Required point rho for CI-lower ≥ 0.263 = 0.427
```

D-N7's literal "CI-lower ≥ 0.263" at n≈113 means any candidate must have **point Spearman ≥ 0.427**
to pass. This is stricter than the 0.3124 bar D-N7 explicitly rejected (D-N7: "would reject
the low-rank candidates and collapse the grid to rank-32, defeating the codegen purpose").
v3 itself fails this bar with its own point estimate (0.263 → CI-lower 0.082).

**Two consistent interpretations of D-N7:**

| Interpretation | Bar | Implication |
|----------------|-----|-------------|
| **A: point ≥ 0.263** | Accept if point Spearman ≥ v3's 0.263 | Matches the "do no harm vs v3" intent; CI-lower used only to flag high-variance measurements for re-run |
| **B: larger n** | Keep "CI-lower ≥ 0.263" literal, but expand judge eval-n | At n≈340 (3× val set), point ≈ 0.34 clears CI-lower ≥ 0.263; requires building a larger judge response pool |

Both interpretations are consistent with the stated purpose. **Planner must surface this to the
user as a blocking clarification before encoding the selection rule.** Until resolved, plan
the selection rule as two conditional branches (A vs B), not a single hard gate.

### D-N4 replay semantics — two interpretations [ASSUMED: must be confirmed]

D-N4 reads: "Center wp_gen codegen replay at ~30%; sweep replay axis {15%, 30%, 50%}."
The parenthetical: "(50% = the v1.2 original 30% judge + 20% wp_gen.)" introduces ambiguity:

| Reading | 15% mix | 30% mix | 50% mix |
|---------|---------|---------|---------|
| **Reading X: wp_gen-only axis** | 73 wp_gen (current) | ~217 wp_gen | ~508 wp_gen |
| **Reading Y: mixed-source at 50%** | 73 wp_gen (current) | 73 wp_gen + additional | 30% judge replay + 20% wp_gen replay |

**Reading X** treats the axis as "total replay %, filled from the wp_gen pool" (supported by D-N4's "more wp_gen = protect codegen" framing).

**Reading Y** treats the parenthetical as prescriptive: the v1.2 50% came from a 30% judge + 20% wp_gen mix, and D-N4 is re-instating that specific mix at the 50% point.

## Item #5: Grid Short-Circuit / Early-Stop (D-N5 — optional)

User de-prioritized cost. Most useful short-circuits:

- **Per-epoch collapse guard:** `--per-epoch-eval-n 8` already in the SFT driver; if an
  epoch shows severe judge collapse (parse rate > 50%), abort remaining epochs of that cell.
- **Rank-dimension elimination:** if rank-8 fails the judge gate at all replay variants,
  rank-8 is ruled out — skip its merge+wp-bench. No structural change needed.
- **Winner fast-path:** if one candidate clearly dominates all others on the pre-merge judge
  score AND its first wp-bench run clears baseline by a meaningful margin (>0.02), run the
  confirmation pass immediately rather than completing remaining grid cells.

These are execution-order optimizations; mark as optional branching steps in the plan, not
mandatory tasks.

## Common Pitfalls

### Pitfall 1: Unmodified merge script hard-fails on MoE-only adapter
**What goes wrong:** `merge_tinker_v3.py` hits `AssertionError: no self_attn LoRA tensors found`
at line 281 of `_build_attention_only_adapter`.
**Why it happens:** Script was written for v3's all-linear adapter; no MoE-only code path exists.
**How to avoid:** Modify the script (add `is_moe_only` detection, gate steps 4+5) as a Wave 0
prerequisite before any grid merge task.
**Warning signs:** Immediate crash on first merge attempt with a clean adapter.

### Pitfall 2: Encoding the judge bar without resolving D-N7 semantics
**What goes wrong:** The selection rule either rejects the entire grid (if "CI-lower ≥ 0.263"
is applied literally at n=113, requiring point ≥ 0.427) or uses a bar the user didn't intend.
**Why it happens:** At n=113, Fisher-z gives CI-lower = 0.082 for v3's own point estimate;
the literal bar is near-unpassable. See Item #2 analysis.
**How to avoid:** Planner surfaces Open Question #1 to user BEFORE scripting the selection
rule. Plan the selection script with a configurable bar parameter, not a hardcoded 0.263.
**Warning signs:** All pre-merge judge gate runs fail despite reasonable Spearman values (0.2-0.3).

### Pitfall 3: Judge bar at 0.3124 collapses the grid
**What goes wrong:** No low-rank candidate can clear the MoE-only probe Spearman (0.3124)
because rank reduction erodes judge skill below the probe level. Grid collapses to rank-32 only.
**Why it happens:** 0.3124 was the full-rank-32 MoE-only probe result; D-N7 explicitly chose 0.263.
**How to avoid:** Use D-N7's intended bar (point 0.263 or CI-lower per clarified semantics). A
candidate that also clears 0.3124 wins on tie-break, but the bar must not require it.

### Pitfall 4: Baseline confusion (0.4537 vs 0.4857)
**What goes wrong:** The HARD gate uses the wrong baseline, either accepting a candidate that
would fail the real comparison or rejecting one that would pass.
**Why it happens:** Two different wp-bench runs: dit02 probe used `WPBENCH_LIMIT=30` (produced
0.4857 for baseline_v2 in `dit02_attribution_result.json`); the HARD gate baseline 0.4537
comes from the full 344-test run in `output/eval_reasoning_v3/04.4_wp_bench_results.json`.
**How to avoid:** Run REVL-04 with `limit: null` (the default in `wp-bench.yaml`). Gate value is 0.4537.

### Pitfall 5: 50% replay may exhaust the phase1 pool
**What goes wrong:** The 50% replay variant needs ~423 additional wp_gen rows. If the phase1
pool lacks distinct functions after dedup and leakage guard, the variant cannot be built at 50%.
**Why it happens:** The exact available count has not been verified; only "large" is known.
**How to avoid:** Wave 0 task — count distinct functions in phase1 pool, subtract already-used
73, verify remainder ≥ 423 before committing to 50% as a grid axis point.

### Pitfall 6: RC-A harness fix not active during eval
**What goes wrong:** Qwen3's default chat template enables thinking. Without `enable_thinking=False`
at both eval_judge call sites, merged model emits unclosed `<think>` blocks → parse failures
masquerade as judge regression → candidates are incorrectly rejected.
**Why it happens:** RC-A was a real eval harness bug, confirmed and fixed (commit b88faa3).
**How to avoid:** Confirm RC-A fix is present in the eval harness before any REVL-04 run. The
post-merge judge confirm at step 2 of the pre-merge filter uses the Tinker sampler
(train_attn=False in training means thinking disabled), but the post-merge vLLM serve requires
the explicit harness fix.

### Wilson CI terse gate (existing — RTRN-05)
```python
# Source: VERIFIED — tinker_reasoning_sft.py:63-71
def _wilson_upper(k, n, z=1.96):
    p = k / n
    denom = 1 + z*z/n
    center = (p + z*z/(2*n)) / denom
    half = (z * math.sqrt(p*(1-p)/n + z*z/(4*n*n))) / denom
    return center + half
# Gate: rate <= 0.10 AND wilson_upper <= 0.15
```
