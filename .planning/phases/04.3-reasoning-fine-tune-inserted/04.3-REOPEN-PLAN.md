## Step 4 — Terse-rate compare (CPU) — reuse capture histogram + Wilson CI

Terse = no `[/REASONING]` close tag (bare-JSON judge mode). For each capture:
`terse = n_total - n_with_close_tag`; rate = terse/n_total; add Wilson 95% CI (n≈120 per
checkpoint — small, so the CI matters). Write `checkpoint_curve/summary.md`.

| ckpt-50 terse rate (Wilson upper) | Root cause | Action |
|---|---|---|
| ≤10% (upper ≤15%) | late collapse only | **STOP retrain** — ckpt-50 is the candidate → Step 6 gate → REVL-03 → REVL-05 |
| >10%, worse at 72 than 50 | LR/epoch overshoot late | **Path A** (Step 5A) |
| >10%, flat 50≈72 | format baked from epoch 1 | **Path B** (Step 5B) |

**PRE-REGISTERED escalation rule (Claude peer-review; commit to this BEFORE seeing numbers
— no post-hoc "looks good enough").** n≈120/checkpoint is thin. If the ckpt-50 and ckpt-72
terse-rate Wilson 95% CIs **overlap at all**, the bisect is INCONCLUSIVE → automatically
widen to **n≥300** (val is 120 cot+ctf; pull train cot+ctf to reach 300, identical slice for
BOTH checkpoints — re-capture ckpt-72 too so the sets match) and recompute before any
decision-table action. Do not read a checkpoint as "cleaner" on overlapping CIs. This
mirrors the REVL-03 CI-straddle that produced the marginal call — same discipline, applied
up front.
