## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 2 - Security/Correctness] Body-keyed leakage guard instead of user-message-keyed**

- **Found during:** Task 2 design (advisor pre-check)
- **Issue:** The plan says "copy the `_codes()` leakage guard VERBATIM" — `_codes()` keys on user-message content. For `build_augmented_train.py`'s wp_judge negatives, the PHP code IS the user message (so it works). For wp_gen rows, code lives in the ASSISTANT message; user message is `<wp_gen> {description}`. A verbatim user-message guard against sentinel is provably vacuous (sentinel is 100% wp_judge format, different message structure entirely). Against val it would only catch description string collisions, not code reuse.
- **Fix:** Primary leakage guard keyed on assistant content (function body). The verbatim `_codes()` guard kept as belt-and-suspenders on user-message descriptions (vacuous for sentinel, harmless for val). Pool exclusion during sampling (before writing) is also keyed on body content.
- **Files modified:** scripts/build_replay_mix.py
- **Commit:** bfc12e4

**2. [Rule 3 - Blocking] .gitignore exception required for output/tinker/**

- **Found during:** Task 1 commit
- **Issue:** `output/` is gitignored; `output/tinker/phase1_pool_count.json` could not be staged without `-f`. No existing exception for `output/tinker/`.
- **Fix:** Added `!output/tinker/` and `!output/tinker/**` negation rules to `.gitignore`, then staged with `-f` (negation pattern requires both dir and glob for git to honour it on existing ignore).
- **Files modified:** .gitignore
