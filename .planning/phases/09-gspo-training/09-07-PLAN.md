<objective>
Make the GSPO gradient step actually run on Tinker with a REAL importance-sampling ratio.

A live 1-step smoke proved the entire RL path end-to-end (Tinker auth, sampling, dual gen/judge
reward, advantage assembly) EXCEPT the gradient step: `tc.forward_backward_custom` was handed
plain dicts (`{completion,reward,advantage,group_id}`) instead of `tinker.Datum` objects
(`AttributeError: 'dict' object has no attribute 'loss_fn_inputs'`), and the rollout discarded
the sampled logprobs — so even with a Datum the loss would fall into its
`except (AttributeError, KeyError)` branch and set `seq_ratio = 1.0`, degrading GSPO to plain
advantage-weighted REINFORCE with no IS correction (the whole point of GSPO/RSPO).

This plan adopts the cookbook trajectory/datum pipeline
(`tinker_cookbook.rl.data_processing.trajectory_to_data` + `compute_advantages` +
`assemble_training_data`) to assemble real Datums carrying `target_tokens`, `logprobs`,
`advantages`, and `mask`, and captures sampled tokens+logprobs from `sample()` into the rollout.

The judge/reward/serving/entrypoint wiring is ALREADY fixed (commit ea728a4) and is OUT OF SCOPE.
This plan is ONLY the Datum + sampling-logprob assembly. The reward path
(`reward_pipeline.py`, `rl_judge_dispatch.py`), KL-halt ordering (CR-04 / GRPO-08), the
entrypoint, and judge dispatch are UNTOUCHED.

Purpose: GSPO trains with a real sequence-level IS ratio on Tinker (D-09-03 primary objective).
Output: a green offline datum-schema guard, the refactored rollout/train modules, updated tests,
and a passing 1-step live re-smoke whose loss fn uses real logprobs (NOT the 1.0 fallback).
</objective>

  with `.get_total_rewards() -> list[float]` (sum of per-transition reward + final reward)

From tinker_cookbook/completers.py:
- `TokensWithLogprobs(tokens: list[int], maybe_logprobs: list[float] | None, stop_reason="stop")`
  with property `.logprobs` that RAISES ValueError if `maybe_logprobs is None`

From tinker_cookbook/rl/data_processing.py (the canonical pipeline this plan adopts):
- `remove_constant_reward_groups(groups: list[TrajectoryGroup]) -> list[TrajectoryGroup]`
  IMPORTANT: when ALL groups are constant it returns `groups[0:1]` (a SINGLETON), NOT `[]`.
- `compute_advantages(groups: list[TrajectoryGroup]) -> list[torch.Tensor]`
  centers PER GROUP: `advantages_G = rewards_G - rewards_G.mean()` (this preserves CR-06).
- `assemble_training_data(groups, advantages_P) -> (list[tinker.Datum], list[dict])`
  delegates to `trajectory_to_data`, returns (data_D, metadata_D), `len(data_D)==len(metadata_D)`.
- `trajectory_to_data(traj, traj_advantage) -> list[tinker.Datum]` bakes, per action token, into
  `datum.loss_fn_inputs`: `target_tokens`, `logprobs` (sampled), `advantages` (traj_advantage on
  action positions, 0 on obs positions), and `mask` (1.0 on action positions, 0.0 on obs).
  Single-turn WP completion = 1 Transition per Trajectory = exactly 1 Datum per trajectory.

From tinker SamplingClient.sample(prompt, num_samples, sampling_params) -> future:
- `future.result().sequences` is `list[SampledSequence]`; each `seq.tokens: list[int]` and
  `seq.logprobs: list[float]` (sampled-token logprobs returned BY DEFAULT, no extra flag —
  confirmed by TinkerTokenCompleter asserting logprobs is not None after a plain sample call).
