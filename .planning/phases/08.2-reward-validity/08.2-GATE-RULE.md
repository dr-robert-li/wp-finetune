**Decision:** D-08.2 (locked 2026-07-01)
**Enforced by:** `scripts/reward_validity_gate.py` (imports `scripts/_reward_validity_oracle.py`)
**Regression test:** `tests/test_reward_validity_gate.py`

## The Rule

> **No reward goes to GPU/Tinker until its offline oracle correlation CI-lower > 0.**

A candidate reward is VALID iff its per-checkpoint trajectory rank-correlates with the
validated downstream target (judge teacher-Spearman) across the existing seedA checkpoint
captures, with a 95% bootstrap confidence interval lower bound strictly greater than zero.

This is the gate Phase 08.1 was missing. Phase 10 RLEV proved the cost of skipping it:
seedA's `fix_correctness` proxy rose +0.028 while teacher-Spearman stayed flat and
wp-bench codegen regressed −0.049 — textbook Goodhart. The oracle confirms the root cause
empirically: fix-correctness trajectory had near-zero (in fact negative) correlation with
the target trajectory. The gradient pointed the wrong direction.

with CI-lower > 0.

## How to Run the Gate

### Test a built-in form (Plans 03, 04, future candidates)

```python
from scripts.reward_validity_gate import run_validity_gate, GateResult

result = run_validity_gate("pairwise_rank_agreement")
print(result)
# GateResult(form='pairwise_rank_agreement', spearman=0.700, ci=[0.147,0.935], valid=True, n_ckpts=11)  [VALID]

assert result.valid, f"Gate FAILED: {result.form} ci_lo={result.ci_lo:.3f} <= 0"
```

### Register and test a new candidate form

```python
from scripts.reward_validity_gate import run_validity_gate

def my_candidate_reward(model: list[float], gt: list[float]) -> float:
    """Custom reward: (model_vec, teacher_gt_vec) -> scalar per-checkpoint reward."""
    ...

result = run_validity_gate("my_candidate", form_fn=my_candidate_reward)
if not result.valid:
    raise RuntimeError(
        f"my_candidate did not pass the reward-validity gate "
        f"(ci_lo={result.ci_lo:.3f} <= 0). Do not proceed to GPU."
    )
```

### Add a new candidate to the FORMS dict (optional, for repeated use)

Edit `scripts/_reward_validity_oracle.py`, add your function, and add it to the `FORMS`
dict. Run `python3 scripts/_reward_validity_oracle.py` to regenerate the oracle JSON with
the new form included.

## Important Caveats

### n=11 / one-run wide CI

The oracle uses 11 checkpoints from a SINGLE seedA run. All checkpoints are near each
other in weight space (same run, sequential). This gives **wide confidence intervals** —
the bootstrap CI reflects genuine uncertainty at this sample size.

**This gate is a NECESSARY SCREEN, not a high-power statistical estimate.**

- A form passing the gate (CI-lower > 0) may still fail in a new run.
- A form failing the gate (CI includes 0) is almost certainly not tracking the target.
- The real empirical test is Plan 05's gated smoke (50/250-step 2-seed RL run with
  within-run teacher-Spearman trend + codegen trip-wire). Do NOT treat passing the offline
  gate as sufficient evidence on its own.

### Val held out (T-09-LEAK)

The oracle uses `openai_val.jsonl` GT for the trajectory correlation. Val GT MUST stay
held out from the reward path. Reward-time GT comes from `openai_train.jsonl` only (see
`scripts/build_reward_gt_sidecar.py` and `data/rl_probe/judge_gt_sidecar.jsonl`).

### Content-hash join (T-082-02)

The sidecar GT-join uses SHA-256 of whitespace-normalized PHP code-under-review as the
join key — NOT the positional `prompt_id`. A pool reorder cannot silently corrupt labels.
