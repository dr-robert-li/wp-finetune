<objective>
Deliverable 4 (RVAL-04 / SC4): an offline reward weight/form sweep on data/rl_probe/judge_probe_corpus.jsonl,
scored by the standing oracle gate, that selects the highest reward<->teacher-correlation form with no codegen
penalty. Reuses the 08.1 lever/MEASUREMENT framework. This is where the parameterized machinery from Plan 03
gets its winning (form, weight) — the config a future gated Plan-05 smoke would carry.

Advisor-locked framing (two corrections that shape the sweep):

1. DUAL-LENS SELECTION. Oracle validity alone will happily pick a gradient-DEAD form (a reward can be
   oracle-VALID globally yet uniform within groups under the 90-100 saturation). So select on BOTH:
   (a) oracle validity — run_validity_gate CI-lower>0 (the weight screen: calib_weight->0 gives fix_correctness
   INVALID -0.24; ->1 gives pairwise VALID +0.70; the sweep finds the mix that flips the COMPOSITE trajectory to
   CI-lower>0), AND (b) 08.1 gradient density — frac_mid / group-reward-std / frac_groups_all_zero on the
   GT-attached probe corpus (the exact 08.1 lens). Winner = VALID and maximally gradient-dense.

2. CODEGEN IS OFFLINE-UNVERIFIABLE. The oracle is codegen-blind (it only sees the judge-Spearman target). So the
   "no codegen penalty" clause CANNOT be proven offline (that needs wp-bench serving = GPU = Plan 05). Substitute
   a STRUCTURAL PROXY offline: the calibration term must not fire (reward high) on non-PHP / verbose / echo-adversary
   outputs — reuse _is_valid_wp_php + the anti-hack + an echo-adversary <=0.30 check. State plainly that the real
   codegen guarantee is deferred to Plan 05's smoke trip-wire; do NOT claim an offline codegen proof.

TWO LENSES RUN ON TWO CORPORA (critical — they are NOT the same dataset):
- ORACLE-VALIDITY lens: the oracle correlates a reward's TRAJECTORY across the 11 checkpoints
  (output/rl_eval/*/judge_responses.jsonl, warmstart..step-500) vs the target trajectory. This needs the
  checkpoint axis — the probe corpus is a SINGLE snapshot and CANNOT produce a trajectory. So the oracle lens
  runs through run_validity_gate on the 11-checkpoint captures, exactly as Plan 01/03 do. It KEEPS USING VAL GT:
  the oracle is an offline diagnostic that never touches training, val is its held-out target, and val-GT there is
  what makes the validity test honest (using TRAIN GT in the oracle would validate the form on the same GT it is
  built from -> circular). TRAIN-GT-only governs the REWARD the model optimizes (the gradient lens + reward-time),
  NOT the held-out oracle diagnostic.
- GRADIENT-DENSITY lens: frac_mid / group-reward-std / frac_groups_all_zero on the PROBE CORPUS — the only corpus
  with G=4 rollout groups. Parse raw_text -> model_overall (parse_judge_scores), attach TRAIN GT via the
  judge_gt_sidecar content-hash join (Plan 01). This is the reward-time path, so TRAIN GT is correct here.
No regeneration of the probe corpus. $0 / CPU only.

Purpose: pick the single (form, weight) that is oracle-VALID (weight screen), maximally gradient-dense (the real
selector), and echo/verbose-robust. Output: reward_form_sweep.py, sweep_results.json (ranked + selected), writeup.
</objective>
