  <action>
**DO NOT run `python -m scripts.phase2_judge` — it calls the Anthropic API.**

Spawn 3-4 parallel Claude Code agents to judge synthetic examples.

**Input:** All JSON files in phase2_synthetic/output/generated/
**Output:** phase2_synthetic/output/judged/{tag}.json for each gap tag

Each agent:
1. Receives a subset of generated JSON files to judge
2. Reads config/judge_system.md for the full rubric
3. For each synthetic function, evaluates against the 9-dimension rubric and produces:
```json
{
  "function_name": "...",
  "file_path": "...",
  "verdict": "PASS" or "FAIL",
  "scores": {
    "wpcs_compliance": 1-10,
    "sql_safety": 1-10,
    "security": 1-10,
    "performance": 1-10,
    "wp_api_usage": 1-10,
    "code_quality": 1-10,
    "dependency_integrity": 1-10,
    "i18n": 1-10,
    "accessibility": 1-10
  },
  "critical_failures": [],
  "training_tags": [],
  "notes": "..."
}
```
4. PASS requires ALL scores >= 8, no critical failures. Security auto-FAIL: security < 5 forces FAIL.
5. **Revision for failures:** If a synthetic example FAILs, the agent attempts ONE revision:
   - Identify the specific issues from scores and notes
   - Rewrite the code to fix those issues
   - Re-assess the revised version
   - If revision also fails, discard the example
6. Adds the assessment to each function object
7. Writes only PASSED functions (original or revised) to phase2_synthetic/output/judged/{tag}.json

**Agent assignment:** Divide generated files roughly equally by total function count.

**Key rules:**
- Be strict: this is training data quality control. When in doubt, FAIL.
- Rejection examples (tagged rejection:proactive_*) should PASS if they correctly add security measures — the "rejection" is about the prompt not mentioning security, not about rejecting the user
- N/A scoring: i18n=7, accessibility=7 when not applicable

  <action>
**DO NOT run `python -m scripts.phase2_judge_dataset` — it calls the Anthropic API.**

Spawn 3-4 parallel Claude Code agents to create rubric-scored judge training data.

**Input sources (mix of quality levels):**
1. Phase 1 passed code (high-quality, should score high)
2. Phase 1 failed code (low-quality, should score low with specific defects identified)
3. Phase 2 mutated code (controlled defects, medium-low scores)
4. Phase 2 judged synthetics (high-quality, should score high)

**Output:** phase2_synthetic/output/judge_training/ with JSON files

Each agent:
1. Receives a batch of code samples from the mixed sources above
2. For each code sample, produces a 6-dimension rubric score (0-100 scale, different from the 9-dim 1-10 judge):
```json
{
  "code": "the PHP code",
  "source": "phase1_passed|phase1_failed|mutated|synthetic",
  "source_repo": "repo name",
  "function_name": "...",
  "scores": {
    "wpcs_compliance": 0-100,
    "security_score": 0-100,
    "performance_score": 0-100,
    "i18n_score": 0-100,
    "accessibility_score": 0-100,
    "overall_quality": 0-100
  },
  "reasoning": "Detailed explanation of each score, citing specific code patterns",
  "training_tags": ["relevant taxonomy tags"]
}
```
3. Scoring guidance:
   - Passed code: expect 70-95 range across dimensions
   - Failed code: expect 20-60 range, with specific dimensions scored low based on failure reason
   - Mutated code: score the defect dimensions low (e.g., SQL injection mutation -> security_score: 10-30)
   - Explain reasoning for each score — this reasoning becomes training data for the judge model
4. Writes output to phase2_synthetic/output/judge_training/{source_type}_{batch}.json

**Agent assignment:**
- Agent 1: Sample ~500 functions from Phase 1 passed (high-quality subset)
- Agent 2: All Phase 1 failed functions (all 24 repos)
- Agent 3: All mutated examples + all judged synthetics
- (Optional Agent 4 if volume warrants it)

**Target:** At least 1,000 rubric-scored examples for judge training data. Aim for roughly equal representation of high/medium/low quality code.
