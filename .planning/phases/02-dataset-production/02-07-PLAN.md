  <action>
**DO NOT run `python -m scripts.phase3_cot` — it calls the Anthropic API.**

CoT generation transforms all pipeline outputs into instruction-response training pairs with reasoning chains. Spawn 5 parallel Claude Code agents.

**Input sources (all pipeline outputs):**
1. phase1_extraction/output/passed/ — extracted, judge-passed real WordPress functions (wp_gen examples)
2. phase2_synthetic/output/judged/ — quality-controlled synthetic examples (wp_gen examples)
3. phase2_synthetic/output/mutated/ — contrastive bad->good pairs (wp_gen examples with reasoning about what's wrong)
4. phase2_synthetic/output/judge_training/ — rubric-scored examples (wp_judge examples)

**Output format — each example is one JSONL line in final_dataset/wordpress_finetune.jsonl:**
```json
{
  "messages": [
    {"role": "system", "content": "You are a senior WordPress developer..."},
    {"role": "user", "content": "<wp_gen> Write a WordPress function that..."},
    {"role": "assistant", "content": "Let me think through this step by step...\n\n```php\n...\n```"}
  ],
  "metadata": {
    "task_type": "gen" or "judge",
    "source": "phase1_passed|synthetic|mutated|judge_training",
    "training_tags": [...],
    "sample_weight": 1.0
  }
}
```

**For wp_gen examples (Phase 1 passed + Phase 2 judged + mutated):**
Each agent:
1. Reads the code from the source
2. Reverse-engineers a natural instruction/task prompt (what would a developer ask to get this code?)
3. Wraps the response in CoT reasoning: "Let me think through this step by step. First, I need to consider security... Then, for performance... Here's the implementation:"
4. Prefixes user message with `<wp_gen>` task token
5. For contrastive/mutated pairs: include reasoning about what the bad version does wrong and why the fix matters
6. For rejection examples (tagged rejection:proactive_*): instruction does NOT mention security, but response proactively adds it with explanation

**For wp_judge examples (Phase 2 judge training data):**
Each agent:
1. Takes the code sample and rubric scores
2. Creates an instruction: "Evaluate this WordPress code for quality across these dimensions: ..."
3. Prefixes user message with `<wp_judge>` task token
4. Response includes the structured rubric scores WITH reasoning for each dimension
5. Response must explain WHY each score was given, citing specific code patterns

**Agent assignment (5 agents):**
- Agent 1: Phase 1 passed repos A-G (alphabetical first third)
- Agent 2: Phase 1 passed repos H-Q (alphabetical second third)
- Agent 3: Phase 1 passed repos R-Z + wordpress-develop subset (sample ~2000 of 11K core functions)
- Agent 4: All Phase 2 judged synthetics + mutated pairs + rejection examples
- Agent 5: All Phase 2 judge training data (these become wp_judge examples)

Each agent writes its output as JSONL to phase3_cot/output/agent_{N}.jsonl

**After all agents complete, combine into final file:**
```bash
cd /home/robert_li/Desktop/projects/wp-finetune
cat phase3_cot/output/agent_*.jsonl > final_dataset/wordpress_finetune.jsonl
echo "Total lines: $(wc -l < final_dataset/wordpress_finetune.jsonl)"
```

**IMPORTANT:** For wordpress-develop (11K+ functions), do NOT process all of them. Sample ~2000 diverse functions to keep the dataset balanced. Otherwise core code would dominate.

**Sample weight guidance:**
- Normal examples: sample_weight = 1.0
- Contrastive/mutated pairs: sample_weight = 1.5 (weighted higher per user decision)
- Rejection examples: sample_weight = 1.5
- Judge training examples: sample_weight = 1.0
