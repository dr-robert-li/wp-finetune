**Public functions (exported API):**

1. `classify_power_zone(avg_watts, peak_temp, gpu_util, thresholds) -> str`
   - Returns one of: "THROTTLED", "CAPPED", "TARGET", "MODERATE", "UNDERUTILIZED"
   - avg_watts=None -> fallback to gpu_util proxy: <30% UNDERUTILIZED, <70% TARGET, <90% MODERATE, else CAPPED
   - avg_watts >= capped_min_watts (80) -> CAPPED
   - avg_watts >= target_min_watts (50) -> TARGET
   - avg_watts < 50 -> UNDERUTILIZED
   - TARGET refined to MODERATE if has_batch_headroom AND effective_headroom_gb > 0
   - THERMAL OVERRIDE: peak_temp >= warning_temp (82) -> THROTTLED regardless of watts

2. `compute_batch_coupling(effective_batch, new_batch) -> int`
   - **CRITICAL (addresses HIGH review concern #1):** Uses `round(effective_batch / new_batch)` NOT `max(1, effective_batch // new_batch)`.
   - Always returns >= 1.
   - Result: new_accum such that `new_batch * new_accum` is within +/-1 of effective_batch.
   - Formula: `max(1, round(effective_batch / new_batch))`
   - Examples: eff=16, batch=8 -> accum=2 (exact). eff=16, batch=5 -> accum=3 (eff=15, drift=1). eff=16, batch=3 -> accum=5 (eff=15, drift=1).

3. `prefer_divisor_batch(effective_batch, current_batch, step_size, ceiling) -> int`
   - When proposing a batch increase, prefer batch sizes that are exact divisors of effective_batch.
   - If current_batch + step_size is a divisor of effective_batch AND <= ceiling, return it.
   - If not, check current_batch + step_size +/- 1 for a nearby divisor <= ceiling.
   - Fallback: return current_batch + step_size (allow bounded drift).
   - Example: eff=16, current=4, step=2, ceiling=8 -> 8 is divisor of 16, but first try 6: not a divisor. Try 8: divisor. Return 8? No -- step_size=2 means propose 6 first. 6 is not a divisor. Check 5 (divisor? no), 7 (no), 8 (yes, divisor). Since 8 is within step_size+2 of current, return 8.
   - Actually simpler: propose = current + step_size. If propose divides eff evenly, use it. Else scan propose-1, propose+1 for a divisor that is > current and <= ceiling. If found, use it. Else use propose (accept drift).

4. `apply_ladder(zone, config, telemetry_agg, scale_info, anchor_store, thresholds) -> dict`
   - Applies Rung 1-5 in order per ADPT-02.
   - Returns dict of changes: {key: (old_val, new_val, reason)}
   - Rung 1 (batch): uses prefer_divisor_batch() for proposal, compute_batch_coupling() for accum.
   - Rung 2-5: prefetch, workers, save_steps, eval_steps as in CONTEXT.md.
   - I/O Bottleneck Guard: if gpu_util <30% AND watts <30W, skip Rung 1.

5. `load_unsloth_actuals(actuals_path) -> dict | None`
   - Reads telemetry/training/_unsloth_actuals.json if exists.
   - Returns dict with actual_batch, actual_grad_accum, config_batch, config_grad_accum.
   - Deletes file after reading (consumed).
   - Returns None if file doesn't exist.

6. `plan_adaptation(thermal_log_path, ratio, config_path, adaptive_config_path) -> dict`
   - Main entry point. Orchestrates: load config, parse JSONL, check Unsloth actuals, classify failure, route by power zone, apply ladder, write config, write log.
   - Returns summary dict with: power_zone, failure_mode, changes, unsloth_override.
