## REVL-06 — disposition: N/A (Option C), recorded 2026-06-02

**Basis:** 0 `<corrected_code>` emissions across 478 eval rows (full cot+ctf set, train+val).
The Phase-4.3 model is judge-only — trained on `<wp_judge>`→`<judge_output>` reasoning, NOT
on CtF fix generation. It was never trained to emit `<corrected_code>`. A `<corrected_code>`
lint/PHPCS/security gate therefore has an empty parseable set → vacuous; running it would
either wave the merge through silently or HARD-fail on a vacuous set — both meaningless.

**Why not the alternatives:**
- Option A (lint teacher-authored `<corrected_code>`) tests the training-data pipeline, not
  the model — a data-QA concern (Phase 4.1/4.2), not a model-quality eval gate.
- Option B (re-prompt 35 ctf snippets with fix framing) tests UNTRAINED behavior; high
  unparseable rate is the expected outcome, not a quality signal — and costs a 2nd vLLM boot.

**Coverage:** fix/execution correctness is already gated by **REVL-04** (wp-bench execution
tests, PASS 0.4616 ≥ 0.4286). The gate slot is RETIRED, not vacuously passed.

**Downstream implication:** W5-01 (REVL-05 human review) becomes **CoT-only** (10 samples,
not 20 mixed). Drop the CtF stream from the review artifact rather than leave a zero-row join.
CtF capability deferred to a future milestone if CtF training data is added.

`scripts/revl06_fix_correctness.py` is NOT built (N/A — nothing meaningful to check).

## REVL-03 — MARGINAL (defer to REVL-05), recorded 2026-06-02

**Verdict: MARGINAL, not a clean PASS.** Point estimate 0.814 (run 3) clears the 0.80
floor, but the gate sits inside its own noise band: two same-taxonomy runs gave 0.852
(@1024) and 0.814 (@2048) — spread 0.038, ~2.7× the 0.014 margin over 0.80. A
within-run bootstrap (2000×, run-3 per-sample coverage) gives 95% CI **[0.751, 0.871]**
— lower bound BELOW 0.80, and 33% of bootstrap draws fall under 0.80. Between-run
variance is additional on top. So REVL-03 cannot be asserted as a clean automated PASS.

**Disposition:** record as marginal (0.81–0.85 across runs, CI straddles 0.80) and hand
the uncertainty to **REVL-05** — the human reviewer is judging reasoning quality anyway
and is the right place to absorb a borderline automated number. This mirrors the REVL-01
approach (document the messy gate honestly rather than assert a clean number).

**Root cause of the noise (characterized, not a bug):** the model is bimodal — 88/120
samples score full coverage (1.0), but 32 are partial, including 14 terse JSON-mode rows
at 0.125 (1/8). The rate rides on how many prompts land in terse-JSON vs full-prose mode,
which flips per-prompt across regenerations even at temp=0 (close-tag count drifted 73→76
between runs). Judge variance (general-purpose agents re-judging borderline rows) adds to
it. Both are real sources, neither fixable by re-running — re-running just redraws.

**Point estimate supporting metrics (run 3):** score_reasoning_consistency_rate 0.968,
mean_coherence 3.97.
Artifacts: `output/eval_reasoning/revl03_aggregate.json`, `revl03_claude_eval.jsonl`
(120 rows), `revl03_agent_plan.jsonl`, `reasoning_merged/captured_responses.jsonl`
(120, parseable_rate 0.992), `revl03_chunks/` (10 chunk + 10 result files).
Earlier runs superseded: 0.810 (wrong D1..D9 taxonomy, VOID); 0.852 (8-dim but @1024
capture). This 0.814 is the authoritative number (2048 capture, parseable 0.992).

**Execution method (reproducibility):**
1. `capture_reasoning_responses.py --max-tokens 2048` — one vLLM boot (488s) on
   reasoning-merged, 120 cot+ctf prompts, temp=0. parseable_rate 0.992, 0 `<corrected_code>`.
2. `revl03_evaluator_agent.py` — emitted 120 opaque agent prompts (8-dim taxonomy).
3. Orchestrating session dispatched **10 chunked Claude Code agents** (12 samples
   each), agentType `general-purpose` (NOT `Agent(model=sonnet, run_in_background)`
   as the docstring's ideal). Chunked-not-1-per-sample by explicit user choice. No
   Anthropic API used — Claude Code agents only, per ROADMAP §4.4.
4. `aggregate_revl03.py` — merged results, computed the gate.

**CRITICAL methodology correction (advisor-caught, 2026-06-02):** the FIRST run
injected a naive D1..D9 set including `error_handling` (D8) + `code_structure` (D9).
The model's ACTUAL prose rubric is 8 dims {wpcs, security, sql_safety, performance,
wp_api_usage, accessibility, code_quality, dependency_integrity} — verified 85/85 CoT
rows; `error_handling` + `i18n` are STRUCTURALLY ABSENT from prose, `dependency_integrity`
(emitted) was missing from the naive set. First 0.810 measured taxonomy mismatch — VOID,
inconsistent with REVL-01 `dim_map.json` (council Option 3). Dimension set now pinned in
`REVL03_DIMENSIONS` (revl03_evaluator_agent.py); denominator = per-sample key count (8).

**Capture-cap investigation (resolved):** a 1024-token first capture was suspected of
truncating reasoning (47/120 lacked `[/REASONING]`). Re-captured at 2048 (GT max=1105).
Finding: the no-close-tag rows are NOT truncated — they are a genuine **bimodal output
mode**. ~44/120 responses are terse direct-JSON judge output (median ~264 tok, max 805,
all <1024) with no reasoning prose; the other ~76 are full prose + `[/REASONING]`. This
is the same prose/JSON bimodality documented in the W0-03 smoke gate. The 2048 recapture
only nudged close-tag 73→76 and lifted parseable 0.967→0.992, confirming the cap was not
the cause.

**Residual coverage gap (genuine model behavior, not artifact):** the ~44 terse JSON-mode
samples skip several dimensions in their (absent) explanation text, pulling coverage from
~1.0 (prose format) down. This is why 0.814 sits near the 0.80 floor — it reflects a real
mix of thorough-prose and terse-JSON judging modes, honestly measured.
