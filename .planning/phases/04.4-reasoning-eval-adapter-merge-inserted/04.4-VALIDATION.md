> Per-phase validation contract for feedback sampling during execution. See `04.4-RESEARCH.md` §"Validation Architecture" for the full architecture derivation.

## Test Infrastructure

| Property | Value |
|----------|-------|
| **Framework** | pytest 8.x (Python) + bash assertions (PHP CLI / PHPCS / security scanner via subprocess) |
| **Config file** | `pytest.ini` (existing) + `scripts/_check_wave0_smoke.py` (NEW Wave 0) |
| **Quick run command** | `pytest -x -q tests/phase4_4/` |
| **Full suite command** | `pytest tests/phase4_4/ -v && bash scripts/verify_04.4_artifacts.sh` |
| **Estimated runtime** | Quick: ~30 s · Full: ~2 min (excludes the full 6–8 h eval cycle itself) |

The Phase 4.4 plan is execution-heavy (long-running eval + merge) — Wave 0 covers the smoke gate and per-step artifact existence assertions. The hard gates (Spearman, PHPCS, wp-bench) are themselves the "tests" — their output JSONs are the source of truth.

---

## Sampling Rate

- **After every task commit:** Run `pytest -x -q tests/phase4_4/`
- **After every plan wave:** Run full suite + verify the wave's expected output artifact exists
- **Before `/gsd:verify-work`:** Full suite green AND all hard-gate JSONs present with `status: pass`
- **Max feedback latency:** 30 s for unit, 2 min for full
