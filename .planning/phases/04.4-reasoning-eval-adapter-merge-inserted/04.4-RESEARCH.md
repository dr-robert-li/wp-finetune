## Summary

The central finding of this research is that **wp-reasoning-v3 uses a third, distinct LoRA tensor convention** — neither the ckpt-72 Unsloth fused-shared-rank convention nor standard HF/PEFT. Tinker exports per-weight-matrix LoRA tensors using a **shared-A / per-expert-B (or per-expert-A / shared-B) 3D layout** indexed as `mlp.experts.w1/w2/w3` (not HF's `gate_up_proj/down_proj`). This means `merge_adapter.py` and `_p0_unsloth_merge_v3.py` (both use `merge_and_unload()`) will NOT correctly merge the MoE expert deltas — the key names don't match HF's fused parameter tree. The correct template is `_p0_merge_unsloth_static_moe.py` with its per-expert delta math **rewritten for Tinker's 3D layout**.

A second decisive finding: **v3 was trained on stock Qwen3-30B-A3B vocab (151,936 tokens)**. The adapter's `unembed_tokens.lora_B` shape is `[151936, 32]`, not 151,938. The v1 task tokens `<wp_gen>` and `<wp_judge>` are NOT in the v3 adapter vocabulary. This means the merge base must be `models/Qwen3-30B-A3B` (stock, on disk), and the extended tokenizer must be re-applied post-merge. `merge_adapter.py`'s single-token verification roundtrip does not apply as written.

On the gate-evidence-transfer problem: Tinker-cloud-sampled results cannot be byte-identical to merged-vLLM-served outputs (different inference stacks, different tokenizers at inference time, `renderer: qwen3_disable_thinking` vs vLLM chat template). The fidelity check must be on **decisions/distributions**, not tokens. The decisive probe is the 24-prompt invalid-PHP sentinel (discrete verdicts on the exact axis REVL-05 rejected on), plus Spearman agreement on 121 val rows. If fidelity passes, REVL-01/02/07/08 may reference Tinker-sampled evidence. REVL-04 wp-bench **must run fresh on merged-served** regardless — it never ran on v3 at all.

**Primary recommendation:** New script `scripts/merge_tinker_v3.py` using Tinker's 3D per-expert/shared convention + `_p0_anchor_forward_rankpath.py` fidelity anchors. CPU-only merge on stock base. Then vLLM bf16 serve at `GPU_MEM_UTIL=0.55`, `max_model_len=8192`, and run fidelity gate before eval cascade.

## The One New Unknown: v3 Tinker Adapter Merge Convention

### Finding 1 — Tensor Layout (VERIFIED: tensor inspection of `models/tinker_export/wp-reasoning-v3/checkpoint.tar`)

```
Total tensor keys: 674
adapter_config.json:
  peft_type: LORA
  target_modules: all-linear
  modules_to_save: null
  r: 32, lora_alpha: 32  → scale = 1.0
  use_rslora: false

MoE expert keys per layer (48 layers × 128 experts):
  mlp.experts.w1  (gate_proj):
    lora_A [1, 32, 2048]    ← SHARED across all 128 experts
    lora_B [128, 768, 32]   ← per-expert
  mlp.experts.w2  (down_proj):
    lora_A [128, 32, 768]   ← per-expert
    lora_B [1, 2048, 32]    ← SHARED across all 128 experts
  mlp.experts.w3  (up_proj):
    lora_A [1, 32, 2048]    ← SHARED across all 128 experts
    lora_B [128, 768, 32]   ← per-expert

Attention keys (standard, per layer):
  self_attn.{q,k,v,o}_proj.lora_A/B  (2D standard)

Vocab/special-token key:
  unembed_tokens.lora_A [32, 2048]
  unembed_tokens.lora_B [151936, 32]   ← STOCK vocab, NOT extended
```

**Key derivation for per-expert delta:**
- w1/w3 (shared-A, per-expert-B): `delta_e = B_e @ A_shared * scale` → shape `[768, 2048]`
- w2 (per-expert-A, shared-B): `delta_e = B_shared @ A_e * scale` → shape `[2048, 768]`
- HF 5.x stores: `gate_up_proj [128, 1536, 2048]` (w1+w3 concat on dim1), `down_proj [128, 2048, 768]`
- gate_up delta: concat `delta_e_w1` and `delta_e_w3` along dim0 → `[1536, 2048]` → add to `gate_up_param.data[e]`
- down delta: add `delta_e_w2` `[2048, 768]` to `down_param.data[e]`

### Finding 2 — Vocabulary (VERIFIED: tensor inspection)

`unembed_tokens.lora_B.shape[0] = 151936` = stock Qwen3-30B-A3B vocab. The v1 extended tokenizer (151,938 = +`<wp_gen>`/`<wp_judge>`) is NOT in the adapter. Tinker trained on base Qwen3 vocab.

**Consequences (CRITICAL — two separate issues):**

**1. Merge base and tokenizer:** Merge base must be `models/Qwen3-30B-A3B` (stock, 151,936 vocab, present on disk), NOT `models/qwen3-30b-wp-30_70-merged-v2` (151,938 vocab — shape mismatch on lm_head). The merged model must be served with the **stock Qwen3-30B-A3B tokenizer** — do NOT inject the extended vocab. v3 was trained on stock vocab; Tinker used `tokenizer_utils.get_tokenizer(Qwen/Qwen3-30B-A3B)` (verified in `scripts/tinker_reasoning_data.py`). The `<wp_gen>`/`<wp_judge>` tags are tokenized as subword sequences in v3, not as single tokens. Injecting the extended tokenizer would produce out-of-range token IDs (151,936/151,937) that index outside the model's embedding matrix — crash or garbage output. `merge_adapter.py`'s single-token assertion does NOT apply to v3.

**2. unembed_tokens LoRA must be applied manually:** Tinker's key `base_model.model.model.unembed_tokens.lora_*` does NOT match HF Qwen3's module tree (which has `lm_head`, not `unembed_tokens`). PEFT's `merge_and_unload()` on an attention-only temp adapter will silently skip the unembed LoRA. Apply it manually: `lm_head.weight.data += (B_unembed @ A_unembed).to(torch.bfloat16) * scale` where B [151936,32] @ A [32,2048] -> [151936,2048]. Confirmed safe: `tie_word_embeddings: False` in `models/Qwen3-30B-A3B/config.json` — writing lm_head does not mutate embed_tokens.

### Recommended Merge Approach — New Script `scripts/merge_tinker_v3.py`

Template from `_p0_merge_unsloth_static_moe.py` but with rewritten MoE delta extraction:

```python
# Tinker convention — shared-A/per-expert-B for w1,w3; per-expert-A/shared-B for w2
# scale = lora_alpha/r = 32/32 = 1.0

for L in range(48):
    gate_up_param = model.model.layers[L].mlp.experts.gate_up_proj  # [128,1536,2048]
    down_param    = model.model.layers[L].mlp.experts.down_proj      # [128,2048,768]

    # w1 (gate_proj): shared A [1,32,2048] → squeeze → [32,2048]; per-expert B [128,768,32]
    A_w1 = tensors[f"...layers.{L}.mlp.experts.w1.lora_A.weight"].squeeze(0)  # [32, 2048]
    B_w1 = tensors[f"...layers.{L}.mlp.experts.w1.lora_B.weight"]             # [128, 768, 32]

    # w3 (up_proj):  same layout as w1
    A_w3 = tensors[f"...layers.{L}.mlp.experts.w3.lora_A.weight"].squeeze(0)  # [32, 2048]
    B_w3 = tensors[f"...layers.{L}.mlp.experts.w3.lora_B.weight"]             # [128, 768, 32]

    # w2 (down_proj): per-expert A [128,32,768]; shared B [1,2048,32] → squeeze → [2048,32]
    A_w2 = tensors[f"...layers.{L}.mlp.experts.w2.lora_A.weight"]             # [128, 32, 768]
    B_w2 = tensors[f"...layers.{L}.mlp.experts.w2.lora_B.weight"].squeeze(0)  # [2048, 32]

    for e in range(128):
        # gate delta [768,2048], up delta [768,2048] → concat → [1536,2048]
        delta_gate = (B_w1[e] @ A_w1) * scale   # [768,32] @ [32,2048] = [768,2048]
        delta_up   = (B_w3[e] @ A_w3) * scale   # [768,32] @ [32,2048] = [768,2048]
        gate_up_param.data[e] += torch.cat([delta_gate, delta_up], dim=0).to(torch.bfloat16)

        # down delta [2048,768]
        delta_down = (B_w2 @ A_w2[e]) * scale   # [2048,32] @ [32,768] = [2048,768]
        down_param.data[e] += delta_down.to(torch.bfloat16)
```

**Attention (PEFT path):** standard `PeftModel.from_pretrained` + `merge_and_unload()` on attention keys only (`self_attn.{q,k,v,o}_proj`) is correct. Extract only these keys into a temp attention-only adapter and use PEFT to merge them.

**unembed_tokens (manual path):** Tinker's key `base_model.model.model.unembed_tokens.lora_*` does NOT match HF's `lm_head` in PEFT's module tree. Do NOT include it in the PEFT temp adapter — it will be silently skipped. Apply manually AFTER the PEFT attention merge:
```python
A_un = tensors["base_model.model.model.unembed_tokens.lora_A.weight"]  # [32, 2048]
B_un = tensors["base_model.model.model.unembed_tokens.lora_B.weight"]  # [151936, 32]
delta_lm = (B_un.float() @ A_un.float()) * scale   # [151936, 2048]
model.lm_head.weight.data += delta_lm.to(torch.bfloat16)
# tie_word_embeddings=False confirmed — lm_head and embed_tokens are independent
```

**Post-merge tokenizer:** save the **stock `Qwen3-30B-A3B` tokenizer** into the merged dir (`AutoTokenizer.from_pretrained('models/Qwen3-30B-A3B')`). Do NOT use the extended 151,938-token tokenizer — v3 was trained on stock vocab; injecting extended vocab would produce out-of-range IDs.

**Per-expert-differ guard** (reuse from template): after merge, verify L0 w2 delta for e0 ≠ e1 (max_diff > 1e-5). If equal, merge silently broadcast — regression to wrong convention.

### Fidelity Gate Design

**Purpose:** prove merged-vLLM-served reproduces Tinker-sampled within decision-level tolerance. If fidelity passes, Tinker-sampled REVL-01/02 results carry; if it fails, re-run REVL-01/02 on merged-served.

**Recommended fidelity protocol (three layers, cascading):**

**Layer 1 — Forward-pass tensor anchor** (reuse `_p0_anchor_forward_rankpath.py`):
- Load merged model + load Tinker-exported adapter on stock Qwen3 (if possible via PEFT)
- Run same 3 prompts through both at layers 0, 23, 47
- Check: cos_sim ≥ 0.99990, rel_l2 ≤ 1e-2 (bf16-calibrated, already proven)
- This verifies merge correctness at the weight level before serving

**Layer 2 — Invalid-PHP sentinel verdict agreement** (most decisive, ~5 min):
- Re-run all 24 held-out invalid-PHP snippets through merged-vLLM-served (same prompts as Tinker-sampled Phase 4.3)
- Compare verdicts (under VERDICT-POLICY.md: PASS iff overall_score ≥ 70 AND no auto-FAIL class)
- Pass threshold: all 24 agree with Tinker-sampled policy verdicts (0/24 policy false-pass in both)
- This is the axis REVL-05 originally rejected on; agreement here proves the quality signal transferred

**Layer 3 — Spearman agreement on 121 val rows**:
- Generate merged-served outputs for the 121 REVL-01 val prompts
- Compare `overall_score` between Tinker-sampled (from `output/eval_reasoning/reasoning_v3_tinker/eval_judge_results.json`) and merged-served
- Pass threshold: Spearman(tinker_scores, merged_scores) ≥ 0.95 AND policy-verdict agreement ≥ 22/24 on the sentinel subset
- If Layer 3 passes: carry REVL-01 Spearman 0.263 as the evidence; no need to re-compute full Spearman from scratch

**If all three layers pass:** Tinker-sampled REVL-01 (0.263) and REVL-02 (1.0) carry. Proceed to REVL-04 wp-bench.
**If Layer 2 or 3 fails:** fidelity not proven; re-run REVL-01/02 on merged-served before REVL-04.

**Critical preconditions (both required for fidelity):**
1. **Tokenizer parity:** vLLM must be served with the stock Qwen3-30B-A3B tokenizer (same as Tinker training). If a different tokenizer is used, prompt tokenization diverges and fidelity comparisons are invalid regardless of merge correctness.
2. **Thinking disabled:** vLLM must be configured to disable thinking (Qwen3 emits `<think></think>` scaffold otherwise). Use `--chat-template` or generation config equivalent to Tinker's `renderer: qwen3_disable_thinking`. The existing REVL-04 fix chain already handles empty `<think>` stripping via `scripts/_wpbench_pth/usercustomize.py` and `eval_gen.py:60` — verify these also apply for REVL-01/02 response collection.

## Common Pitfalls

### Pitfall 1: Using `merge_and_unload()` for MoE expert keys
**What goes wrong:** PEFT walks the model tree looking for modules named `gate_up_proj` / `down_proj`. Tinker's expert keys are `mlp.experts.w1/w2/w3`. PEFT will either error on key mismatch or silently no-op the expert deltas. The resulting merged model would have correct attention + lm_head merge but BASE (unmodified) MoE expert weights — same bug class that affected the v1 baseline (where gate_up_proj LoRA was never trained, but for a different reason).
**How to avoid:** Custom per-expert delta loop (see merge code above). Verify with per-expert-differ guard.
**Warning signs:** `per_expert_delta_differ_check` in `merge_report.json` is 0.0 or < 1e-5.

### Pitfall 2: Merging onto wrong base model or using extended tokenizer
**What goes wrong (A):** Merging onto `models/qwen3-30b-wp-30_70-merged-v2` (151,938 vocab) with a v3 adapter trained on stock vocab (151,936) causes shape mismatch on lm_head (`[151938,2048]` base vs `[151936,2048]` adapter delta).
**What goes wrong (B):** Serving the merged v3 model with the extended 151,938-token tokenizer causes `<wp_gen>`/`<wp_judge>` tags to be emitted as token IDs 151,936/151,937 — out of range for a 151,936-row embedding matrix. Crash or silent garbage output.
**How to avoid:** Merge base = `models/Qwen3-30B-A3B` (stock). Serve with stock tokenizer (`AutoTokenizer.from_pretrained('models/Qwen3-30B-A3B')`). v3 encodes task tags as subword sequences via stock vocab — this is how Tinker-sampled gates passed.
**Warning signs:** lm_head shape assertion failure during merge; vLLM CUDA index-out-of-range on inference; terse or degenerate output for `<wp_judge>` prompts.

### Pitfall 3: vLLM thinking-mode mismatch
**What goes wrong:** Qwen3-30B-A3B emits `<think>\n\n</think>` prefix by default. Tinker used `renderer: qwen3_disable_thinking`. If vLLM serves without equivalent disable, fidelity comparison is contaminated AND wp-bench knowledge scoring breaks (already fixed with `_wpbench_pth/usercustomize.py` strip, but REVL-01/02 response collection also needs think-stripped outputs).
**How to avoid:** Use vLLM flag to disable thinking, OR ensure `strip_think_blocks()` from `eval/output_parsers.py` is applied before any scoring. The `eval_gen.py:60` strip is already present; verify it's active.
**Warning signs:** raw responses start with `<think>` text; parseable_rate drops; Spearman degrades unexpectedly.

### Pitfall 4: Treating Tinker-sampled Spearman as gate evidence without fidelity check
**What goes wrong:** v3's REVL-01A 0.263 was computed offline from Tinker-sampled outputs. Merged-served outputs may differ enough to shift Spearman below 0.171 baseline. Carrying this number without fidelity proof is an untested assumption.
**How to avoid:** Run fidelity gate (invalid-PHP sentinel + Spearman agreement) BEFORE treating Tinker evidence as carrying to merged-served. If fidelity fails, re-run REVL-01/02 on merged-served.

### Pitfall 5: Staging path not isolated from canonical
**What goes wrong:** merge writes directly to `models/qwen3-30b-wp-30_70-reasoning-merged/` (canonical), which was previously occupied by the ckpt-72 merge. If the v3 merge fails mid-run or anchors fail, canonical is corrupted.
**How to avoid:** Write to `models/_staging/qwen3-30b-wp-30_70-reasoning-merged-v3/` (per locked staging-then-promote decision). Promote to canonical only after all gates pass + human spot-check.

### Pitfall 6: wp-bench prereqs not initialized
**What goes wrong:** REVL-04 requires `wp-env start` (Docker + WordPress containers), npm global `@wordpress/env`, IPv4 NODE_OPTIONS. These are NOT captured in the repo and must be set up fresh each session.
**How to avoid:** Run the prereq checklist from `REVL-04-COMPLETION.md` before launching: `npm install -g @wordpress/env`, `cd wp-bench/runtime && wp-env start`. If site is broken: `wp-env destroy && wp-env start`.
