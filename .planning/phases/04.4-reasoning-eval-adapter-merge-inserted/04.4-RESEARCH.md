# Phase 4.4: Reasoning Eval & Adapter Merge - Research

**Researched:** 2026-05-28
**Domain:** MoE-expert LoRA merge architecture, vLLM bf16 serving on GB10, Qwen3-30B-A3B reasoning evaluation harness
**Confidence:** HIGH (gating decision triangulated across 4 primary sources)

## Summary

Phase 4.4 must run the v1.2 eval-and-merge sequence against reasoning-adapter `adapters/qwen3-30b-wp-30_70-reasoning/checkpoint-72` and ship `models/qwen3-30b-wp-30_70-reasoning-merged/` on full-pass + human approval. The central blocker carried from Phase 4.3 was: how to serve and evaluate a Qwen3-MoE adapter with `target_parameters: [mlp.experts.gate_up_proj, mlp.experts.down_proj, down_proj]` on GB10 (121 GiB unified memory) when (a) vLLM cannot serve parameter-level LoRA as runtime adapters, (b) HF Transformers + bnb 4-bit collapses MoE routing into degenerate output, and (c) HF bf16 OOMs.

**Primary recommendation: Path 1 — Unsloth (or raw-PEFT) merge → vLLM bf16 serve.** Direct source reads on the production PEFT (0.18.1, locally verified) confirm `ParamWrapper.merge()` exists at `peft/tuners/lora/layer.py:2405` and correctly fuses `target_parameters` LoRA deltas into wrapped `nn.Parameter`s in place. PEFT's standard `merge_and_unload()` walks every `ParamWrapper` and invokes this method, so the resulting bf16 checkpoint has the MoE-expert LoRA baked into expert weights. Both Unsloth's `FastLanguageModel.save_pretrained_merged(save_method='merged_16bit')` and the existing `scripts/merge_adapter.py` (raw HF + PEFT, line 108) delegate to `merge_and_unload()` and therefore both should produce a correct merged checkpoint. The pre-existing `scripts/serve_30_70_vllm.sh` GPU_MEM_UTIL=0.55 bf16 pattern then serves the result over OpenAI-compatible HTTP — satisfying D-02's HARD constraints (HTTP for wp-bench AND `target_parameters` MoE-LoRA semantics preserved).

**Critical caveat:** The merge is path-verified at the code level (`[VERIFIED]`) but end-to-end correctness on Qwen3-30B-A3B's packed-expert layout is `[ASSUMED]` until a 3-prompt smoke test (D-02 acceptance gate) confirms coherent generation that differs measurably from the partial-30/70 baseline. Operationalize as a Wave 0 gate — fail-fast before booting the full vLLM/eval cycle.

### Pattern 1: Merge target_parameters MoE-LoRA via Unsloth's save_pretrained_merged

**What:** Use Unsloth's `FastLanguageModel.save_pretrained_merged(save_method='merged_16bit')` which delegates to PEFT's `merge_and_unload()`. `merge_and_unload()` walks every PEFT layer including `ParamWrapper` instances; for each `ParamWrapper`, calls its `merge()` method which fuses the LoRA delta into the wrapped `nn.Parameter` in place (e.g., `mlp.experts.gate_up_proj`).

**When to use:** Merge step of Phase 4.4, after Wave 0 smoke gate passes.

**Example (Unsloth path — recommended):**
```python
# Source: github.com/unslothai/unsloth/blob/main/unsloth/save.py:584-585 (delegation)
#         github.com/huggingface/peft/blob/main/src/peft/tuners/lora/layer.py:2405 (ParamWrapper.merge)
from unsloth import FastLanguageModel

model, tokenizer = FastLanguageModel.from_pretrained(
    model_name="adapters/qwen3-30b-wp-30_70-reasoning/checkpoint-72",
    dtype=None,             # auto bf16
    load_in_4bit=False,     # CRITICAL: 4-bit collapses MoE router (4.3-SUMMARY)
    device_map={"": "cpu"}, # merge on CPU; ~120 GiB unified RAM peak
)

model.save_pretrained_merged(
    save_directory="models/qwen3-30b-wp-30_70-reasoning-merged",
    tokenizer=tokenizer,
    save_method="merged_16bit",
)
```

**Alternative (raw-PEFT path — existing `scripts/merge_adapter.py`):**
```python
# Source: scripts/merge_adapter.py:86-108 (already in repo)
model = AutoModelForCausalLM.from_pretrained(local_dir, dtype=torch.bfloat16, device_map="cpu")
model = PeftModel.from_pretrained(model, adapter_dir, config=peft_config)
merged_model = model.merge_and_unload()
merged_model.save_pretrained(merged_path)
```

Both paths invoke `merge_and_unload()` → `ParamWrapper.merge()` → fuses target_parameters. Recommendation: keep the existing `merge_adapter.py` scaffolding (idempotency check + special-token roundtrip verify) and swap the load+save to Unsloth's wrapper, since the adapter was trained with Unsloth's `FastLanguageModel.get_peft_model` and the Unsloth loader does adapter-specific corrections (`unsloth_fixed: true` in adapter_config).

### Pattern 3: Sequential eval-gate loop with vLLM kept alive (run_eval_triage.py pattern)

**What:** Single vLLM boot → eval_gen → eval_judge → eval_gate → wp-bench → stop vLLM. Avoid restart overhead (~10 min weight load + ~2 min CUDA graphs per restart).

**When to use:** Step 4 (automated gates).

**Example structure (adapted from `run_eval_triage.py:_start_vllm_with_lora` + `run_eval_and_wpbench_for_experiment`):**
```python
endpoint = "http://localhost:8020/v1"
vllm_proc = None
try:
    vllm_proc = _start_vllm_merged(merged_path)         # boot vLLM via serve script
    if not _wait_for_vllm(endpoint, timeout_s=900):     # 30B MoE: ~10 min load
        raise RuntimeError("vLLM failed to boot")

    # Halt-on-regression discipline: each gate checks before proceeding
    gen_result = eval_gen.run_eval(...)                 # REVL-02
    if gen_result.phpcs_pass_rate < baseline - 0.02:    # 2pp regression → halt before judge
        write_failure_summary("PHPCS regression beyond 2pp"); return False

    judge_result = eval_judge.run_eval(...)             # REVL-01
    if judge_result.spearman < baseline_spearman:       # drop → halt before wp-bench
        write_failure_summary("Spearman regression"); return False

    eval_gate.run_gate(...)                             # informational aggregation

    wpbench_result = run_wpbench(...)                   # REVL-04 HARD GATE
    if wpbench_result.score < baseline_wpbench:
        write_failure_summary("wp-bench regression"); return False
finally:
    _stop_vllm(vllm_proc)
```

Halt conditions per CONTEXT D-05: any HARD-gate failure aborts before wasting subsequent eval time.

### Anti-Patterns to Avoid

- **Do NOT load ckpt-72 in 4-bit for any purpose.** Phase 4.3 SUMMARY proved int4 quantization of Qwen3-MoE router weights collapses top-8 expert routing → degenerate output (`## Admin × 80`, `<think>\n`, `@param × 80`). This is independent of the trained adapter and persists at base model level too. 4-bit gate is architecturally invalid for this model class.
- **Do NOT attempt vLLM `--enable-lora` for ckpt-72.** vLLM rejects `target_parameters` LoRA at runtime (per `serve_30_70_vllm.sh` comment block + `run_eval_triage.py:_is_vllm_crash_looping` fatal-pattern detection). Merge-then-serve is the only viable HTTP path.
- **Do NOT regenerate `output/triage_decision.md` from `run_eval_triage.py` without the HUMAN_OVERRIDE sentinel.** CONTEXT D-03b: the previous auto-regen clobbered the 30_70 human-override row into NO_SURVIVORS. The committed `aef1b6a` version is the authoritative pre-clobber state; check there first if the working-tree copy needs recovery (`git show HEAD:output/triage_decision.md`).
- **Do NOT pass `device_map="auto"` for the merge load on GB10.** MoE models cannot auto-offload to disk; `auto` device-maps GPU+CPU which exhausts the unified pool before merge math finishes. Use `device_map="cpu"` explicitly (per `merge_adapter.py:91`).
- **Do NOT skip the smoke gate.** D-02 makes it the path-acceptance criterion. Without it the only way to discover a silent-drop merge bug would be after a 6-8h eval cycle finishes producing baseline-equal-but-no-reasoning numbers.

## Common Pitfalls

### Pitfall 1: System RAM exhaustion during merge

**What goes wrong:** `device_map="cpu"` puts base (~60 GiB bf16) + adapter (~5 GiB) + output (~60 GiB) all in CPU RAM simultaneously. GB10 has 121 GiB unified shared between system and GPU. Peak working set during merge approaches the full pool — heavy desktop apps (Chromium, Slack) or a parallel GPU process can OOM the merge.

**Why it happens:** MoE models cannot auto-offload to disk (Hugging Face MoE bug — `auto` device_map allocates all experts in memory). The merge math materializes the LoRA delta tensor before the in-place `param.data += delta` (see PEFT `ParamWrapper.merge()` at `layer.py:2425-2435`), so there's a transient peak above 120 GiB even though the final merged model is the same size as the base.

**How to avoid:** Add a pre-flight gate distinct from the GPU-floor gate carried from Phase 4.3:

```python
def _check_system_ram_gib(min_gib: float = 80.0) -> float:
    """Check available system RAM. Abort if below floor."""
    import psutil
    avail = psutil.virtual_memory().available / (1024**3)
    if avail < min_gib:
        raise RuntimeError(
            f"Pre-merge floor failed: {avail:.1f} GiB available, need ≥{min_gib} GiB. "
            f"Close Chromium/heavy apps and retry."
        )
    return avail
```

Plus: close Chromium/Slack/Code-Insiders before merge launches (same pattern as 4.3 streaming-log Popen / pre-flight floor; carry-over from `scripts/_run_parse_check_ckpt72.py`).

**Warning signs:** `pmap` shows merge process exceeding 100 GiB resident; system swap activity spikes; merge process hangs at "Merging adapter into base model …".

### Pitfall 2: Tokenizer drift between merge and serve

**What goes wrong:** Unsloth's `save_pretrained_merged` saves the tokenizer alongside weights. If the partial-30/70 base tokenizer is loaded by `FastLanguageModel.from_pretrained(checkpoint-72)` AND the adapter directory has a stale tokenizer copy, the saved merged checkpoint can ship with the wrong tokenizer — `<wp_gen>`/`<wp_judge>` either re-tokenize to multiple tokens or fail to encode at all.

**Why it happens:** The base tokenizer at `models/qwen3-30b-wp-30_70-merged/` has the special tokens baked in (per Phase 3 MODL-02). ckpt-72's adapter dir might or might not have a tokenizer copy. PEFT's standard merge flow doesn't deduplicate.

**How to avoid:** Reuse `scripts/merge_adapter.py:_verify_merged_model` after `save_pretrained_merged` completes. The verify step roundtrips `<wp_gen>` and `<wp_judge>` through the saved tokenizer and asserts they encode as single tokens at their expected IDs. Fail-fast and re-merge with explicit tokenizer source if drift detected.

**Warning signs:** `tokenizer.encode("<wp_gen>", add_special_tokens=False)` returns more than 1 token ID after merge.

### Pitfall 3: vLLM weight-load timeout on 30B MoE

**What goes wrong:** Default `--health-timeout` in run_eval_triage.py is 600s (10 min). Qwen3-30B-A3B bf16 takes 8-10 min to load weights + ~2 min for CUDA graphs. Cold boots can exceed 600s and trigger a false-negative health-fail → merge-and-serve fallback retry.

**Why it happens:** GB10 has slower NVMe bandwidth than DGX H100s; bf16 60 GiB weight transfer over unified memory + safetensors deserialize adds non-trivial latency.

**How to avoid:** Bump `--health-timeout` to 900s for 4.4 vLLM boots (per skill docs Step 0c recommendation). Monitor via `docker logs -f wp-30_70-vllm | grep -E "loading|prefill|Application startup"`.

**Warning signs:** `vLLM did not become healthy within 600s` in `run_eval_triage.py` log even though `docker logs` shows weight loading is still progressing.

### Pitfall 4: triage_decision.md regeneration clobbers human override

**What goes wrong:** Any path that re-runs the Phase 4 triage workflow (e.g., re-running `run_eval_triage.py` for a fresh baseline) regenerates `output/triage_decision.md`. The auto-regen overwrites the HUMAN_OVERRIDE row (30_70 accepted despite Spearman 0.5698) with the gate-fail NO_SURVIVORS result. Phase 4.3 hit this once already (see STATE.md note + CONTEXT D-03b).

**Why it happens:** `triage_ratios.py` is purely gate-driven and has no sentinel-recognition logic. The human override is layered on top in markdown but not preserved through regeneration.

**How to avoid:** Two options for the planner to pick from:
1. **Sentinel guard:** Add a `HUMAN_OVERRIDE: <ratio> <ISO-timestamp>` sentinel line at the top of `output/triage_decision.md` and patch `scripts/triage_ratios.py:write_triage_decision` to refuse to overwrite if sentinel is present (require `--force-override` flag).
2. **Move the override to a separate file:** Write `output/triage_human_override.md` that `run_eval_triage.py` reads and appends after regeneration; `triage_decision.md` becomes the pure automated output.

Either way: do not let the working-tree clobbered copy enter git. Use `git restore output/triage_decision.md` before the first 4.4 commit.

**Warning signs:** `output/triage_decision.md` working-tree diff shows STATUS line flipping from "HUMAN_OVERRIDE — 30_70 ACCEPTED" to "NO_SURVIVORS"; the committed `aef1b6a` version differs from working tree.

### Pitfall 5: Comparing reasoning-merged numbers to a partial baseline

**What goes wrong:** A naive reader of 4.4 results may interpret "reasoning-merged passes 30_70 baseline" as "v1.2 matches the full-quality 30_70 model." It does not. The 30_70 base used here (`models/qwen3-30b-wp-30_70-merged/`) was itself produced by a raw-PEFT merge before PEFT had working `ParamWrapper.merge()` — its `target_parameters` MoE LoRA was silently dropped (per `serve_30_70_vllm.sh` comment block). So Phase 4 Spearman 0.5698 + PHPCS 1.0000 reflects partial-30/70, not true-30/70.

**Why it happens:** Layered historical merges with different PEFT versions.

**How to avoid:** State the asymmetry explicitly in the Phase 4.4 report:
> "Phase 4.4 measures reasoning-LoRA-on-partial-30/70 merged with current PEFT (`ParamWrapper.merge()` fuses target_parameters) vs partial-30/70-base. The baseline is itself a partial merge (raw-PEFT, older). Both arms exclude part of the original 30_70 training delta. This is the right comparison FOR THIS PHASE — apples-to-apples within the current merge tooling — but is NOT a measurement of v1.2 vs true-30/70."

Do NOT propose re-merging the 30_70 base in this phase — that's scope creep. Flag it as a v2/v3 consideration if needed.

**Warning signs:** Any 4.4 narrative that says "reasoning passes the 30_70 baseline" without qualifying "partial".

    results = []

### Example 4: REVL-07 confusion matrix from eval_judge per-example data

```python
# Source: per-example JSONL schema from EVAL-06 + Phase 3 eval_judge pattern
# Standard sklearn pipeline — no hand-roll
import json
from sklearn.metrics import confusion_matrix, precision_recall_fscore_support

def revl07_confusion_matrix(
    eval_judge_pairs_jsonl: str,
    output_path: str,
    score_threshold: float = 7.0,
) -> dict:
    """Compute per-dimension TP/TN/FP/FN at given score threshold.

    Per-dimension: model classifies dimension as PASS (score ≥ threshold) vs FAIL.
    Ground truth: same on GT score.
    """
    per_dim = {f"d{i}": {"y_true": [], "y_pred": []} for i in range(1, 10)}
    with open(eval_judge_pairs_jsonl) as f:
        for line in f:
            ex = json.loads(line)
            gt = ex["gt_scores"]; pred = ex["model_scores"]
            for d in per_dim:
                per_dim[d]["y_true"].append(int(gt[d] >= score_threshold))
                per_dim[d]["y_pred"].append(int(pred[d] >= score_threshold))

    result = {}
    for d, arrs in per_dim.items():
        tn, fp, fn, tp = confusion_matrix(arrs["y_true"], arrs["y_pred"], labels=[0, 1]).ravel()
        p, r, f1, _ = precision_recall_fscore_support(
            arrs["y_true"], arrs["y_pred"], average="binary", zero_division=0,
        )
        result[d] = {"tp": int(tp), "tn": int(tn), "fp": int(fp), "fn": int(fn),
                     "precision": float(p), "recall": float(r), "f1": float(f1)}

    with open(output_path, "w") as f:
        json.dump({"score_threshold": score_threshold, "per_dimension": result}, f, indent=2)
    return result
```

### Example 5: REVL-08 reasoning length distribution

```python
# Source: AutoTokenizer-based length analysis; tokenizer must match training tokenizer
import json, re
from transformers import AutoTokenizer
import numpy as np

REASONING_RE = re.compile(r"\[REASONING\](.*?)\[/REASONING\]", re.DOTALL)

def revl08_length_distribution(
    eval_judge_pairs_jsonl: str,
    output_path: str,
    tokenizer_dir: str = "adapters/tokenizer",   # SAME tokenizer used in training
    flag_p95_max: int = 6000,
    flag_median_min: int = 500,
) -> dict:
    tok = AutoTokenizer.from_pretrained(tokenizer_dir)
    lengths = []
    with open(eval_judge_pairs_jsonl) as f:
        for line in f:
            ex = json.loads(line)
            m = REASONING_RE.search(ex.get("response", ""))
            if m:
                lengths.append(len(tok.encode(m.group(1), add_special_tokens=False)))
    arr = np.array(lengths) if lengths else np.array([0])
    summary = {
        "n_samples": int(arr.size),
        "median": int(np.median(arr)),
        "p95": int(np.percentile(arr, 95)),
        "max": int(np.max(arr)),
        "mean": float(np.mean(arr)),
        "flags": {
            "p95_too_high": bool(np.percentile(arr, 95) > flag_p95_max),
            "median_too_low": bool(np.median(arr) < flag_median_min),
        },
    }
    with open(output_path, "w") as f:
        json.dump(summary, f, indent=2)
    return summary
```

## Open Questions

1. **Should the planner use Unsloth's `save_pretrained_merged` wrapper or extend the existing raw-PEFT `merge_adapter.py`?**
   - What we know: Both paths invoke PEFT `merge_and_unload()`; both should produce identical merged tensors. Existing `merge_adapter.py` has idempotency check + special-token verify roundtrip already coded.
   - What's unclear: Whether `FastLanguageModel.from_pretrained` does adapter-specific corrections at load time that raw-`PeftModel.from_pretrained` skips (e.g., the `unsloth_fixed: true` flag in adapter_config suggests yes).
   - Recommendation: Use Unsloth wrapper at load time + keep merge_adapter.py's `_verify_merged_model` roundtrip after save. Best of both worlds. Planner can task-1 the extension cleanly.

2. **Threshold for REVL-03 "score-reasoning consistency rate" gate** (Claude's discretion per CONTEXT D-05)
   - What we know: REVL-03 is HARD-block per ROADMAP §4.4 SUCCESS criterion 3, but the numeric threshold isn't fixed.
   - What's unclear: What rate counts as "passing"? 80%? 90%? Spec needs to be planner-set.
   - Recommendation: Set conservative initial gate at ≥80% (rate of agent-judged consistent dimension claims across 100+ samples), allow human-override at review checkpoint if below. Document the threshold in the plan and STATE.md decisions log.

3. **Sample size for REVL-03 Claude-evaluator agent run**
   - What we know: 9 dimensions × per-example check; agents run in parallel; cost per agent is non-trivial.
   - What's unclear: How many eval examples to feed the agent — full eval set (~500-1000 examples), a stratified sample (~100), or the human-review subset (20)?
   - Recommendation: Run on the full eval_judge.pairs.jsonl set (matches eval_judge n, ~500). The metric is a rate, so it stabilizes with sample size. Run in batches of 50 agents in parallel per the `wp-finetune:run-data-pipeline` Agent pattern.

4. **Should `<corrected_code>` blocks in CtF samples be PHP-linted directly, or extracted-then-linted?**
   - What we know: REVL-06 is deterministic (PHPCS + security scanner on corrected code). CtF reasoning produces a `<corrected_code>...</corrected_code>` block.
   - What's unclear: Whether the corrected code is always syntactically self-contained (a function body or full file) or sometimes a snippet needing context.
   - Recommendation: Treat extracted code as a function body, wrap in a `<?php ` prolog if missing, lint via `php -l` + `phpcs --standard=WordPress`. If a CtF sample's `<corrected_code>` is too fragmentary to lint cleanly, flag it as "unparseable" rather than failing — the model behavior under measurement is whether the correction is syntactically defensible, not whether the harness handles edge cases.

## Path 2 + Path 3 short-form treatment (CONTEXT D-01 candidates 2 and 3)

**Path 2 (Unsloth FastLanguageModel in-process bf16) — ELIMINATED.** The Unsloth in-process path can correctly load ckpt-72 + the partial-30/70 base AND apply target_parameters LoRA at generate time (`ParamWrapper.forward()` adds delta at inference). This satisfies REVL-01/02/03/06/07/08 (the in-process gates that call `.generate()` directly). However, REVL-04 (wp-bench HARD GATE) requires an OpenAI-compatible HTTP server. Path 2 has no HTTP layer — eliminated by D-02's "MUST serve via OpenAI-compatible HTTP" constraint. Documenting for completeness only; do not pursue.

**Path 3 (GGUF Q8 via llama.cpp) — DEFERRED.** Verified at high level: Unsloth ships a public Q8_0 (32.5 GB) GGUF for the base Qwen3-30B-A3B (`huggingface.co/unsloth/Qwen3-30B-A3B-GGUF`), confirming the llama.cpp conversion path supports Qwen3MoE expert weights. Conversion of a target_parameters-merged checkpoint to GGUF would invoke `convert_hf_to_gguf.py` on the post-merge HF model — same input as Path 1's vLLM serve, just a different runtime. Routing fidelity at Q8 is empirically far better than int4 (8-bit per weight preserves enough router precision for top-8 selection to remain stable). Per CONTEXT Deferred Ideas, do NOT pursue Path 3 in 4.4 if Path 1 works — defer Q8 quantization-tier exploration to v3.0 Phase 15 (packaging). Path 3 remains the contingency if the smoke gate fails AND a deeper PEFT bug is identified that affects vLLM bf16 serving (low likelihood given verified merge path).

## Step 4 Sequential Gate Loop Design (resolves CONTEXT specifics + ROADMAP §4.4)

After the Wave 0 smoke gate passes and the merge completes, the eval loop runs sequentially with vLLM kept alive between gates (pattern from `run_eval_triage.py:run_eval_and_wpbench_for_experiment`):

1. **Boot vLLM** (`scripts/serve_30_70_vllm.sh` with `MODEL_DIR=models/qwen3-30b-wp-30_70-reasoning-merged`, GPU_MEM_UTIL=0.55, max-model-len 8192, port 8020).
2. **Wait for health** (poll `/health` for up to 900s; `_is_vllm_crash_looping()` early-aborts on fatal patterns).
3. **REVL-02: eval_gen** (full or `--limit 500` per planner discretion); writes `output/eval_reasoning/eval_gen_results.{json,jsonl}`. **Halt condition:** PHPCS pass rate regresses >2pp from baseline → halt before continuing.
4. **REVL-01: eval_judge** writes `eval_judge_results.{json,pairs.jsonl}`. **Halt condition:** Spearman drops below baseline (0.5698 stored) → halt before wp-bench.
5. **eval_gate aggregation** (informational, uses EVAL-07-fixed per-dimension reading); writes `eval_gate_results.json`.
6. **REVL-04: wp-bench HARD GATE** runs against same live vLLM endpoint via `wp-bench run --config <tmp_config_pointing_at_reasoning_merged>`; writes `wp_bench_results.{json,jsonl}`. **Halt condition:** score below stored baseline → halt before merge.
7. **Stop vLLM, reclaim memory.**
8. **REVL-06/07/08** post-process the per-example JSONL outputs from steps 3-4 (deterministic, no vLLM needed).
9. **REVL-03: Spawn Claude evaluator agents in parallel batches** (~50 agents/batch via `Agent(run_in_background=true)`); aggregate dimension coverage + consistency + coherence rates; writes `revl03_claude_eval.jsonl` + summary.
10. **REVL-05: Build human review file** (`scripts/build_human_review.py` reads stratified samples from eval_judge.pairs.jsonl + critique_then_fix output); writes `output/v1.2_human_review.md`. Present file to human; await HUMAN_APPROVED sentinel.
11. **MERGE GATE:** Verify all HARD gates passed AND HUMAN_APPROVED sentinel present → run the actual write of `models/qwen3-30b-wp-30_70-reasoning-merged/` to the canonical final location (idempotency anchor = `config.json` existence + special-token roundtrip pass).
12. **Post-merge validation:** Load merged model, run 10 × `<wp_gen>` + 10 × `<wp_judge>` samples; verify coherent + task-token routing correct.

**Note:** Step 11 is structurally redundant if the merge produced in Step 1 already lives at the canonical path. In practice, Step 1 can produce to a staging path (`models/qwen3-30b-wp-30_70-reasoning-merged.staging/`) and Step 11 promotes to canonical on full-pass + human approval — preserving D-04's "merge blocked until human sign-off" semantics. Planner chooses; both patterns work.
