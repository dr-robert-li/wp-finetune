# Phase 4.4 — D-05 failure diagnosis (post REVL-05 rejection)

**Date:** 2026-06-02. **Trigger:** REVL-05 REJECTED. CPU-only diagnosis before
iterate-vs-abandon, on the 2048-capture (120 cot+ctf) + eval_judge pairs (121).

## D3-A — terse-JSON census

| Mode | Count | % |
|------|-------|---|
| prose (per-dim `score X/10 —` or `[/REASONING]`) | 76 | 63.3% |
| **terse JSON-only** (bare `overall_score` blob, no reasoning prose) | **42** | **35.0%** |
| other/short | 2 | 1.7% |

35% terse — above the 25% "structural collapse" line in the D-05 decision tree.

## D3-B — terse trigger correlation (looking for a scoped cause)

- **Code length:** terse prompt median 684 chars vs prose 636 — no separation.
- **Training format:** terse cot/ctf = 31/13; prose cot/ctf = 54/22 — same ~36% terse
  ratio in BOTH formats. Not format-specific.
- **Difficulty (gt_canonical):** terse median 94.8 vs prose 94.8 — identical. The model
  does NOT go terse only on easy/high-score code.

**No trigger cluster.** Terse collapse is uniform across format, length, and difficulty —
consistent with a structural/format-stability issue, not a fixable data-pattern subset.

## D3-C — scoring calibration (false-positive pass + over/underscoring)

- model_overall ≥70 & **gt_canonical** <60: **5/121 = 4.1%** (low)
- model_overall ≥70 & **gt_teacher** <60: **17/83 = 20.5%** (high — but gt_teacher is the
  noisier SOFT diagnostic GT, not canonical)
- model OVERscores gt_canonical by >20: 6/121 (5%)
- model UNDERscores gt_canonical by >20: 43/121 (36%)

The model is, in aggregate, HARSHER than calibrated GT (underscores 36% vs overscores 5%),
so it is not globally lax. But the human-found invalid-PHP-pass + the 20.5% teacher-FP rate
show specific judgment failures (passing code it should fail). Calibration is not cleanly
broken (4.1% canonical FP) but is not trustworthy on the failure tail.

## Decision-tree read (from the D-05 plan)

- terse-JSON 35% (>25%) with NO trigger cluster → "structural collapse, adapter not
  salvageable AT THIS CONFIG" (points Option 2 / abandon-this-checkpoint).
- BUT the prose 63% majority is genuinely strong (human FULL samples 1–4: accurate
  security + complexity analysis) → capability exists, config is wrong, not the approach.

**Synthesis:** this is a *training-config / format-stability* failure (ckpt-72 collapses
to the bare-JSON judge schema 35% of the time), not a data-quality or approach failure.
That favors a TARGETED Phase 4.3 re-train (fix format stability — e.g. data-mix rebalance
toward prose-CoT, format-token / template consistency, or earlier checkpoint / lower LR to
avoid late-collapse), NOT permanent abandonment. The v1 baseline (merged-v2) remains the
available fallback to unblock Phase 7 in parallel if needed.

## Recommendation

Re-open Phase 4.3 with a format-stability objective targeting the terse-JSON collapse,
keeping the v1 merged-v2 baseline as the parallel fallback. Do NOT promote ckpt-72.
Root-cause of the collapse (data mix vs LR/epoch vs format tokens) is the first task of
the 4.3 re-plan — the diagnosis here scopes it to "format stability", not "bad data".
