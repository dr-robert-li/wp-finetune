<revl06_blocking_decision>
## BLOCKING — orchestrator must choose a REVL-06 disposition before W4-06 runs (Task W4-06-decision)

**Finding (verified, systemic, not a val-split artifact):** REVL-06 was specified to extract `<corrected_code>` from CtF model responses and lint/PHPCS/security-check them. But:
- All 35 val ctf rows + all 140 train ctf rows are `<wp_judge>` critique prompts whose targets are `<judge_output>` + prose "Fix:" suggestions — **0 `<corrected_code>` blocks**.
- The model was never trained on `<corrected_code>`; re-inferencing ctf prompts yields judge output, not fixes.
- Result: `extract_corrected_code_from_xml` returns None for every sample → parseable set empty → a HARD gate over the parseable subset is **vacuously PASS**, silently approving the merge. Strictly worse than no gate.

The three options the W4-06-decision checkpoint presents:

- **Option A — Lint teacher-authored `<corrected_code>`** from `data/phase4_reasoning/critique_then_fix/batches/*.json` (~6 `<corrected_code>` occurrences across 23 batch files; sparse). Runnable today. Tests training-data quality, NOT the model → cannot be a model-quality HARD gate. Demote to SOFT/informational if chosen. **Input-schema caveat:** the batch files are NOT in the captured `{example_idx, task_type, prompt, response, model_scores}` schema that `revl06_fix_correctness.py` consumes — `response` is the field the checker reads `<corrected_code>` out of. If Option A is chosen, an ADAPTER step is required: read each `data/phase4_reasoning/critique_then_fix/batches/*.json`, locate the field carrying the teacher `<corrected_code>` (the batch record's fix/completion text, NOT a captured `response` key), and re-emit a captured-shaped JSONL `{example_idx: <batch row id>, task_type: "ctf", prompt: <batch prompt or "">, response: <text containing <corrected_code>>, model_scores: null}` before pointing `--captured-jsonl` at it. Do NOT feed raw batch JSON to the checker — its schema differs from the captured `{response:...}` schema and `extract_corrected_code_from_xml(row["response"])` would see nothing.
- **Option B — Re-prompt the 35 ctf code snippets with explicit fix-task framing** (reuse `generate_critique_then_fix`'s `<corrected_code>` prompt) to elicit zero-shot fixes. Tests UNTRAINED behavior → expect a high unparseable rate → near-vacuous gate. Costs a second model-inference pass.
- **Option C — Declare REVL-06 N/A** for a judge-only reasoning model; document rationale; do NOT block merge on it (REVL-04 HARD already gates fix-quality via execution-correctness). REVL-06 becomes a recorded N/A in the gate ledger.

**Discriminating question:** REVL-06 tests whether *the model* produces correct fixes. Option A tests the wrong artifact; Option B tests untrained behavior. If neither is acceptable, Option C is the honest disposition. `scripts/revl06_fix_correctness.py` (the deterministic checker) is built regardless — it is reusable under A or B — but its EXECUTION + pass-rule is gated behind this decision.
</revl06_blocking_decision>
