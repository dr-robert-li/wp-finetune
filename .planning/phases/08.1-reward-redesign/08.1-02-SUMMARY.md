# Phase 08.1 Plan 02: Extended Diagnostic Logging Core Summary

Implemented measurement-independent diagnostic logging core (Priorities 1-3 of D-81-03 / 09-RL-LOGGING-REQS.md). The flat-reward episode was undiagnosable because rl_metrics.jsonl logged only the composite reward; frac_groups_all_zero — the primary kill/continue signal — was never emitted. This plan adds exactly the visibility that would have caught collapse immediately and is the prerequisite for Plan 04's offline signal-check.

## One-liner

Pre-drop per-group stats (frac_groups_all_zero, fix_correctness_mean, etc.) aggregated inside compute_rollout_advantages before the constant-group drop, threaded through the 3-tuple meta into an extended _log_step that now emits all Priority-1+2 fields to rl_metrics.jsonl every step.

## Implementation Details

### Task 1: Pre-drop aggregation in compute_rollout_advantages

Inserted a group-stats aggregation block at lines 503-579 of `scripts/rl_rollouts.py`, between the empty-input early-return and the `remove_constant_reward_groups(groups)` call (line 580). This satisfies the critical insertion-order constraint: frac_groups_all_zero is computed on the **unfiltered** groups list before uniform groups are dropped.

**New meta keys (merged via `**group_stats`):**
- `fix_correctness_mean`, `fix_correctness_std` — from Transition.logs per convention
- `consistency_mean`, `consistency_std`
- `group_reward_std_mean` — mean of per-group std(rewards)
- `frac_groups_all_zero` — fraction of groups where all rewards == 0.0 (pre-drop)
- `frac_groups_all_one` — fraction where all rewards > 0.9
- `frac_groups_nonuniform` — fraction where std(rewards) > 0
- `frac_reward_gt_0.9`, `frac_reward_lt_0.1` — flat sample fractions

Both return paths carry all new keys (None on empty-input early-return, computed values on normal path). The 3-tuple arity `(data, advantages, meta)` is preserved — no 4th element added.

The tinker_cookbook import was moved to after the empty-input check so the empty-path is testable without tinker installed.

### Task 2: _extract_entropy + _compute_e_frac_trend + extended _log_step

**`_extract_entropy(kl_metrics)`:** Reads from `kl_metrics["optim/entropy"]` — the actual key populated by `_compute_kl_metrics` (confirmed at rl_train.py:393-396). Returns None when absent or when the value is None. NOT `fb_out.metrics["entropy"]` — that key does not exist in production.

**`_compute_e_frac_trend(history, window=10)`:** OLS slope over the last `window` values of e_frac. Returns None when fewer than 2 data points. A negative slope signals routing collapse. Uses module-level `_efrac_history` deque (maxlen=200), appended in `_log_step` from `moe_metrics["e_frac_with_tokens:mean"]` each step.

**`_log_step` extension:** New `group_stats=None` kwarg (backwards-compatible — existing callers without the arg continue to work). Emits all Priority-1+2 fields. All numpy scalars wrapped in `float()` for JSON serialisation.

**`run_training_step`:** Stops discarding `_meta` from `compute_rollout_advantages` (line 675). Extracts group_stats dict from meta and passes it to `_log_step`. No other behaviour changed.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocking] tinker_cookbook import placed before the empty-input early-return**
- **Found during:** Task 1 implementation — empty-input test failed with `ModuleNotFoundError: No module named 'tinker_cookbook'`
- **Issue:** The original code imported tinker_cookbook at function entry (before the `n_input == 0` check). The empty-input early-return was unreachable in offline tests without tinker.
- **Fix:** Moved the `from tinker_cookbook.rl.data_processing import ...` to after the early-return guard.
- **Files modified:** `scripts/rl_rollouts.py`
- **Commit:** 6864e90

**2. [Rule 2 - Correctness] _extract_entropy source clarified — kl_metrics not fb_out.metrics**
- **Found during:** Pre-implementation advisor check (before writing any code)
- **Issue:** The plan specified `_extract_entropy(fb_out)` reading `fb_out.metrics['entropy']`, but `_compute_kl_metrics` (rl_train.py:393-396) is the only place entropy is computed, and it stores it in the kl_metrics dict under `'optim/entropy'`. `fb_out.metrics` never contains an `entropy` key in production (confirmed: the mock in `_wire` has `e_frac_with_tokens:mean`, `e_max_violation:mean`, `e_max_violation:max` only). Using `fb_out.metrics` would have passed unit tests while logging `entropy: null` in every production run.
- **Fix:** `_extract_entropy` accepts the `kl_metrics` dict (not `fb_out`) and reads `kl_metrics.get("optim/entropy")`. Tests updated accordingly.
- **Files modified:** `scripts/rl_train.py`, `tests/test_rl_rollouts_reward_shape.py`
- **Commit:** 4b9625a
