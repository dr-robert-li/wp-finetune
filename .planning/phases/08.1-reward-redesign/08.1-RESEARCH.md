## Research Question 2: REWARD-SHAPE MATH

### Lever 1: Graded `fix_correctness`

**Current code (`rl_rollouts.py:624-632`):**
```python

if _is_parseable_php(corrected):
    rubric = _extract_verifiable_signals(corrected)
    fix_score = float(rubric.overall) / 100.0  # [0,100] -> [0,1]
else:
    fix_score = 0.0   # <-- hard parse cliff
```

**Candidate fix forms (selection after measurement):**

Form A — Partial credit for parse failures (addresses parse-cliff 0-mode):
```python
# Instead of else: fix_score = 0.0
if _is_parseable_php(corrected):
    rubric = _extract_verifiable_signals(corrected)
    fix_score = float(rubric.overall) / 100.0
elif corrected is not None and len(corrected.strip()) > 0:
    # Non-empty but unparseable: small partial credit
    fix_score = 0.10  # tunable constant; probe post-edit to check spread
else:
    fix_score = 0.0   # empty / no corrected block
```

Form B — Sub-check fraction (graded within parseable — relevant if checks are not lax):
```python
# rubric.overall already does sub-check weighted sum; expose component breakdown
rubric = _extract_verifiable_signals(corrected)
# rubric has per-dimension subscores; weighted fraction:
fix_score = (
    rubric.phpcs_score * _W_PHPCS +
    rubric.security_score * _W_SECURITY +
    rubric.verpo_score * _W_VERPO
) / 100.0  # normalized to [0,1]
```
Note: `reward_pipeline.py` VeRPO machinery (`_verpo_group`, lines 336-409) already does difficulty-weighted partial credit for WP_STANDARDS_CHECK_IDS. Reuse this rather than reimplementing.

Form C — Fix-delta vs original buggy code (discriminates effort):
```python
orig_rubric = _extract_verifiable_signals(original_buggy_code)
new_rubric  = _extract_verifiable_signals(corrected)
delta = (new_rubric.overall - orig_rubric.overall) / 100.0
fix_score = max(0.0, delta)  # only reward improvement; floor at 0
```
Delta form requires the original buggy code to be available in the rollout context.

### Lever 2: Weight Rebalance

**Current `combine_judge_reward` (`rl_rollouts.py:172-204`):**
```python
def combine_judge_reward(fix_correctness, consistency, weight=0.3):
    if weight > 0.5:
        raise ValueError(f"weight {weight} > 0.5: fix_correctness must be primary")
    return (1 - weight) * fix_correctness + weight * consistency
```

**Hard constraint:** `weight` (consistency fraction) capped at 0.5 by the guard. Raising beyond 0.5 requires changing the guard itself (not silently violating it). Doing so requires rationale: the guard exists because fix_correctness is the verifiable-quality anchor; making consistency dominant risks training toward "sounding right" without code quality.

**Safe rebalance within guard:**
```python
# Shift from default 0.3 → 0.45 consistency weight (still within guard)
combine_judge_reward(fc, cons, weight=0.45)
# Result: fix_correctness contributes 0.55x, consistency 0.45x
```

**If guard must be relaxed (measurement shows consistency is well-graded, fix_correctness truly binary):**
```python
# Remove or relax the guard with explicit rationale comment
# New cap: 0.7 (consistency can now dominate)
if weight > 0.7:
    raise ValueError(...)
```

**Key invariant:** `combine_judge_reward` result feeds into `Transition.logs["fix_correctness"]` and `Transition.logs["consistency"]` separately, so the component logging (Priority 1 from logging spec) must aggregate BOTH fields from `Transition.logs`, not from the combined output.

### Lever 3: Per-Group Advantage Shaping

**Where GSPO groups are processed (`rl_rollouts.py:442-508`):**
```python
def compute_rollout_advantages(rollouts, ...):
    rollouts = remove_constant_reward_groups(rollouts)  # DROPS uniform groups
    advantages = compute_advantages(rollouts, ...)       # per-group normalization
    return assemble_training_data(rollouts, advantages, ...)
```

**`remove_constant_reward_groups` behavior (confirmed from tinker_cookbook):**
- Drops groups where all G rewards are equal
- Returns `groups[0:1]` (singleton with zero advantages) when ALL groups are constant — NOT empty list
- So if all groups are uniform, training still runs but with zero-gradient data

**Advantage floor approach:**
```python
# In compute_advantages or before normalization:
# Add small noise floor to prevent exactly-zero advantages
advantages = (rewards - mean_r) / (std_r + eps)
advantages = advantages + advantage_floor  # e.g., 0.05 * sign(rewards - mean_r)
```

**Raw-non-normalized blend (alternative):**
```python
# Blend normalized advantages with raw reward signal
alpha = 0.2  # blend factor
advantages_final = (1 - alpha) * advantages_normalized + alpha * (rewards - global_mean)
```

**Diversity shaping (group-level):**
```python
# Scale group advantages by group diversity measure
group_std = np.std(group_rewards)
diversity_weight = min(1.0, group_std / target_std)  # target_std ~ 0.2
advantages *= diversity_weight  # zero-weight for uniform groups, full-weight for diverse
```

Note: Lever 3 is structural — it helps regardless of which reward term is binary. It is the weakest fix if the root cause is reward-signal poverty (all groups genuinely have the same score); better to fix the reward signal than to amplify zero-signal groups.

def compute_rollout_advantages(rollouts, ...):
    # ---- NEW: pre-drop group stats ----
    group_rewards = {}
    fix_correctness_vals = []
    consistency_vals = []
    for tg in rollouts:
        gid = tg.group_id  # gen-<k> or judge-<k>
        rewards = tg.get_total_rewards()
        group_rewards[gid] = rewards
        for t in tg.transitions:
            fc = t.logs.get("fix_correctness")
            cons = t.logs.get("consistency")
            if fc is not None: fix_correctness_vals.append(fc)
            if cons is not None: consistency_vals.append(cons)

    per_group_stds = [np.std(v) for v in group_rewards.values()]
    n_groups = len(group_rewards)
    group_stats = {
        "fix_correctness_mean": float(np.mean(fix_correctness_vals)) if fix_correctness_vals else None,
        "fix_correctness_std":  float(np.std(fix_correctness_vals))  if fix_correctness_vals else None,
        "consistency_mean":     float(np.mean(consistency_vals))      if consistency_vals else None,
        "consistency_std":      float(np.std(consistency_vals))       if consistency_vals else None,
        "group_reward_std_mean": float(np.mean(per_group_stds)) if per_group_stds else None,
        "frac_groups_all_zero": sum(1 for v in group_rewards.values() if all(x == 0 for x in v)) / n_groups,
        "frac_groups_all_one":  sum(1 for v in group_rewards.values() if all(x > 0.9 for x in v)) / n_groups,
        "frac_groups_nonuniform": sum(1 for v in group_rewards.values() if np.std(v) > 0) / n_groups,
        "frac_reward_gt_0.9":   sum(1 for fc in fix_correctness_vals if fc > 0.9) / max(len(fix_correctness_vals), 1),
        "frac_reward_lt_0.1":   sum(1 for fc in fix_correctness_vals if fc < 0.1) / max(len(fix_correctness_vals), 1),
    }
    # ---- end new ----

    rollouts = remove_constant_reward_groups(rollouts)
    advantages = compute_advantages(rollouts, ...)
    data = assemble_training_data(rollouts, advantages, ...)
    return data, group_stats  # return stats alongside data

import torch
def _extract_entropy(fb_out):
    if hasattr(fb_out, "entropy"):
        return fb_out.entropy.mean().item()
    if "entropy" in getattr(fb_out, "metrics", {}):
        return fb_out.metrics["entropy"]
    # Fallback: approximate from logprobs
    if hasattr(fb_out, "logprobs"):
        lp = fb_out.logprobs  # shape [B, T]
        p = torch.exp(lp)
        return -(p * lp).sum(-1).mean().item()
    return None

```python
def _compute_e_frac_trend(metrics_history: list[dict], window: int = 10) -> float | None:
    if len(metrics_history) < window:
        return None
    recent = [m["e_frac"] for m in metrics_history[-window:] if "e_frac" in m]
    if len(recent) < 3:
        return None
    import numpy as np
    x = np.arange(len(recent))
    slope, _ = np.polyfit(x, recent, 1)
    return float(slope)
```

## Common Pitfalls

### Pitfall 1: Computing `frac_groups_all_zero` After `remove_constant_reward_groups`

**What goes wrong:** Inserting the stat computation after the `remove_constant_reward_groups` call — the uniform groups that define the pathology have already been dropped, so you compute stats on the non-uniform subset and get a falsely healthy reading.
**Why it happens:** The drop call appears first in `compute_rollout_advantages`; it's easy to assume the remaining rollouts are the right population.
**How to avoid:** All per-group stats aggregation must happen BEFORE the `remove_constant_reward_groups` call, using the full unfiltered `rollouts` list.
**Warning signs:** `frac_groups_all_zero` reports 0.00 even when reward is flat.

### Pitfall 2: Attempting to Histogram `rubric.overall` from RLEV-01 Captures

**What goes wrong:** Running `_extract_corrected_php` on `output/rl_eval/*/judge_responses.jsonl` and finding 0 parseable outputs, then concluding "no data available."
**Why it happens:** RLEV-01 uses `eval_judge.run_eval` which elicits PROSE critique — a different prompt format from the RL training judge prompt (`_augment_judge_prompt`). The eval-path responses contain no corrected PHP blocks.
**How to avoid:** Use fresh probe completions generated with the training judge prompt. Extend `_probe_rl_reward.py` rather than replaying RLEV-01 captures.
**Warning signs:** Zero parseable hits on the entire RLEV-01 corpus (exactly as found in this session).

### Pitfall 3: Silently Violating `combine_judge_reward` Weight Guard

**What goes wrong:** Setting `weight > 0.5` in the Lever 2 rebalance without modifying the guard → `ValueError` at runtime → training crash.
**Why it happens:** The guard is a `raise ValueError`, not a clamp — it throws, it doesn't silently ignore.
**How to avoid:** If weight must exceed 0.5, explicitly modify the guard AND document the rationale. Within 0.5, safe to change just the call site.
**Warning signs:** `ValueError: weight X > 0.5: fix_correctness must be primary` in training logs.

### Pitfall 4: Histograming Only Parseable Subset Without Reporting Parse Failure Rate

**What goes wrong:** The histogram looks graded (spread 10-90) but 60% of outputs are parse failures, so the actual discriminating signal is still poor — the histogram alone looks good but the measurement conclusion is wrong.
**How to avoid:** Always report parse failure rate alongside the histogram. The two numbers together determine the pathology type.

### Pitfall 5: Advantage Floor Applied After MO-GRPO Normalization

**What goes wrong:** Lever 3 advantage floor added after `_mo_grpo_norm` in `reward_pipeline.py:200-220` — the floor is re-normalized away on the next batch.
**Why it happens:** MO-GRPO normalization is applied per-group; adding a constant before normalization is equivalent to no change after normalization.
**How to avoid:** Advantage floors / diversity weights must be applied AFTER `_mo_grpo_norm`, outside the normalization step. Or: skip normalization for the floor signal and blend raw + normalized.

### Pitfall 6: `remove_constant_reward_groups` Returns Singleton, Not Empty List

**What goes wrong:** Assuming that when all groups are uniform, training stops. Actually, `remove_constant_reward_groups` returns `groups[0:1]` (one group with zero advantages), so training continues with effectively zero-gradient data but no error.
**Why it matters:** The training loop appears healthy (no crash, metrics emit) but gradient is zero. Only `frac_groups_all_zero` in the new logging reveals this.
