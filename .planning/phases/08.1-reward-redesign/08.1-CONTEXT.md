<domain>
## Phase Boundary

Restore a **per-group-discriminating reward signal** so a GSPO rerun gets a non-zero
advantage and an actual learning gradient ‚Äî the first full Phase 9 run was mechanically
perfect but FLAT (training reward ~0.27 over 250 steps; RLEV-01 teacher-Spearman flat,
no checkpoint beyond noise vs warmstart).

8.1 is a corrective iteration on Phase 8's shipped+verified reward pipeline, triggered by
Phase 9's empirical flat-reward finding. The leading mechanism is reward-shape collapse:
the dominant 0.7 `fix_correctness` term is *suspected* effectively-binary (4 samples of a
"fixable" prompt all ~1, "unfixable" all ~0 ‚Üí uniform GSPO groups ‚Üí normalized advantage
~0 ‚Üí vanishing gradient).

**Critical framing (locked D-81-01):** that "binary" diagnosis is CIRCUMSTANTIAL.
`fix_correctness = rubric.overall/100` is graded *in code* (`rl_rollouts.py:629`); the
binary claim comes from n=15 *divergent* rollouts (extreme by construction, "biased toward
|fix-cons|>0.3 but indicative" per Section H) and `frac_groups_all_zero` was never logged.
So 8.1 MEASURES the real reward-shape pathology before designing the fix ‚Äî the two plausible
worlds (parse-cliff 0-mode vs verifiable-checks-too-lax >90 cluster) need different fixes.

**Out of scope (own phases):**
- The GSPO rerun itself ‚Äî Phase 9 (fresh warm-start from v4, NO resume; gated by 8.1's
  logging confirming a restored gradient on a short signal-check).
- RL-vs-SFT comparative evaluation ‚Äî Phase 10.
- Any change to Phase 8's verified guarantees (security hard gate, MO-GRPO normalization,
  anti-hack thresholds) beyond the reward-shape of the verifiable term ‚Äî those must still pass.

<decisions>
## Implementation Decisions

### Sequencing ‚Äî measure before designing (D-81-01)
- **D-81-01 (LOCKED):** **Measure-first, then design.** 8.1's first deliverable is
  instrumentation, not a reward edit. Replay `_extract_verifiable_signals` over captured
  judge completions (and/or a short probe run) and histogram `rubric.overall` on the
  **parseable** subset + per-group reward std. Design the reward shape from the OBSERVED
  distribution. Do NOT pre-lock "graded partial credit = fraction of sub-checks" ‚Äî that
  only fixes one of the two candidate pathologies.
- **Discriminating measurement (advisor-mandated):** on parseable judge completions ‚Äî
  (a) `rubric.overall` spread across 10‚Äì90 ‚Üí rubric is graded, the parse gate (`else: 0.0`)
  creates the 0-mode ‚Üí fix = handle the parse cliff / partial credit; (b) clusters >90
  (which `frac>0.9=0.47` on parseable code implies) ‚Üí verifiable checks too LAX to
  discriminate ‚Üí fix = finer/harder checks or shift weight onto the graded consistency term.
- **Data-source caveat:** `judge_failures*.jsonl` is failure-biased (don't histogram it
  alone). Prefer the RLEV-01 captures (`output/rl_eval/*/judge_responses.jsonl` ‚Äî real
  judge completions on the val set at temp 0.0) or a fresh short probe.

### Reward-shape levers in-scope (D-81-02) ‚Äî all four, selection pending measurement
- **D-81-02 (LOCKED):** ALL FOUR levers are in-scope for 8.1; which combination ships is
  decided from the measurement (D-81-01), not pre-committed:
  1. **Graded `fix_correctness`** ‚Äî partial credit on the verifiable term. Candidate forms:
     fraction of weighted PHPCS/security/syntax sub-checks; de-saturate the parse cliff
     (graded penalty instead of `else: 0.0`); or fix-delta vs the original buggy code.
  2. **Weight rebalance** ‚Äî move weight off the saturated 0.7 anchor toward the
     already-graded consistency term (frac_mid=0.73) or the graded gen rewards. Cheap, no
     new scoring code. Must respect the existing weight-validation invariant in
     `combine_judge_reward` (`rl_rollouts.py:173` ‚Äî fix_correctness anchored as primary;
     revisit that guard if rebalancing past it).
  3. **Per-group advantage shaping** ‚Äî advantage floor / raw-non-normalized blend /
     diversity shaping so uniform groups still contribute gradient (handoff Mechanism-3).
     Structural ‚Äî helps regardless of which term is binary.
  4. **Harder/finer verifiable checks** ‚Äî only relevant in the "lax-checks >90 cluster"
     world; add finer/harder checks so the term discriminates.

### Logging / telemetry scope (D-81-03) ‚Äî FULL spec
- **D-81-03 (LOCKED):** Implement the **FULL** `09-RL-LOGGING-REQS.md` spec (all 6 priority
  tiers), not just the priority-1‚Äì3 minimum:
  1. Reward-signal diagnostics: component means (fix_correctness, consistency), group-level
     variance (`group_reward_std_mean`, `frac_groups_all_zero`/`all_one`/`nonuniform`),
     saturation sentinels. **`frac_groups_all_zero` is the primary kill/continue signal.**
  2. Policy/optimizer health: `entropy` (the single most informative add), `grad_norm`,
     `lr`, `loss`, `e_frac_trend_10`.
  3. Per-container telemetry: VRAM, tokens/sec, step wall-time, checkpoint-wrote bool;
     judge + consistency latency / score-distribution histograms / queue depth.
  4. Score-distribution histograms (judge + consistency) ‚Äî earliest positive-learning signal.
  5. Structured status-tick JSON + automated window means.
  6. Codified kill/continue decision rule (`should_flag_for_review`).
- This is what would have diagnosed the flat reward immediately; it is the prerequisite that
  lets the Phase 9 rerun confirm a restored per-group gradient on a SHORT signal-check before
  spending ~3 days.

### Path scope ‚Äî gen vs judge (D-81-04)
- **D-81-04 (LOCKED):** **Decide from measurement.** Histogram BOTH the gen-path
  (Phase 8 composite ‚Äî PHPCS/security/VeRPO/MO-GRPO) and judge-path (`fix_correctness` +
  consistency) reward components; touch only whichever path the data shows is saturated.
  The flat composite mixes both paths ‚Äî do not assume judge-path-only. If gen-path is healthy,
  leave it untouched to preserve Phase 8's verified guarantees and limit blast radius.

### Claude's Discretion
- Exact reward-shape math (sub-check weighting scheme, parse-cliff penalty curve, advantage
  floor value, fix-delta formulation) ‚Äî chosen by research/planning from the measured
  distribution, then validated against the new per-group logging.
- Implementation mechanics of the logging (where to aggregate `Transition.logs`, JSONL
  schema details) ‚Äî follow the touch-points already enumerated in `09-RL-LOGGING-REQS.md`.

- Reward shape: `rl_rollouts.py` fix_correctness block + `reward_pipeline.py` composite ‚Äî
