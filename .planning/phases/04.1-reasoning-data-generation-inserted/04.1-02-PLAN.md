**Seed loading:**
- `load_critique_seeds()` -- loads both seed files, filters to `seed_type == "critique_then_fix"`, returns list (expected 59 seeds)
- `sample_seeds(seeds, n=3)` -- same boundary-weighted random sampling as deep_judge_cot.py

**Few-shot exemplar formatting for critique seeds:**
- `format_critique_seed_as_exemplar(seed)` -- formats showing: defective_code (truncated 1500 chars), then human_critique with summary, dimensions dict (severity + issue + fix per dimension), key_observation, then corrected_code in `<corrected_code>` tags

**Source function loading:**
- `load_failed_functions()` -- scans FAILED_DIR only (per D-02, critique-then-fix uses failed functions, NOT mutation pool). Loads each JSON file, extracts individual functions. Each becomes dict with `code`, `source_file`, `function_name`. Filters out functions with code length < 50 chars. Returns list.

**PHP lint check (addresses review concern #2):**
- `php_lint_check(code)` -- writes `code` to a temporary file, runs `php -l <tmpfile>`, returns a dict:
  - `valid`: bool -- True if `php -l` exits with code 0
  - `errors`: str -- stderr output if invalid, empty string if valid
  Uses `subprocess.run(["php", "-l", tmpfile_path], capture_output=True, text=True, timeout=10)`. Uses `tempfile.NamedTemporaryFile(suffix=".php", delete=False)` and cleans up after.
  If `php` is not available (FileNotFoundError), returns `{"valid": True, "errors": "php not available - skipped"}` with a warning printed to stderr (graceful degradation -- don't block generation if PHP CLI isn't installed).

**Critique-fix alignment check (addresses review concern #2):**
- `check_critique_fix_alignment(critique, defective_code, corrected_code)` -- for each dimension in `critique["dimensions"]` where severity is "critical" or "high":
  1. Extract the `fix` suggestion text
  2. Extract key API names mentioned in the `fix` text (scan for WP_API_CITATIONS strings)
  3. For each API mentioned in the fix: check if it appears in `corrected_code` but NOT in `defective_code` (i.e., the fix was actually applied)
  4. Returns a dict:
     - `critical_high_issues`: int -- count of critical/high severity issues
     - `addressed_issues`: int -- count where fix APIs appear in corrected_code
     - `unaddressed_issues`: list[dict] -- each with `dimension`, `severity`, `fix_text`, `missing_apis`
     - `alignment_ratio`: float -- addressed / critical_high_issues if > 0, else 1.0
  Note: This is a heuristic, not a proof. An alignment_ratio < 0.3 is a strong signal of broken fix. Ratio > 0.3 passes. Issues with no API names in the fix text are counted as "addressed" (can't verify non-API fixes automatically).

**Core generation function:**
- `generate_critique_then_fix(code, source_info, seeds, client)` -- takes defective PHP code, source metadata, list of critique seeds, and anthropic client. Samples 3 seeds. Constructs prompt with:
  1. System instruction: "You are a WordPress code quality critic producing structured critiques for training data. You receive defective PHP code and must: (a) produce a structured critique analyzing ALL 9 dimensions with severity (critical/high/medium/low), issue description, and fix suggestion for each; (b) produce the corrected version of the code wrapped in <corrected_code> XML tags. IMPORTANT: The corrected code must actually fix the issues you identify. If you cite a missing $wpdb->prepare, the corrected code must contain $wpdb->prepare."
  2. The 3 formatted critique seed exemplars
  3. "NOW CRITIQUE AND FIX:" block with the target defective code (truncated to 3000 chars)
  4. Explicit output schema: "Return JSON with keys: summary (string), dimensions (object with ALL 9 dimensions: wpcs_compliance, sql_safety, security, performance, wp_api_usage, code_quality, dependency_integrity, i18n, accessibility -- each having severity (critical|high|medium|low), issue (string), fix (string)), key_observation (string), corrected_code (string containing the fixed PHP code). IMPORTANT: The corrected_code value must be the actual corrected PHP source, not wrapped in additional tags."
  5. WP API citation instruction: "When WordPress APIs appear in the defective code or are needed in the fix, name them explicitly: $wpdb->prepare(), wp_verify_nonce(), esc_html(), current_user_can(), etc."

- Calls `call_with_backoff(client, model="claude-sonnet-4-6-20250514", max_tokens=3072, messages=[{"role": "user", "content": prompt}])` (higher max_tokens than CoT because corrected code is included)
- Extracts result via `extract_json(resp.content[0].text)`
- If `corrected_code` not in extracted JSON, attempts fallback extraction: search for `<corrected_code>` XML tags in the raw response text via regex `re.search(r'<corrected_code>([\s\S]+?)</corrected_code>', resp.content[0].text)` and inject into result dict
- Returns the parsed dict or None

**Quality gate per example (strengthened -- addresses review concern #2):**
- `passes_quality_gate(result, defective_code=None)` -- checks:
  1. `summary` is non-empty string
  2. `dimensions` has all 9 required dimensions
  3. Each dimension has `severity` in SEVERITY_LEVELS, `issue` (non-empty), `fix` (non-empty)
  4. `corrected_code` is non-empty string with length > 20
  5. `corrected_code` differs from `defective_code` (if provided) -- the fix must actually change something
  6. **NEW -- PHP lint check:** If `defective_code` is provided (meaning we're in full validation mode, not just schema check), run `php_lint_check(corrected_code)`. If `valid` is False, return False. Log the lint errors for debugging.
  7. **NEW -- Critique-fix alignment:** If `defective_code` is provided, run `check_critique_fix_alignment(result, defective_code, corrected_code)`. If `alignment_ratio < 0.3` AND `critical_high_issues > 0`, return False (the fix doesn't address the critique).
  8. Returns True/False

**Training example formatter:**
- `format_training_example(source_info, result, defective_code=None)` -- assembles: `source_file`, `function_name`, `defective_code` (from source), `critique` (summary + dimensions + key_observation from result), `corrected_code` (from result), `dimensions_addressed` (always all 9), `generation_method` ("seed_few_shot_agent"), `php_lint` (dict from php_lint_check), `critique_fix_alignment` (dict from check_critique_fix_alignment if defective_code provided)

**Pilot validation:**
- `validate_pilot_batch(examples)` -- checks:
  1. All 9 dimensions appear across examples (check critique.dimensions keys)
  2. All 4 severity levels appear at least once across examples
  3. At least 3 WP_API_CITATIONS appear in critique text
  4. corrected_code is non-empty for all examples
  5. **NEW -- PHP lint pass rate:** count examples where php_lint.valid is True, compute percentage
  6. **NEW -- Alignment rate:** compute mean critique_fix_alignment.alignment_ratio across examples
  7. Returns dict with missing_dimensions, severity_coverage, api_citations_found, php_lint_pass_rate, mean_alignment_ratio

**Structured Review Rubric (addresses review concern #4):**

Review 3-5 examples from each stream using this rubric. For each example, score PASS/FAIL on each criterion:

**Deep Judge CoT Rubric:**
| # | Criterion | PASS condition | FAIL condition |
|---|-----------|----------------|----------------|
| R1 | Dimension completeness | All 9 dimensions present in dimension_analysis | Any dimension missing |
| R2 | Score ranges | All scores are integers 1-10 | Any score outside 1-10 |
| R3 | Citation grounding | API names cited in analysis actually appear in the source code, OR are explicitly described as missing/needed | API names cited as present but not in source code (hallucination) |
| R4 | Reasoning specificity | Analysis text references specific code patterns, line behaviors, or function calls -- not generic statements | Analysis is generic (e.g., "code quality is good" without specifics) |
| R5 | Verdict consistency | Verdict (PASS/FAIL) is consistent with scores (FAIL if security < 5 or overall < 40) | Verdict contradicts scores |
| R6 | Citation accuracy metadata | `citation_accuracy.hallucination_ratio` < 0.5 for this example | hallucination_ratio >= 0.5 |

**Critique-Then-Fix Rubric:**
| # | Criterion | PASS condition | FAIL condition |
|---|-----------|----------------|----------------|
| C1 | Dimension completeness | All 9 dimensions present in critique.dimensions | Any dimension missing |
