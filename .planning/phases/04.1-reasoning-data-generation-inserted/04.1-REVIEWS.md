# Cross-AI Plan Review — Phase 4.1

## Gemini Review

### 1. Summary
The implementation plans for Phase 4.1 are technically sound, logically sequenced, and rigorously aligned with the project's architectural decisions (D-01 through D-12). The "Test-First" approach in Wave 1 and the "Human-in-the-loop" gate in Wave 2 are excellent safeguards for ensuring the high data quality required for reasoning-based fine-tuning. By leveraging the 118 "golden seeds" and Phase 1 outputs, the plan efficiently bypasses the empty mutation pool issue. However, the primary risk lies in the **low data volume** relative to a 30B parameter model and the potential for **reasoning hallucinations** (e.g., incorrect line citations) that could degrade the model's performance as a judge.

### 2. Strengths
*   **Groundedness:** The requirement for WP-specific line citations and dimension-by-dimension analysis ensures the generated reasoning is not generic but tied to specific code artifacts.
*   **Methodological Rigor:** Starting with "Wave 0" test stubs (Task 0) ensures that the complex parsing logic (9 dimensions, XML tags, severity levels) is verified before bulk API spend.
*   **Effective Use of Prior Work:** Utilizing Phase 1 "Passed" and "Failed" functions as the substrate for generation is a smart way to scale the 118 seeds into a larger dataset without needing a mutation engine.
*   **Safety Gates:** The inclusion of a pilot batch (40 examples) and a blocking human review gate is critical for reasoning data, where "correct-looking but wrong" logic is more damaging than simple parse failures.
*   **Tool Reuse:** Correctly identifies and utilizes existing pipeline utilities (`extract_json`, `call_with_backoff`, `checkpoint/resume`), which reduces implementation time and increases reliability.

### 3. Concerns

#### **Volume vs. Model Capacity (MEDIUM SEVERITY)**
The targets (~200 CoT and ~100 CtF examples) are very low for a model with 3B active parameters. While reasoning data is "high signal," this volume may be insufficient to "shift" the model's internal logic significantly. It risks being treated as a style-transfer task rather than a deep reasoning capability.

#### **Citation Hallucination (HIGH SEVERITY)**
Claude agents are prone to hallucinating line numbers or specific code snippets when performing "dimension-by-dimension analysis" on long functions. If the quality gate only checks for the *presence* of citations and not their *accuracy*, the model will be trained on "hallucinated reasoning," making it an unreliable judge.

#### **Critique-Fix Alignment (MEDIUM SEVERITY)**
In Plan 02, the quality gate validates "severity levels" and "code length," but there is no explicit check that the `corrected_code` actually addresses the issues raised in the `critique`. Training on mismatched triples will break the model's causal reasoning chain.

#### **Routing Token Integration (LOW SEVERITY)**
The plans mention `format_training_example()` but do not explicitly state that the `<wp_gen>` and `<wp_judge>` task tokens will be prepended. If these aren't baked into the reasoning examples, the MoE routing may not align.

### 4. Suggestions
*   **Add Citation Verification:** In `passes_quality_gate()`, add a simple regex or string-match check to ensure that any code snippet mentioned in the CoT actually exists in the source function.
*   **Triple-Validation in CtF:** Update the critique-then-fix quality gate to verify that `corrected_code` is syntactically valid (e.g., basic linter for PHP) and different from the original `defective_code`.
*   **Scale the Target:** If the pilot is successful, consider increasing the bulk target to cover 100% of Phase 1 functions (~400 examples).
*   **Exemplar Diversity:** In `sample_seeds()`, ensure selection is "dimension weighted" too — if a target function is a "Security" failure, provide security-focused seed exemplars.

### 5. Risk Assessment

**Overall Risk Level: MEDIUM**

The technical execution plan is high-quality. The "Medium" rating is driven by the data density problem: 300 examples for a 30B model is a very thin layer. Success depends on the signal-to-noise ratio being exceptionally high.

## Codex Review

### Plan 01

**Summary:** Plan 01 is a solid foundation: test stubs first, seed import with schema validation, and a structured deep judge CoT generator. The main weakness is dataset selection policy, seed-to-target leakage prevention, and what exactly counts as acceptable citation quality.

**Strengths:**
- Starts with test stubs before implementation
- Seed import validates schema against D-05
- Uses existing utilities rather than inventing new infrastructure
- Quality gate checks all 9 rubric dimensions and score ranges
- Boundary-weighted seed sampling increases exemplar diversity

**Concerns:**
- **HIGH:** No train/eval contamination guard. Phase 1 functions may overlap with later eval sets.
- **HIGH:** "WP API citations present" is underspecified. Presence-only checks are easy to game.
- **HIGH:** Requiring all 9 dimensions on every example may create forced low-quality reasoning when dimensions are genuinely inapplicable.
- **MEDIUM:** Test stubs alone not enough — no fixture design, golden-output assertions, malformed-response tests, or checkpoint corruption tests.
- **MEDIUM:** Seed import from external desktop path creates reproducibility risk without checksums.
- **MEDIUM:** No token-budget strategy for deep 9-dimension analyses with citations.
- **LOW:** No duplicate/near-duplicate Phase 1 function detection before generation.

**Suggestions:**
- Define strict citation schema with source-text alignment validation
- Add leakage controls: manifest input IDs, deduplicate against seeds and eval sets
- Allow "not applicable" under constrained conditions for inapplicable dimensions
- Expand tests beyond stubs: malformed JSON, fake citations, retry exhaustion, checkpoint recovery
- Version imported seed bundle with manifest containing hashes

**Risk:** MEDIUM

### Plan 02

**Summary:** The most important control point — introduces the second stream and human review gate. Correctly aligned with D-02 and DGEN-03. Main issue: review criteria too qualitative, critique-then-fix stream lacks safeguards for fix correctness.

**Strengths:**
- Correctly documents D-02 override
- Mirrors deep judge script structure for consistency
- Human review is explicitly blocking
- Pilot execution for both streams before bulk

**Concerns:**
- **HIGH:** Critique quality gate too weak — valid severity labels and non-empty corrected_code don't prove the critique is accurate or the fix resolves the issue.
- **HIGH:** No execution- or lint-based validation for corrected code. Risk of training on broken "fixes."
- **HIGH:** Human review criteria underspecified — without a rubric, reviewers may approve inconsistent outputs.
- **MEDIUM:** XML-tag extraction fallback suggests output-format fragility.
- **MEDIUM:** No policy for multi-defect failed functions (over-fix, under-fix).
- **LOW:** No mention of balancing pilot sampling across defect subtlety.

**Suggestions:**
- Add automated fix validation: syntax parse, PHP lint, static checks
- Define structured human review rubric with pass/fail fields
- Require critique outputs to map each issue to code span or line reference
- Stratify pilot sampling by defect type and subtlety

**Risk:** MEDIUM-HIGH

### Plan 03

**Summary:** Intentionally narrow bulk-generation wave. Correctly gates on Plan 02 human approval. Too thin operationally — missing distribution balance, deduplication, failure handling, and final parse-quality enforcement.

**Strengths:**
- Correctly gates on human approval
- Reuses checkpoint/resume
- Conservative targets reduce cost and review burden
- Clean stream separation

**Concerns:**
- **HIGH:** Success checks are count-based, not quality-based. ">= 100 examples" is too weak.
- **HIGH:** Output volume probably too small for meaningful standalone fine-tuning impact on 30B MoE.
- **HIGH:** No explicit parse-failure accounting despite < 2% requirement.
- **MEDIUM:** No deduplication or diversity checks across bulk outputs.
- **MEDIUM:** No drift check after pilot — bulk may produce different quality at scale.
- **LOW:** No export/packaging step for downstream training ingestion.

**Suggestions:**
- Add bulk-level acceptance metrics: parse-failure rate, citation-validity rate, duplicate-rate threshold
- Reframe volume as bootstrap dataset, not standalone capability driver
- Perform deduplication on prompt and output sides
- Add post-bulk audit sample with fixed review quotas
- Define final output schema and packaging contract

**Risk:** MEDIUM-HIGH

## Consensus Summary

### Agreed Strengths
- **Phased workflow with human gate** — both reviewers praised the pilot-then-review-then-bulk sequencing as the right design for reasoning data (Gemini: "excellent safeguards", Codex: "strongest part of the design")
- **Effective reuse of existing infrastructure** — extract_json, call_with_backoff, checkpoint/resume pattern from prior phases
- **Test-first approach** — Wave 0 stubs before implementation reduces regression risk
- **D-02 override properly documented** — mutation pool workaround is clean and explicit

### Agreed Concerns
1. **HIGH — Citation/reasoning quality gates are too shallow** — Both reviewers flagged that checking for citation *presence* without verifying *accuracy* is dangerous for reasoning training data. Gemini: "hallucinated reasoning." Codex: "presence-only checks are easy to game."
2. **HIGH — Critique-then-fix alignment unverified** — No check that corrected_code actually fixes the critiqued issues. No PHP lint/syntax validation on generated fixes. Both reviewers flagged this independently.
3. **HIGH — Volume may be insufficient** — ~300 total examples is very thin for a 30B-class model. Both reviewers noted this is acceptable as high-quality supplement within a larger training mix, but insufficient as a standalone capability driver.
4. **MEDIUM — Human review criteria underspecified** — No structured rubric for the reviewer, risking inconsistent approval decisions.

### Divergent Views
- **"All 9 dimensions always" policy** — Codex flagged forcing all 9 dimensions as potentially creating low-quality forced reasoning for inapplicable dimensions. Gemini did not raise this concern. This is worth investigating — perhaps allow "N/A" with justification for genuinely inapplicable dimensions.
- **Train/eval contamination** — Codex specifically flagged leakage risk between Phase 1 functions and future eval sets. Gemini did not mention this. Worth adding provenance tracking.
- **Seed import reproducibility** — Codex recommended checksumming imported seeds for version control. Gemini did not raise this. Low cost to add.
