## Summary

Phase 4.1 generates two parallel streams of reasoning training data for the 30/70 adapter. The primary mechanism is few-shot agent generation: Claude Code agents receive 3-5 golden seed examples as in-context exemplars and produce new examples for Phase 1 functions following the seed patterns. This is a content production phase, not a systems integration phase — all infrastructure patterns (agent loops, checkpoint/resume, extract_json, Claude Code agent spawning) already exist in the codebase.

The seed data is well-structured and already confirmed in the source directory (`~/Desktop/data/wp-finetune-data/`). Seeds have two distinct schemas: `deep_judge_cot` seeds carry `human_reasoning` with `verdict`, `dimension_analysis`, and `overall_score`; `critique_then_fix` seeds carry `defective_code`, `human_critique` with `dimensions` dict (severity per dimension), and `corrected_code`. These schemas define the target output format for generated examples — agents must produce output matching seed structure.

The key planning decisions are: (1) seed import step must come first to copy seeds into `data/seeds/`; (2) pilot batch is a hard gate before bulk generation — confirm WP-specific API citations appear by name; (3) both agent scripts need multi-strategy JSON extraction via existing `extract_json()` utility; (4) volume targets are agent-throughput driven, not hardcoded — plan should use percentage-based targets.

**Primary recommendation:** Build two new scripts (`scripts/generate_deep_judge_cot.py` and `scripts/generate_critique_then_fix.py`) following the `phase2_judge_dataset.py` agent-loop pattern, with seed-sampled few-shot context injected into each agent call. Pilot mode is a flag (`--pilot`) producing 20-50 examples with human review gate before bulk runs.

### Pattern 1: Seed-Injected Few-Shot Agent Call

**What:** Load 3-5 seeds of the matching type, format them as in-context exemplars, inject into a system or user prompt, then call Claude to produce a new example for a given code function.

**When to use:** Both deep_judge_cot and critique_then_fix generation — the seeds ARE the few-shot learning signal.

**Example:**
```python
# Source: project pattern from scripts/phase2_judge_dataset.py + seed schema confirmed from data
import json, random
from pathlib import Path
from scripts.utils import extract_json, call_with_backoff

def sample_seeds(seeds: list[dict], n: int = 3) -> list[dict]:
    """Sample n seeds of the same type for few-shot context."""
    return random.sample(seeds, min(n, len(seeds)))

def format_seed_as_exemplar(seed: dict) -> str:
    """Format a deep_judge_cot seed as few-shot exemplar text."""
    hr = seed['human_reasoning']
    dims = hr.get('dimension_analysis', {})
    return f"""EXAMPLE (seed_id={seed['seed_id']}):
CODE:
```php
{seed['code'][:1500]}
```
REASONING:
Verdict: {hr['verdict']}
Dimension Analysis:
{json.dumps(dims, indent=2)}
Overall Score: {hr['overall_score']}
Key Observation: {hr.get('key_observation', '')}
"""

def generate_deep_judge_cot(code: str, seeds: list[dict], client) -> dict | None:
    exemplars = "\n---\n".join(format_seed_as_exemplar(s) for s in sample_seeds(seeds, 3))
    prompt = f"""You are a WordPress code quality assessor producing deep reasoning chains.
Study these exemplars, then analyze the target code in the same style.

{exemplars}

---
NOW ANALYZE:
```php
{code[:3000]}
```

Return JSON with keys: verdict, dimension_analysis (one entry per dimension with score 1-10 and
analysis text citing specific line references), overall_score, key_observation.
Each dimension MUST name specific WordPress APIs if they appear: $wpdb->prepare(),
wp_verify_nonce(), esc_html(), current_user_can(), check_ajax_referer().
"""
    resp = call_with_backoff(
        client, model="claude-sonnet-4-6-20250514",
        max_tokens=2048, messages=[{"role": "user", "content": prompt}]
    )
    return extract_json(resp.content[0].text)
```

### Pattern 4: Parse Failure Tracking

**What:** Track and reject all examples where `extract_json()` returns None. Compute failure rate at the end. Hard reject if >2% parse failure.

**When to use:** Both generation scripts — DGEN success criterion requires <2% parse failure.

**Example:**
```python
parse_attempts = 0
parse_failures = 0

result = extract_json(resp_text)
parse_attempts += 1
if result is None:
    parse_failures += 1
    continue  # hard reject

failure_rate = parse_failures / parse_attempts if parse_attempts else 0
if failure_rate > 0.02:
    print(f"WARNING: parse failure rate {failure_rate:.1%} exceeds 2% threshold")
```

### Pattern 5: Dimension Coverage Verification (Pilot Gate)

**What:** After pilot batch, check that all 9 rubric dimensions appear at least once across generated examples. Also check that WP-specific API names appear by string match in reasoning text.

**When to use:** Pilot validation step in both scripts.

**Example:**
```python
REQUIRED_DIMENSIONS = {
    "wpcs_compliance", "sql_safety", "security", "performance",
    "wp_api_usage", "code_quality", "dependency_integrity", "i18n", "accessibility"
}
WP_API_CITATIONS = [
    "$wpdb->prepare", "wp_verify_nonce", "check_ajax_referer",
    "esc_html", "esc_attr", "esc_url", "current_user_can", "wp_kses"
]

def validate_pilot_batch(examples: list[dict]) -> dict:
    dims_seen = set()
    api_citations_seen = set()
    for ex in examples:
        reasoning_text = json.dumps(ex.get("reasoning", {}))
        for dim in REQUIRED_DIMENSIONS:
            if dim in reasoning_text:
                dims_seen.add(dim)
        for api in WP_API_CITATIONS:
            if api in reasoning_text:
                api_citations_seen.add(api)
    return {
        "missing_dimensions": REQUIRED_DIMENSIONS - dims_seen,
        "api_citations_found": api_citations_seen,
    }
```

### Anti-Patterns to Avoid

- **Hardcoding volume targets as integers:** Use percentage-based computation from source counts (e.g., `len(passed_functions)` or `len(seeds)`). The `pipeline_orchestrator.py` precedent is clear.
- **Using mutation pool as a source for critique-then-fix:** D-02 explicitly forbids this — `contrastive_mutations.json` is empty and regenerating it is out of scope.
- **Batch API for LLM work:** Project memory mandates Claude Code agents over API where possible. For these scripts, direct API calls in the script body (via `call_with_backoff`) are acceptable since they run as local scripts — but heavy orchestration should use agent spawning. Do NOT use Anthropic Batch API here.
- **Skipping `extract_json` multi-strategy extraction:** Never use `json.loads()` directly on LLM output — always use `extract_json()` from `scripts/utils.py`.
- **Sharing context between eval agent and model under test:** REVL-03 requires independently spawned evaluator — relevant for Phase 4.4, not 4.1, but the pattern must not bleed into generation scripts.

## Common Pitfalls

### Pitfall 1: Partial Dimension Coverage in Generated Reasoning
**What goes wrong:** Agents generate dimension_analysis covering only the 2-3 dimensions the seed addressed, mirroring the seed's partial coverage instead of covering all 9.
**Why it happens:** Seeds themselves only address `dimensions_addressed` — not all 9. Agents learn the partial pattern.
**How to avoid:** Explicit prompt instruction: "You MUST analyze all 9 dimensions regardless of seed coverage. Seeds only show 2-3 dimensions as examples of analysis depth, not as the complete set of dimensions to cover."
**Warning signs:** Generated examples with `dimensions_addressed` list shorter than 9 entries.

### Pitfall 2: Missing WP-Specific API Citations in Reasoning Text
**What goes wrong:** Agent generates abstract reasoning ("this function lacks input validation") without citing specific WordPress APIs by name.
**Why it happens:** Generic code reasoning without WordPress-specific grounding.
**How to avoid:** Prompt must include explicit instruction: "When WordPress APIs appear in the code or are missing from the code, name them explicitly: `$wpdb->prepare()`, `wp_verify_nonce()`, `esc_html()`, `current_user_can()`, `check_ajax_referer()`, `esc_attr()`, `esc_url()`. Do not describe behavior without naming the API."
**Warning signs:** Pilot validation check for WP_API_CITATIONS strings finds few matches.

### Pitfall 3: Corrected Code Without `<corrected_code>` Delimiters
**What goes wrong:** Critique-then-fix agent wraps corrected code in ```php ``` blocks instead of `<corrected_code>` XML tags.
**Why it happens:** Markdown code blocks are the default LLM output format.
**How to avoid:** Prompt must specify: "Wrap the corrected code in `<corrected_code>` XML tags, not markdown code blocks."
**Warning signs:** Parse logic that looks for `<corrected_code>` returns None; fallback extraction needed.

### Pitfall 4: Seeds Not Imported Before Generation Scripts Run
**What goes wrong:** Generation script fails with FileNotFoundError because `data/seeds/` directory doesn't exist yet.
**Why it happens:** Seeds are in `~/Desktop/data/wp-finetune-data/` — not in the project's `data/` directory.
**How to avoid:** Seed import (Wave 0 or first task) must complete before any generation script. Plan must have explicit seed import step.
**Warning signs:** `data/seeds/` directory does not exist at the start of generation.

### Pitfall 5: Scoring Schema Mismatch
**What goes wrong:** Generated deep_judge_cot uses 0-100 scoring (from `phase2_judge_dataset.py` pattern) instead of 1-10 (from `config/judge_system.md` rubric).
**Why it happens:** Two score scales exist in the codebase — judge training data uses 0-100 overall but 1-10 per-dimension.
**How to avoid:** Deep judge CoT examples must use **1-10 per dimension** (matching `judge_system.md` rubric), with the `overall_score` derived from the weighted formula. The seeds use this scale — verify against seed data before prompting.
**Warning signs:** Generated `dimension_analysis` scores fall outside 1-10 range.

### Pitfall 6: Not Handling Failed Functions That Are Empty/Minimal
**What goes wrong:** Some Phase 1 failed JSON files contain functions with very short bodies (e.g., 2-byte files visible in directory listing). Sending minimal code to an agent produces low-quality or nonsensical critiques.
**Why it happens:** Phase 1 extraction kept all functions regardless of size.
**How to avoid:** Filter source functions by minimum length (`len(code) > 100` chars) before including in generation queue.
**Warning signs:** Generated critiques referencing "this empty function" or "this minimal stub."

# Source: verified from /home/robert_li/Desktop/data/wp-finetune-data/ seed inspection
import json
from pathlib import Path

SEEDS_DIR = Path("data/seeds")  # after import step

def load_seeds_by_type(seed_type: str) -> list[dict]:
    """Load all seeds of a given type from both ugc and boundary files."""
    seeds = []
    for fname in ["ugc_seeds.json", "ugc_boundary_seeds.json"]:
        path = SEEDS_DIR / fname
        if path.exists():
            with open(path) as f:
                all_seeds = json.load(f)
            seeds.extend(s for s in all_seeds if s["seed_type"] == seed_type)
    return seeds
```

def format_critique_then_fix_example(defective_code: str, critique: dict,

## Issue #1 Strategic Context (dr-robert-li/wp-finetune#1)

**Title:** Post-Training Pipeline: RL → Compression → Packaging Strategy for Gen+Judge Model

### Relevance to Phase 4.1

Issue #1 is the authoritative strategic document for the entire post-training pipeline. Phase 4.1 implements **Step 2 of the issue's action plan** — expanding the 145 golden CoT seeds (118 actual) into a larger RL-ready data pool via rejection sampling.

### Key Constraints from Issue #1

1. **Pipeline order is RL BEFORE MoE-Sieve** — routing should reflect reward-aligned behavior, not SFT pre-training usage. This means Phase 4.1 data will eventually feed both SFT continued fine-tuning (Phase 4.3) AND the RL training pool (reordered v2.0/v3.0).

2. **RL pool must cover both generation and critique tasks** — Issue #1 Step 2 explicitly states: "Ensure RL pool covers both generation tasks (code output) and critique tasks (rubric scoring + reasoning)." Phase 4.1's two streams (deep judge CoT + critique-then-fix) directly serve this requirement.

3. **Anti-hack eval set** — Issue #1 requires a held-out eval set penalizing verbosity reward hacking, template critique collapse, and self-preference bias. While the eval set design is deferred to Phase 10, Phase 4.1 data generation should be aware that some generated examples may later serve as anti-hack eval candidates. Generated examples should vary in length and style to avoid creating a homogeneous dataset that collapses during RL.

4. **Verifiable reward signals alongside rubric** — Issue #1 calls for PHP syntax checks, PHPCS standards, and functional test pass/fail as verifiable rewards. Phase 4.1 should ensure generated examples include enough metadata for downstream reward computation (source file, PHPCS result if available, function signatures).

5. **Protected expert set** — The 30/70 adapter's emergent judge behavior is "likely tied to specific expert routing." Routing audit happens before RL, but Phase 4.1 data must be diverse enough to exercise both gen and judge expert pathways — not just judge-heavy examples.

6. **GSPO preferred over GRPO** — Sequence-level policy optimization for MoE stability. This doesn't directly affect Phase 4.1 generation, but the data format should be compatible with both GSPO (preference pairs at sequence level) and GRPO (group-relative rewards). Recommendation: each generated example should be a complete, self-contained sequence — not partial responses.

### Risk Register Items Affecting Phase 4.1

| Risk | Impact on 4.1 | Mitigation |
|------|---------------|------------|
| Reward hacking on rubric judge signal (HIGH likelihood) | If generated judge CoT examples are formulaic/templated, they become exploitable during RL | Vary seed selection per call; include boundary seeds for diversity; validate pilot batch shows heterogeneous reasoning styles |
| Emergent judge behavior lost during RL (MEDIUM likelihood) | If Phase 4.1 data is low quality, continued SFT in 4.3 could degrade 30/70 adapter | Pilot validation gate catches quality issues before bulk generation |

### Data Dual-Use: SFT + Future RL

Phase 4.1 data serves two downstream consumers:
1. **Phase 4.3 (immediate):** Continued SFT at lower LR on reasoning-enriched examples
2. **Phase 10-11 (future):** RL training pool for GSPO/GRPO — the same examples may be used as reference completions for reward computation

This dual-use means the output format should include enough structure for both supervised training (messages format) AND reward computation (metadata with scores, verdicts, dimension breakdowns).
