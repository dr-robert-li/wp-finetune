# Phase 4.1: Reasoning Data Generation - Context

**Gathered:** 2026-04-08
**Status:** Ready for planning

<domain>
## Phase Boundary

Generate two streams of reasoning training data for the 30/70 adapter (triage winner):
1. **Deep judge CoT** — dimension-by-dimension analysis with human-grounded reasoning chains and structured scores
2. **Critique-then-fix** — defective code receives structured critique with severity per dimension, followed by corrected version

Source material: 118 human-curated golden seeds (93 ugc_seeds + 25 boundary_seeds) expanded via Claude agent rejection sampling + existing Phase 1 passed/failed functions.

</domain>

<decisions>
## Implementation Decisions

### Source Material
- **D-01:** Seed expansion is the primary data generation strategy. The 118 golden CoT seeds are used as style/quality references. Claude agents generate new examples following the seed patterns via rejection sampling.
- **D-02:** The mutation pool (`contrastive_mutations.json`) is empty and will NOT be regenerated. Critique-then-fix examples are generated from seeds + Phase 1 failed functions instead.
- **D-03:** Phase 1 passed (194) and failed (204) functions are secondary source material — Claude agents generate reasoning for these using seed style as reference.

### Seed Data
- **D-04:** Seeds are imported from `~/Desktop/data/wp-finetune-data/` into `data/seeds/` (4 JSON files: ugc_seeds.json, ugc_seeds_summary.json, ugc_boundary_seeds.json, ugc_boundary_seeds_summary.json).
- **D-05:** Seed structure: `seed_id`, `seed_type` (deep_judge_cot | critique_then_fix), `source_url`, `source_platform`, `source_file`, `code`, `human_reasoning` (dict with dimension analysis + scores), `dimensions_addressed`, `defect_subtlety` (boundary | clear-cut), `annotation_type` (ugc_expert).
- **D-06:** Seeds cover real-world security advisories with expert reasoning — these set the quality bar for generated examples.

### Pipeline Reorder (from Issue #1)
- **D-07:** The milestone pipeline is reordered per issue #1 guidance: RL (GSPO/GRPO) BEFORE MoE-Sieve. Routing should reflect reward-aligned behavior, not SFT pre-training usage. Current v2.0 and v3.0 will swap order.
- **D-08:** GSPO (sequence-level) is the primary RL objective for MoE stability. GRPO + larger group size + Pro-GRPO expand-then-prune is an optional fallback — whether to evaluate GRPO is deferred to Phase 9 planning time.
- **D-09:** AIMER is the primary pruning method (calibration-free, fast iteration). REAP is optional — decide at pruning stage whether to run a comparison.
- **D-10:** A "protected expert set" must be identified during routing audit — dual-purpose experts (activated by both gen and judge) must not be pruned.
- **D-11:** Anti-hack eval set required before RL — penalizes verbosity reward hacking, template critique collapse, self-preference bias in judge outputs.

### Winning Ratio
- **D-12:** 30/70 (gen/judge) is the winning ratio from Phase 4 triage. Only ratio producing parseable judge output (Spearman 0.57, 497 valid pairs). All work in Phase 4.1+ uses the 30/70 adapter.

### Claude's Discretion
- Prompt template structure for agent-generated examples (should follow seed patterns but Claude can optimize for consistency)
- Volume targets — bounded by seed count and agent throughput, Claude determines practical targets during planning

</decisions>
