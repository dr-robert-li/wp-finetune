# Cross-AI Plan Review — Phase 4.1: Reasoning Data Generation

## Gemini Review

### Summary
The implementation plan for Phase 4.1 successfully pivoted from a failing direct-API/heavy-context approach to a highly scalable multi-agent execution pattern using Claude Code. The transition from Plan 01/02 (Pilot) to Plan 03 (Bulk) demonstrated strong technical reactivity. While the volume (1,363 examples) far exceeds the initial 300-example bootstrap target, the shift to a 2-exemplar prompt raises questions about the diversity and depth of the reasoning "flavor" compared to the original 118 human-curated golden seeds. The plan's strength lies in its automated quality gates, but it faces lingering risks regarding data contamination from Phase 1 functions.

### Strengths
- **Multi-Agent Orchestration:** Moving bulk LLM work to parallel Claude Code agents successfully bypassed standard API rate limits and timeout issues that typically plague long-form reasoning generation.
- **Surgical Context Management:** The decision to reduce exemplars to ~3KB (2 inline examples) was a critical "save" that allowed the wave to complete, demonstrating a sophisticated understanding of agent token/timeout trade-offs.
- **Rigorous Rubric Alignment:** Forcing dimension-by-dimension analysis across 9 specific WordPress-centric criteria ensures that the "Reasoning" isn't just a generic summary, but a structured technical audit.
- **Integrated Validation:** The use of PHP linting for Critique-Then-Fix (CtF) provides a hard "ground truth" gate that prevents syntactically broken code from entering the fine-tuning set.
- **Manifest-Based Tracking:** The inclusion of an `input_function_id` manifest is essential for the MoE architecture to prevent "memorization" over "reasoning" during training.

### Concerns
- **Data Contamination (HIGH):** Using Phase 1 functions for Phase 4.1 generation creates a high risk of train/eval leakage. If these functions appear in the Phase 4.4 evaluation sets, the model may "reason" based on having seen the function in earlier SFT phases rather than performing zero-shot analysis.
- **Reasoning Homogeneity / Mode Collapse (MEDIUM):** Reducing the prompt from the full seed set to just 2 exemplars likely narrowed the stylistic variance of the reasoning. With 1,163 CoT examples generated from only 2 templates, there is a risk the model will learn a repetitive, "stochastic parrot" reasoning pattern.
- **Permissive Bias (MEDIUM):** The 741 PASS vs. 251 FAIL ratio in CoT suggests the generator may be over-weighting "safe" or "average" code as "Good." In a "Deep Judge" context, the training data should ideally be balanced or weighted toward identifying subtle failures to improve the model's discriminative power.
- **Citation Hallucination (MEDIUM):** While the plan mentions a "retroactive citation-accuracy audit," it is unclear if this is automated at scale or purely a human sample (N=20). Hallucinated citations in reasoning data are "poison" for a judge model.

### Suggestions
- **Manifest Cross-Check:** Immediately run a set-intersection script between `manifests/cot_input_function_ids.json` and any planned evaluation datasets for Phases 4.2-4.4. Any overlap must be purged from the training mix.
- **Exemplar Rotation:** For future waves, implement a "sliding window" or "random shuffle" for exemplars. Instead of 2 fixed exemplars for all 1,163 examples, each agent should be assigned a different pair from the 93 golden seeds to maintain stylistic diversity.
- **Negative Sampling:** Explicitly target "Borderline" functions (scores 5-7 from Phase 1) for a dedicated CtF wave to force the model to reason through "grey area" WordPress patterns rather than obvious failures.
- **Automated Citation Linker:** Implement a regex-based validator in the assembly phase (Phase 4.2) to ensure that any reasoning claiming a "WPCS Violation" actually cites a valid rule name found in the `wp-coding-standards` repository.

### Risk Assessment
**Overall Risk: MEDIUM**

The execution mechanism (Claude Code agents) is now proven and highly scalable, which was the largest technical hurdle. However, the move to a "skinny" prompt to enable that scale creates a secondary risk of lower-quality, repetitive reasoning. The success of the project now depends on whether Phase 4.2 (Dataset Assembly) can effectively filter the 1,363 "raw" examples down to a "gold-standard" subset.

## Codex Review

### Summary
The Phase 4.1 plans are strong operationally: they define seed sources, pilot gates, bulk acceptance criteria, and a clear execution constraint around Claude Code agents, and the team adapted correctly after the Wave 1 stall failure. The outcome shows the plan was executable at scale, but the evidence provided is not yet enough to conclude that the final 1,363 examples actually satisfy the stated quality gates. The biggest unresolved issues are dataset accounting, train/eval contamination boundaries, and whether the reduced 2-exemplar prompt preserved reasoning quality rather than merely throughput.

### Strengths
- The phase is decomposed well into seed import, pilot validation, and bulk generation, which reduces the chance of scaling a broken generator.
- The plans include concrete acceptance gates instead of vague "quality" language.
- Citation-accuracy checks are the right control for reasoning data; hallucinated justification would poison the judge model quickly.
- The critique-then-fix stream has an objective structural check (`php -l`), which is stronger than pure LLM self-evaluation.
- The contamination guard via input function manifest is the right design direction.
- The team identified and fixed the real scaling bottleneck from Wave 1 instead of treating agent stalls as incidental noise.
- Framing the dataset as a bootstrap supplement rather than the full reasoning corpus is realistic and lowers immediate risk.

### Concerns
- **HIGH**: There is no evidence here that the final 1,363 examples passed the plan's bulk gates. Completion count is not the same as acceptance count. The review needs final metrics for parse failure, citation validity, duplicate rate, lint pass rate, alignment ratio, and human audit results.
- **HIGH**: Train/eval contamination risk is not fully closed. Using Phase 1 failed functions for CtF is reasonable for data generation, but if Phase 4 evaluation also draws from the Phase 1 universe, leakage becomes likely unless function IDs are globally excluded from eval downstream.
- **HIGH**: The CoT class balance looks suspiciously permissive or at least unexplained. 741 PASS vs. 251 FAIL implies about 75% PASS. That may reflect the input distribution, but without source-label priors or score histograms it raises the risk that the generator over-approves code.
- **HIGH**: The CoT totals do not reconcile. 1,163 CoT examples generated versus 741 PASS + 251 FAIL = 992 leaves 171 unaccounted for. Those could be parse failures, abstains, partials, or rejected examples, but the gap must be explicit before the dataset is trusted.
- **MEDIUM**: The 2-exemplar prompt fix likely improved throughput, but it may have narrowed reasoning style and rubric coverage. Two exemplars can anchor output shape too strongly.
- **MEDIUM**: The acceptance gates are partly achievable but may be too weak for reasoning data. `citation_validity >= 70%` still allows a large amount of partially invalid reasoning into a small high-value dataset.
- **MEDIUM**: `alignment_ratio >= 0.50` for CtF is a low bar if the goal is reliable critique-to-repair behavior. Half-aligned outputs are not strong supervision.
- **MEDIUM**: Agent spawning worked after prompt shrinkage, but the failure mode suggests context size sensitivity is a first-order scaling constraint. That creates operational fragility as prompt templates evolve.
- **LOW**: Post-bulk human audit of N=20 per stream is small relative to 1,363 outputs; it may miss systematic errors in niche rubric dimensions.

### Suggestions
- Require a final Phase 4.1 acceptance report with explicit denominators (generated, parsed, accepted, rejected, deduped, audited, exported).
- Add a per-example provenance record: source function ID, source split, generator version, prompt template hash, seed/exemplar version, acceptance metrics.
- Freeze a global exclusion manifest now so any Phase 1 function used in Phase 4.1 data generation is barred from all later eval sets.
- Tighten the reasoning-quality gate for CoT: separate citation presence from citation correctness, require dimension coverage across all 9 rubric areas, score contradiction rate between verdict and rationale.
- Audit the 2-exemplar prompt against the original full-seed prompt on a matched sample.
- Check PASS/FAIL calibration by sampling from known-bad and known-good pools.
- Raise CtF acceptance standards if dataset volume allows.
- Add backpressure and health controls to the agent pattern.

### Risk Assessment
**Overall risk: MEDIUM**

The phase clearly succeeded operationally, and the team handled a real scalability issue well, so this is not a failed plan. The remaining risk is quality-system risk, not execution risk: the current evidence does not prove that the 1,363 examples meet the intended gates, the CoT accounting is incomplete, and contamination boundaries need to be locked before later phases.

## Consensus Summary

Both reviewers independently identified the same core risk profile: **operational success with quality-system concerns.**

### Agreed Strengths
- Claude Code agent orchestration was the right scaling move and was executed well
- Concrete acceptance gates (not vague quality language) are a strong design choice
- Manifest-based contamination guard is the correct approach
- Wave 1 recovery (241KB seed files -> 3KB inline exemplars) was correctly identified as the critical bottleneck fix
- PHP lint on CtF is a strong structural validation

### Agreed Concerns
1. **Data Contamination (HIGH, agreed by both)** — Phase 1 functions used for reasoning generation risk train/eval leakage. Both reviewers stress this must be locked before Phase 4.4 evaluation.
2. **2-Exemplar Prompt Narrowing (MEDIUM, agreed by both)** — Both flagged that reducing from 118 seeds to 2 inline exemplars likely collapsed reasoning style diversity. Both recommended comparing 2-exemplar outputs against full-seed prompts.
3. **PASS/FAIL Calibration (MEDIUM, agreed by both)** — Both noted the 741:251 ratio is suspicious. Neither knows if the generator is permissive or if the input distribution is skewed.
4. **Missing Acceptance Evidence (HIGH, agreed by both)** — Neither reviewer found evidence that the 1,363 examples actually passed the stated quality gates. Completion != acceptance.
5. **Unaccounted Examples (HIGH, Codex only)** — Codex flagged 1,363 generated vs. 992 accounted (741 PASS + 251 FAIL) = 171 missing. This is a real accounting gap.

### Divergent Views
- **Gemini** focused more on reasoning diversity ("mode collapse" risk) and suggested negative sampling for borderline functions.
- **Codex** was more precise on accounting: demanded explicit denominators, per-example provenance, and a global exclusion manifest. Codex also flagged the `alignment_ratio >= 0.50` CtF gate as unusually low.

### Recommended Actions Before Phase 4.2
1. Run a set-intersection between the function_id manifest and all planned eval sets
2. Account for all 1,363 examples (explain the 171 gap)
3. Audit the 2-exemplar prompt against full-seed on a matched sample (verdict agreement, rubric coverage, citation accuracy)
4. Freeze the global exclusion manifest and document it
5. Require a final Phase 4.1 acceptance report with explicit denominators
