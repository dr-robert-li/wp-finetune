## Phase Boundary

</domain>

<decisions>
## Implementation Decisions

### Score consistency validation
- **D-01:** All consistency checks use Claude Code agents. No deterministic rules, regex, or threshold-based heuristics. Deterministic rules are a known failure mode — the judge parse rate from Phase 4 confirmed this.

### Rejection handling
- **D-02:** Examples failing consistency checks are auto-discarded with full logging. Rejected examples are also re-queued for regeneration via Claude Code agents (not manual re-submission).

### Export split strategy
- **D-03:** Domain stratified split first (preserve taxonomy coverage from metadata.json), then random within domains. Ensures rare taxonomy buckets are represented in both train and val splits.

### Canonical template enforcement
- **D-04:** Strict regex matching on the canonical output template. Each example must have: dimension-by-dimension analysis prose, `[/REASONING]` separator, JSON scores block inside `<judge_output>` tags. No deviations allowed.

### Training mix assembly
- **D-05:** Unified format with `format` field (cot / ctf / judge_replay / wp_gen_replay). Weighted sampling: 60% CoT + 25% CtF + 15% replay. Weighted toward contrastive reasoning as the harder, higher-value capability. Qwen3 MoE-friendly — no format switching overhead.

### Claude's Discretion
- Exact regex patterns for template validation
- Batch sizes and parallel agent count for consistency validation
- Regeneration retry limits for rejected examples

</decisions>
