## Structure

### Data loading
- Read CoT examples from `data/phase4_reasoning/deep_judge_cot/deep_judge_cot_bulk.json`
- Read CtF examples from `data/phase4_reasoning/critique_then_fix/critique_then_fix_bulk.json`
- Accept a `--source` flag: `cot`, `ctf`, or `both` (default: `both`)

### Batched Claude Code agent validation (D-01)

Route ALL examples through Claude Code agents — no heuristic pre-filter. Every example gets agent scrutiny. To keep agent count manageable, batch examples into groups of ~20 and spawn agents in parallel (run_in_background=true). Use `claude_agent.py` wrapper.

For each batch, spawn a Claude Code agent with:
```
You are a WordPress code quality consistency validator. Your job is to check whether the written reasoning is consistent with the numeric scores.

Check: Does the reasoning describe issues that should result in low scores, but the scores are actually high? Or vice versa?

Look for contradictions like:
- Reasoning describes "critical SQL injection vulnerability" but security dimension score >= 7
- Reasoning says "no issues found" but overall_score < 50
- Critique describes "critical XSS" but the corrected_code only adds comments, not actual fix
- Reasoning lists multiple "critical" issues but overall_score >= 80

Input format:
{
  "examples": [
    {
      "source_file": "...",
      "function_name": "...",
      "code": "...",
      "reasoning": {
        "verdict": "PASS|FAIL",
        "dimension_analysis": { "wpcs_compliance": {"score": 8, "analysis": "..."}, ... },
        "overall_score": 78,
        "key_observation": "..."
      },
      ...
    }
  ]
}

Output ONLY one line per example:
CONSISTENT|INCONSISTENT: <reason in 10 words or less>

Example output:
INCONSISTENT: critical injection described but security score is 8

Do NOT add explanations. One line per example only.
```

Wait for all agents to complete. Parse responses and classify each example as consistent or inconsistent.

### Output
Write to `data/reasoning_dataset/consistency_valid.jsonl`:
```jsonl
{"source_file": "...", "function_name": "...", "stream": "cot|ctf", "consistency_status": "consistent|inconsistent", "inconsistency_reason": "... or null"}
```

Also write to `data/reasoning_dataset/consistency_rejected.jsonl`:
```jsonl
{"source_file": "...", "function_name": "...", "stream": "cot|ctf", "inconsistency_reason": "critical injection described but security score is 8"}
```

### Exit behavior
- If --auto-regenerate flag is set: re-queue rejected examples via agent for regeneration. Write rejected examples to `data/reasoning_dataset/requeue_for_regeneration.json` with `regenerate_prompt` field containing corrected guidance.
- Print summary: total processed, consistent, inconsistent, auto-regenerated.

### CLI interface
```bash
python scripts/validate_reasoning_consistency.py [--source cot|ctf|both] [--auto-regenerate]
```

## Canonical template enforcement (D-04)
Transform each reasoning example into the canonical template format:
```
{
  "messages": [
    {"role": "user", "content": "<wp_judge> Evaluate this WordPress code:\n\n```php\n{code}\n```"},
    {"role": "assistant", "content": "{dimension_analysis_prose}\n\n[/REASONING]\n\n<judge_output>{json_scores}</judge_output>"}
  ],
  "metadata": {
    "source_file": "...",
    "function_name": "...",
    "stream": "cot|ctf",
    "format": "cot|ctf",
    "dimensions_addressed": [...],
    "source_dir": "..."
  }
}
```

For CoT: build dimension analysis prose from `reasoning.dimension_analysis.{d}.analysis` for each dimension, then the JSON scores block from scores.

For CtF: build from `critique.summary` + `critique.dimensions.{d}.issue` + `critique.dimensions.{d}.fix`, then the JSON from severity scores.

CRITICAL: Per D-04, the canonical template MUST include:
1. Dimension-by-dimension analysis prose (natural language)
2. `[/REASONING]` separator on its own line
3. `<judge_output>` JSON block containing the structured scores

Example canonical assistant content:
```
D1 WPCS Compliance: score 8/10 — Function follows WordPress coding standards...
D2 SQL Safety: score 5/10 — Uses direct $wpdb->query() without prepare()...
D3 Security: score 2/10 — Critical SQL injection vulnerability through unsanitized $_GET input...

[/REASONING]

<judge_output>
{
  "wpcs_compliance": 8,
  "sql_safety": 5,
  "security": 2,
  "overall_score": 45,
  "verdict": "FAIL"
}
</judge_output>
```

## Training mix assembly (D-05)
Target ratio: 60% CoT + 25% CtF + 15% replay.

Calculate targets from the total after consistency filtering:
- `reasoning_total = len(cot_examples) + len(ctf_examples)`
- `replay_target = max(30, round(reasoning_total * 15 / 85))` (at least 30 replay examples for statistical validity)
- `cot_target = round(reasoning_total * 60 / 85)`
- `ctf_target = round(reasoning_total * 25 / 85)`

If cot/ctf counts are less than target (they will be: 196 + 179 = 375), use all available. Replay fills to 15%.

For replay sampling: stratify by source_file (domain) from metadata.json. Sample proportionally from each domain. This preserves taxonomy coverage per D-03.

## Stratified export (D-03)

2. Within each domain, randomly split 80/20 (seed=42)
3. Concatenate domains → train_set, val_set

|----------|-------------|
| Claude Code agents (consistency validation) | Agent prompts receive full source code and reasoning — agents must not leak data or store it |
| Replay data source (final_dataset) | Trusted internal data — no external input, but verify it does not contain the same functions as CoT/CtF (contamination) |
| Regeneration re-queue | If --auto-regenerate is used, rejected examples are re-submitted to agents with guidance — agents must not use the rejected output as a exemplar |

| T-04.2-02 | Data Leakage | Replay source data | accept | Replay data is from the project's own dataset; no external PII. Contamination check between replay and reasoning data in metadata |
| T-04.2-03 | Tampering | metadata.json | mitigate | Write metadata atomically (write to .tmp, rename). Verify JSON parsability after write |
