**3. Push.** `mkdir -p logs && bash scripts/_pub4_upload.sh 2>&1 | tee logs/hf_upload_27.log`

This is a long-running push of a 20-30+ GiB file. It uploads sequentially per-file with a stall watchdog:
`/proc/<pid>/io` `wchar` is summed across the `hf` wrapper and its forked children every 30s; 10
consecutive non-growth checks (5 min) → TERM then KILL the process group → retry, 3 attempts per file
with 30s backoff. That machinery exists because the parallel-folder upload path is a MEASURED deadlock on
this host, not a style preference.

**If the watchdog fires:** it retries on its own. Let it. If all 3 attempts fail for a file, the script
prints `FATAL: <remote> failed after 3 attempts` and exits 1 — do not switch tools to work around it.
Re-run `bash scripts/_pub4_upload.sh` instead: the manifest is an allowlist and `hf upload` is
content-addressed, so a re-run re-verifies or re-pushes each file and converges without corrupting what
already landed. Task 3's API listing is the arbiter of whether it actually converged — trust that, not
the log's optimism.
