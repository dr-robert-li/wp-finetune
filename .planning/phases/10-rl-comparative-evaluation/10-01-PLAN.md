## MANDATORY RECONCILIATIONS (stated visibly per orchestrator instruction)

1. **eval_gate.py deviation.** CONTEXT D-10-04 says "reuse the project's existing CI-aware gate implementation in `eval/eval_gate.py` — do not invent a new one." RESEARCH finding #1 + PATTERNS confirm `eval/eval_gate.py` is a pure POINT-threshold comparator (`actual >= target`) with NO bootstrap CI. The bootstrap machinery (`bootstrap_ci`) lives in `scripts/compute_concentration.py`. We therefore create a NEW `scripts/bootstrap_gate.py` that WRAPS `compute_concentration.bootstrap_ci` (reusing the project's existing, battle-tested statistics — Phase 7 + Phase 8 use the same function) and adds a pair-level Spearman-improvement bootstrap. This satisfies the INTENT of the locked decision (reuse the project's CI-aware machinery, do not invent new thresholds) while correcting the file reference. We do NOT reuse `eval_gate.py`'s threshold logic for the CI gates; it is used only as a point-estimate helper where appropriate.

2. **ROADMAP staleness (mirrors Phase 9 D-09-01).** ROADMAP Phase 10 skill text says `dgx.execute("eval_toolbox", ...)` and Python-side `Agent(run_in_background=true)`. CORRECTED actual wiring: the RL model is Tinker-exported (`tinker://` → HF archive), merged via `scripts/merge_tinker_v3.py` (MoE-only path), and served via **vLLM-on-DGX** (Phase 4.4 re-bench pattern, `http://localhost:8020/v1`). Python-side LLM-judged dims dispatch via the **`scripts/claude_agent.py`** subprocess path (NOT the Anthropic API; `Agent(run_in_background=true)` is a skill-orchestrator-only construct, never inside Python eval scripts).

3. **Retention bar is an OPEN DECISION (D-10-04 #4).** We do NOT hard-code Phase 7's `0.9426` as the RL retention bar — that value measures SFT cross-run per-layer Jaccard stability, a DIFFERENT quantity from RL per-step `jaccard_protected`. We use a **configurable provisional bar = 0.85**; the RLEV-02 report presents the full `jaccard_protected` trace regardless of disposition; the final threshold is confirmed at the D-10-04 human-review checkpoint. Tests assert against the configurable bar, not 0.9426.

4. **D-10-03 aggregation method (wp-bench aggregate gate).** CONTEXT D-10-03 says the wp-bench HARD GATE passes when the "aggregate wp-bench **bootstrap lower bound** >= the v1.2 baseline aggregate point estimate." However, the 0.4616 baseline IS wp-bench's own DETERMINISTIC **weighted** overall (`metadata.scores.overall`, roughly equal knowledge/execution sub-type weight) — NOT a simple per-task mean. A flat-array bootstrap of the 344 binary per-task scores produces a count-proportional simple mean (~93% knowledge / ~7% execution by task count), so comparing its CI lower bound against the weighted baseline is apples-to-oranges and can falsely PASS a model D-10-03 requires to FAIL (concrete: knowledge=0.50, execution=0.38 → simple-mean lower bound ≈0.49 PASSES but true weighted overall ≈0.44 FAILS). Per the plan-checker BLOCKER fix we therefore compare the candidate's weighted `metadata.scores.overall` DIRECTLY (`>= 0.4616`), honoring D-10-03's INTENT (meet/exceed the weighted baseline + the sub-type floors) while correcting the aggregation. wp-bench grader scores are deterministic per run, so a direct point comparison is exact; stratified resampling that reproduces wp-bench's sub-type weighting was the bootstrap-preserving alternative but point comparison is chosen for determinism and stated. The per-DIMENSION regression gates and the jaccard retention gate KEEP their `bootstrap_ci`; the sub-type floors (knowledge >= 0.45, execution >= 0.375) are UNCHANGED.
