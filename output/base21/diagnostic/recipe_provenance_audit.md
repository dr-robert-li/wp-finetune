# Phase 21 Forensic Audit — Is the Training Recipe/Data Mismatched to Qwen3.6-35B-A3B?

**Scope:** read-only audit. No new training, no new eval runs. All numbers below are taken directly
from committed artifacts (paths cited inline) or computed from them with one-off read-only scripts
(shown inline, not persisted).

**Bottom line up front:** Yes — the data is mismatched, but not primarily for the "old-base
idiosyncrasy" reason the questions anticipated. The dominant, load-bearing finding is a **task-mix
mismatch**: the adapter shipped as the "generation" role is fine-tuned on a corpus that is **86%
judge/critique-reasoning content and only 13% actual code-generation examples**, at a **high LR
(4.99e-4) for 3 epochs on only 563 train rows**, and this measurably **drags the model's raw
code-generation ability down** (wp-bench 0.4897 raw → 0.372 trained, a **-0.118 regression**, landing
*below* both the untrained baseline and the old base's own trained result of ~0.4616). The judge role,
trained on the same recipe against human-authored ground truth, **improves** (rho 0.7477 raw → 0.8358
trained). Same LR, same epoch count, same base, same LoRA config — opposite outcomes. The discriminator
is the *target distribution*, not the hyperparameters.

## 2. Question 1 — Data-era mismatch

### 2.1 Provenance is mixed, not uniformly "old-base teacher" as hypothesized

`data/reasoning_dataset/metadata.json` records two generations:

- **Original 418 examples (`generated_at: 2026-04-23`, commit `dca52c9`, 2026-05-21):** 181 CoT + 175
  CTF + 62 replay. The CoT/CTF streams were generated by **Claude Code agents (Sonnet)**, per
  `.planning/ROADMAP.md` lines 246-264 ("Bulk deep judge CoT agent generates... via Claude Code
  agents"), *not* by the old fine-tuned base's own outputs. So the premise "targets came from a weaker
  fine-tuned teacher" is **false for the original 418** — the teacher was an external frontier model,
  not Qwen3-30B-A3B.
- **Backfill +550 raw / +245 included (`commit e71c05d`, 2026-05-25, "D-03 CoT backfill"):**
  `metadata.json`'s `backfill_provenance` records `"generator": "vLLM Qwen/Qwen3.6-35B-A3B
  (enable_thinking=false) @ 192.168.1.61:30000"` — **generated by the very base being fine-tuned now**,
  gated by spawned Claude Sonnet consistency review (432/550 passed, 245 included after balancing).

So roughly 245 of 423 CoT examples (58%) are **self-distilled from Qwen3.6-35B-A3B's own raw
completions**, not "weaker-teacher" targets in the sense the question framed. This is evidence
*against* a simple "regression to a weaker teacher" story for the CoT majority stream, and evidence
*for* a different mechanism: training a model on a rejection-sampled subset of its own distribution
plus externally-styled Claude CoT is a form of narrow-distribution fitting (see §3), independent of
teacher strength.

The **replay stream** (15% of the mix, sourced in `scripts/assemble_reasoning_dataset.py:204-249` from
`data/final_dataset/openai_train.jsonl`, the original Phase 2/3 code-gen corpus) is genuinely
old-era/base-agnostic plain code (no reasoning scaffold) — verified by direct sampling (below).

### 2.2 The real mismatch: task-mix, not just era

Direct tag count over the 563-row train file used by GEN-02 (`data/reasoning_dataset/openai_train.jsonl`):

```
<wp_judge>  482  (85.6%)
<wp_gen>     73  (13.0%)
other         8  ( 1.4%)
```

The adapter that ships as the **"generation" role** (`wp-gen-v4`, per `ROADMAP.md` line 36-37: "v1.2
SFT generation model... for the wp_gen role") is fine-tuned on a corpus that is **86% deep-judge
CoT/critique-then-fix reasoning content**, wrapped in the `[REASONING]...[/REASONING]` /
`<judge_output>{...}</judge_output>` scaffold (confirmed by direct sampling of the first 20 rows — all
`<wp_judge>`-tagged, all containing `[/REASONING]` and `<judge_output>`), and only **13% plain
`<wp_gen>` code-completion examples** (sampled directly — no reasoning wrapper, just raw PHP function
bodies, e.g. an Elementor icon-widget Backbone template and a `is_free_plan()` getter). Assistant-target
length averages 2650 chars (median 2785, max 5371) — dominated by the long structured judge prose, not
short code completions.

Continuing to fine-tune a shared-weights LoRA adapter on a corpus this lopsided toward one task format,
at a high LR, for 3 epochs, on only 563 rows, is a textbook setup for cross-task interference: the
gradient signal is overwhelmingly "produce judge-scaffold prose," and the much smaller code-completion
signal (73 rows) is easily swamped or memorized rather than generalized.

### 2.3 Empirical confirmation: this measurably breaks code generation

`output/base21/gen03_wpbench.json` (the only artifact in this audit that directly answers "is training
on these targets net-negative for the new base"):

| | wp-bench overall |
|---|---|
| **Raw new base, untrained** (`fresh_new_base_anchor`) | **0.4897** |
| **New base after GEN-02/03 SFT on the reused v1.2-era mix** | **0.372** |
| Inherited acceptance floor (old base's shipped result) | 0.4286 |
| Old base's own SFT result (v1.2, cited in `ROADMAP.md` line 36) | ~0.4616 |

Fine-tuning the new base on this data **cost 11.8 points of wp-bench** relative to not training it at
all, and landed **below the old base's raw score (0.4033, per the base-anchor commit
`57f6947`) and well below the old base's own trained result (0.4616)**. This is the single strongest
piece of evidence in the audit: it is not a hypothetical "targets might be weaker than 0.4897" question
— the artifact shows the trained model performing *worse than doing nothing*. That is only possible if
either (a) the training signal pulls the model toward a genuinely lower-quality code-gen distribution,
or (b) the training signal is dominated by non-code-gen content that overwrites code-gen skill (§2.2),
or both. Given the 86/13 task-mix finding, (b) is very likely the primary mechanism, with (a) (the
replay stream's plain-code targets being stylistically old and possibly lower-quality than the new
base's own native completions) as a secondary contributor. This audit cannot cheaply separate the two
without a new training run (see §6).

## 3. Question 2 — LR/epoch audit

### 3.1 The LR is not new-base-specific, and it is not a "carried-over mistake"

`gen01_format_decision.json`'s `lr_rationale` field states the 4.99e-4 figure is Tinker's own
`hp.get_lr(BASE_MODEL, is_lora=True)` heuristic, **queried fresh against the new base string**, and
that ROADMAP's literal "LR<=2e-5" text is a stale reference to an **abandoned DGX/Unsloth pipeline**
(confirmed independently: `telemetry/training/phase4.3_20260526T111327Z/_summary.md` shows a
*different*, continued-fine-tune run with initial loss 1.22 and LR starting at 1.985e-5 — this is a
low-LR *continuation* of an already-instruction-tuned checkpoint, not the from-base Tinker SFT this
audit is about, and it was superseded).

Critically, the **old base's actual shipped Tinker runs used the identical 4.99e-4 LR**:
`logs/tinker_p4_full.log` (`wp-reasoning-v2`, `base=Qwen/Qwen3-30B-A3B`): `lr=4.99e-04`.
`logs/relabel_sft/v131.log` (`wp-reasoning-relabel-v131`, same base): `lr=4.99e-04`. So GEN-02/JUDGE-02
did not "inherit a stale value tuned for a different architecture" — Tinker's own auto-LR heuristic
independently resolves to the same value for both the 128-expert old base and the 256-expert new base
(both are billed at Tinker's "30B-A3B-Base" price tier per `V4-RERUN-ROADMAP.md` line 85-86, suggesting
Tinker's heuristic keys off active-parameter count, not expert count). This is a genuine finding but
cuts the other way from the question's premise: the per-active-parameter update magnitude is *the same*
between old and new base by Tinker's own model, since both have ~3B active params. Whether Tinker's
heuristic correctly accounts for a 256-expert fan-out (vs 128) is unverifiable from this audit — no
project artifact benchmarks LR sensitivity specifically against expert count — but it is not evidence
of a base-mismatched LR; it's evidence of an *identically-applied* LR across two different expert counts,
which is a more defensible framing than "carried over by mistake."

### 3.2 Per-epoch loss deltas: new base converges to a floor ~40% lower than the old base

Using the last logged step of each epoch (`output/base21/gen02_full_log.txt` vs `logs/tinker_p4_full.log`,
identical recipe: rank 32, MoE-only, bs 8, 70 steps/epoch, same train file):

| | Old base (Qwen3-30B-A3B) | New base (Qwen3.6-35B-A3B) |
|---|---|---|
| Step 1 loss | 12.40 | **7.97** |
| ep1 end (step 70) | 6.76 | 4.53 |
| ep2 end (step 140) | 4.46 | 2.52 |
| ep3 end (step 210) | **2.40** | **1.46** |
| ep1→ep2 Δ | -2.30 (-34%) | -2.01 (-44%) |
| ep2→ep3 Δ | -2.06 (-46%) | -1.06 (-42%) |

Two things stand out:

1. **The new base starts at a much lower loss (7.97 vs 12.40)** on the identical targets at
   initialization — it already assigns these old-era judge-scaffold targets meaningfully higher
   likelihood than the old base did, before any training. This is consistent with the new base being a
   generally better-calibrated/lower-entropy LM, not necessarily evidence the targets match its
   "style" — but it does mean less distributional distance needs to be closed by training.
2. **The new base's terminal loss (1.46) is ~40% lower than the old base's (2.40)** under the identical
   3-epoch/560-row regime. Judge-02 shows the same pattern (new base s0/s1/s2 land at 1.46-1.58;
   old-base `wp-reasoning-relabel-v131` — same 4.99e-4 LR, 3 epochs, but 270 steps against a larger
   augmented+replay30 file — lands at 2.25). Fitting the *same-shaped*, small, homogeneous corpus this
   much more tightly in the same number of epochs, combined with the wp-bench regression on held-out
   real-world code (§2.3), is the classic train-loss/generalization divergence signature of
   **overfitting/memorization on a narrow 560-row corpus**, not a training-instability or LR-explosion
   signal (no loss spikes, no divergence — the curve is smooth and monotone in both cases, ruling out a
   "too-hot LR blew up training" story in favor of a "too-hot LR for too many epochs on too little data,
   too fast" story).

## 4. Question 3 — Renderer/masking interaction

`scripts/tinker_reasoning_data_v4.py` uses `qwen3_5_disable_thinking` (resolved live against the
installed `tinker_cookbook` registry, matched to the base's actual `Qwen3_5MoeForCausalLM` class per
the module docstring) with `train_on_what=LAST_ASSISTANT_MESSAGE`. The docstring documents a verified
check: the HF chat template inserts an empty `<think>\n\n</think>\n\n` block into the assistant header,
but that header gets `weight=0` under `LAST_ASSISTANT_MESSAGE` — confirmed empirically by decoding only
the weight>0 span and checking for `<think>`/`</think>` tokens (`empty_think_injected: false` in
`gen01_format_decision.json`). So the loss-weighted training signal never touches the native `<think>`
mechanism at all — the model is trained purely to emit the project's own `[REASONING]...[/REASONING]`
scaffold as plain assistant text, with the base's native thinking machinery masked out on both sides
(train: zero-weighted; eval: `gen03_wpbench.json` also runs with `"enable_thinking": false` for both the
raw-anchor and trained-model bench passes). This is an apples-to-apples control — the wp-bench
regression (§2.3) is **not** an artifact of a thinking-mode mismatch between train and eval, since both
conditions disable thinking consistently. This rules out Q3 as the primary driver; the renderer/masking
mechanics check out as correctly implemented and symmetric between train and eval.

One second-order note: `FEATURES.md` (§1, row 2b/Stage 3) flags an independent community report of an
**18% output-format-noncompliance rate** and identifier hallucination on this exact base family. The
`terse_gate` result in `gen02_run.json` (`gen02_fs_gate.json`) shows a 14.18% terse rate on the raw
141-row val set, but the receipt's own disposition traces all 20 terse hits to the 21 replay
(code-gen, no `[/REASONING]` expected) rows and the canonical re-scored cot+ctf-only gate passes clean
(0.0%/0.83% at temp 0/0.7, both under the 10%/15% Wilson-upper thresholds). This is a plausible,
internally-consistent explanation and not obviously wrong, but it was **not independently re-derived in
this audit** (would require re-running `scripts/tinker_fs_gate.py`) — flagged as a secondary,
lower-confidence item rather than folded into the primary verdict.

## 5. Question 4 — Judge vs. Gen asymmetry

| | Raw new base | Trained (new base) | Old base (same measurement method) | Δ vs raw |
|---|---|---|---|---|
| **Gen** (wp-bench) | 0.4897 | 0.372 | 0.4616 (v1.2) | **-0.118** |
| **Judge** (Tinker-capture rho, s1 best seed) | 0.7477 (`judge03_capture_rho.json` baseline_rho) | 0.8358 | 0.8274 (v1.3, `21-03-PLAN.md` line 80) | **+0.088** |
| **Judge** (3-seed ensemble) | 0.7477 | 0.8160 | 0.8075 (vLLM-served, `ROADMAP.md` line 92) | +0.068 |

Same base, same recipe (rank 32, MoE-only LoRA, LR 4.99e-4, 3 epochs), same general reasoning-scaffold
format. The only structural difference is the **target source**: GEN-02 trains against
`data/reasoning_dataset/openai_train.jsonl` (86% judge-CoT/critique content generated by Claude/self-
distillation + a 13% old-era plain-code replay slice, evaluated on a *different* task — code
generation); JUDGE-02 trains against `data/reasoning_dataset/openai_train_relabel_v1.jsonl`, whose
labels trace to `data/relabel_v1/labels.json` — **603 items, human-annotated, M=3 median-aggregated**
(`V4-RERUN-ROADMAP.md` line 57), evaluated on the *same* task the labels were written for (judge
scoring).

This is a clean natural experiment and the strongest single piece of converging evidence in the audit:
identical hyperparameters, identical base, identical rendering/masking mechanics, opposite outcomes,
perfectly tracking whether (a) train-time targets and eval-time task match, and (b) target provenance is
human/era-agnostic vs. mixed/self-distilled-and-old. The `s1 0.8358 > old-base 0.8274` framing in the
question is directionally correct but should be read cautiously — the new s1's bootstrap CI
`[0.774, 0.879]` comfortably contains 0.8274, so this specific single-seed comparison is not a
statistically decisive "beat," just a nominal one. The baseline-relative gain (+0.088, `rho_new`
0.7872/`delta_vs_baseline` 0.0395 in the vLLM-served `eval_summary.json`, and the larger +0.088 Tinker-
capture gain) is the more load-bearing number, and it is unambiguous: **judge training helps, gen
training hurts, same recipe.**

## 6. Verdict — ranked hypotheses + cheapest confirming experiments

**H1 (primary, well-supported): Task-mix / shared-recipe mismatch.** The "gen" adapter is trained on a
corpus that is 86% judge-reasoning content and 13% code-completion content, at a hot LR for 3 epochs on
560 rows. This is sufficient by itself to explain a code-gen regression even before considering data
era — training a code-gen role predominantly on a different task's targets is expected to hurt that
role's native skill via representational interference, independent of who produced the judge-CoT
targets. Confirming experiment (cheapest, ~$2 Tinker run): **re-run GEN-02 restricted to the 73
`<wp_gen>`-tagged replay rows only (or a rebalanced 50/50 gen/judge mix)** and re-score wp-bench. If
wp-bench recovers toward 0.4897, H1 is confirmed as dominant.

**H2 (secondary, partially supported): Data-era / self-distillation ceiling on the replay+CTF streams.**
58% of the CoT stream is genuinely self-distilled from this exact base (not "weaker old-base teacher" as
hypothesized), but the 15% replay stream and 25% CTF stream (corrected-code targets) are old-era
(Phase 2/3, pre-Qwen3.6) and were never re-verified against the new base's own raw output quality.
Confirming experiment (cheap, no training): **score the 73 replay-stream targets themselves with the
same wp-bench-style rubric/PHPCS checks used for `gen03_wpbench.json`**, and separately generate the new
base's own completions for those same 73 prompts at temp 0 (`enable_thinking: false`) and score those —
compare distributions. If replay targets score materially below the new base's own raw completions,
H2 is confirmed as a contributing (not sole) cause.

**H3 (weak/ruled-out as primary): LR too hot for a 256-expert architecture.** The LR is empirically
identical to the old base's LR (same Tinker heuristic call, same value, same monotone-smooth curve
shape) — not a mismatch introduced by this base swap. The lower terminal loss (1.46 vs 2.40) is more
parsimoniously explained by H1 (narrow, homogeneous, small corpus + hot LR + 3 epochs = memorization)
than by an expert-count-specific LR miscalibration, since no artifact isolates expert-count as a
variable. Confirming/ruling-out experiment: a **1-epoch bench checkpoint** (already recorded —
`per_epoch_sampler_paths.ep1` in `gen02_run.json` — no new training needed) scored on wp-bench. If ep1
already regresses below 0.4897, the "too many epochs on too little data" framing of H1/H3 sharpens
further; if ep1 holds near 0.4897 and only ep2/ep3 regress, that's a clean overfitting-onset signal.

**H4 (ruled out): Renderer/thinking-mode masking artifact.** Verified symmetric between train (weight-0
header, confirmed no `<think>` leakage into the loss-weighted span) and eval (`enable_thinking: false`
on both raw-anchor and trained-model bench passes). Not the driver.

**Recommended next step (cheapest, highest information):** run the **ep1 bench probe** (H3 test) first
— it requires zero new training (checkpoint already exists) and only a `~20 min` wp-bench eval pass,
and its result sharpens which of H1/H2 to spend the next `~$2` Tinker run on. If ep1 already regresses,
prioritize the **gen-role rebalance re-run (H1 test)** next; if ep1 holds, prioritize the **replay/CTF
target-quality scoring (H2 test)**, since the damage is then attributable to epoch-count-driven
memorization of the specific old-era code content rather than the task-mix ratio itself.
